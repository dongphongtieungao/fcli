# Business Requirements — OpenCode PrivateGPT Bridge MVP

## Purpose

Define the published business scope and observable outcomes for the OpenCode PrivateGPT Bridge MVP. This document translates the canonical requirement into business-facing requirements that guide design, implementation, verification, pilot acceptance, and handover.

## Appendix Relationship

This is the business-requirements appendix of [the canonical requirement](./00.requirement.md). It may refine business outcomes, rules and acceptance evidence, but it must not introduce a new API surface, provider capability, runtime dependency or deployment model. Those details belong to the architecture and Provider SPI appendices only when consistent with the parent requirement.

## Context

Developers need a stable terminal-based coding workflow that uses the organisation's PrivateGPT capability without depending on the existing IntelliJ plugin runtime. The MVP provides a local adapter between OpenCode CLI and PrivateGPT, retaining PrivateGPT Agent API integration while leaving repository access, file editing, command execution, and user approval to OpenCode.

## Business Outcome

The MVP is successful when an authenticated developer can configure OpenCode to use the local adapter, select the supported Gemini 2.5 Pro model, receive a PrivateGPT-backed response through a synchronized server-side agent, and conduct a non-destructive repository analysis without the adapter executing tools or exposing sensitive credentials.

## Actors And Responsibilities

| Actor | Responsibility | Out of responsibility |
|---|---|---|
| Developer / OpenCode user | Logs in, configures OpenCode, selects the supported model, and approves edits or commands in OpenCode. | Direct calls to PrivateGPT internal APIs; management of PrivateGPT agent internals. |
| OpenCode CLI | Provides the coding workflow, workspace access, file operations, command execution, and approval prompts. | PrivateGPT token lifecycle and request translation. |
| OpenCode PrivateGPT Bridge | Authenticates to PrivateGPT, resolves the approved model, synchronizes the PrivateGPT Agent, maps requests, parses responses, and exposes the local OpenAI-compatible interface. | Reading or editing repository files; executing commands or plugin-style tool calls. |
| PrivateGPT platform | Hosts the model, Agent API, and Conversation API used by the adapter. | OpenCode workspace operations and local user approvals. |

## Scope

### In Scope

- A local Python adapter that listens on localhost and exposes the MVP OpenAI-compatible interface.
- Browser-based login, secure local token handling, token refresh, and login status reporting.
- Only the public model alias `gemini-2.5-pro` in the MVP.
- Required PrivateGPT Agent create, read, update, and synchronization behaviour for the active workspace.
- OpenCode-compatible chat completion processing through a mapped PrivateGPT conversation request with `metadata.agent_id`.
- Parsing PrivateGPT SSE output into an OpenAI-compatible response.
- OpenCode provider configuration help, direct HTTP smoke testing, and an OpenCode compatibility pilot.

### Out Of Scope

- Multi-model routing, fallback to another model, or support for models other than Gemini 2.5 Pro.
- Mandatory LiteLLM runtime, a central shared gateway, or an IntelliJ runtime dependency.
- Adapter-side file operations, shell command execution, tool-loop execution, or parsing/execution of plugin-style `<tool_call>` content.
- A full file-upload UI, image input, persistent full-chat history, browser UI automation, and multi-user deployment.

## Business Rules And Constraints

1. The adapter is a translation and integration boundary, not a second coding agent.
2. PrivateGPT Agent API is mandatory for the MVP; inline instruction fallback is not permitted when agent synchronization fails.
3. Gemini 2.5 Pro availability must be resolved against the PrivateGPT platform or explicitly configured; another model must not be substituted.
4. A conversation has no shared global conversation identifier in the MVP; future session mapping requires a separate decision.
5. The adapter must reject unsupported tools or tool requirements rather than executing them.
6. Runtime evidence from the old Java plugin is reference evidence for PrivateGPT integration behaviour, not a target runtime design to be copied.

## Acceptance Gate

The MVP may be accepted only when every published business requirement has evidence appropriate to its type:

| Evidence Area | Minimum Evidence |
|---|---|
| Authentication and security | Browser login test, token-refresh test, secure-store decision, and redacted log review. |
| Model and agent readiness | Model discovery result, unsupported-model rejection, Agent API create/update evidence, and cached agent metadata evidence. |
| Conversation compatibility | Direct HTTP smoke result, captured/redacted OpenCode request shape, SSE parser result, and OpenAI-compatible response result. |
| OpenCode pilot | Provider configuration, non-destructive analysis, one proposed small fix, and approval behaviour evidence. |
| Boundaries and safety | Evidence that adapter has no file/tool/command executor and does not log credentials or raw source code. |

## Known Validation Conditions

The following items are explicitly required before implementation or pilot acceptance, because the canonical requirement identifies them as runtime information still to be confirmed:

- The PrivateGPT internal model ID for Gemini 2.5 Pro.
- Current responses for supported-model discovery, Agent API create/update, and Conversation API SSE.
- The actual OpenCode request shape and whether OpenCode requires streaming or native tool calls.
- The approved local token-store implementation and PrivateGPT policy for sending internal source code.

These are validation conditions, not alternative business requirements. They must be resolved with runtime evidence or an approved decision before the related implementation is claimed complete.
