# Requirements

## Purpose

Provide a reusable requirements control framework for projects created from this repository.

## Context

This file is the active requirement entrypoint. It should remain useful before a product domain is known by defining how requirements are captured, reviewed, traced, and promoted.

## Requirement Principles

| Principle | Rule | Evidence |
|---|---|---|
| Source-backed | Every requirement must link to a user request, decision, discovery note, or source artifact. | `raw.md`, `intake.md`, decision log, ticket, meeting note |
| Testable | A requirement should have acceptance criteria or a clear verification gap. | Use case, feature file, QA matrix, verification note |
| Scoped | In-scope and out-of-scope behavior must be explicit. | Requirement brief or task intake |
| Traceable | Requirements should link forward to use cases, BPMN, contracts, architecture, and tests when those exist. | Related links and source IDs |
| Reviewable | Draft assumptions must not be treated as accepted product behavior. | Status and open questions |

## Requirement Workflow

| Step | Action | Output |
|---|---|---|
| Intake | Capture raw request and classify task profile. | `llm-wiki/tasks/current/raw.md`, `intake.md` |
| Clarify | Identify users, problem, scope, assumptions, risks, and open questions. | Requirement brief |
| Draft | Write candidate requirements and acceptance criteria. | Requirement entry or project-specific requirement doc |
| Map | Connect requirement to use case, BPMN, contract, QA, or architecture. | Trace links |
| Promote | Move accepted knowledge from workspace to official docs. | Updated requirement/use case/spec |

## Requirement Record Shape

| Field | Description |
|---|---|
| ID | Stable requirement identifier. |
| Statement | One clear requirement written as observable behavior or constraint. |
| Source | User decision, ticket, task, discovery note, or official artifact. |
| Status | `draft`, `reviewed`, `published`, or `deprecated`. |
| Acceptance | Testable criteria or explicit verification gap. |
| Links | Use case, BPMN, contract, QA, architecture, or runbook references. |

## Promotion Rules

Promote to this file or a project-specific requirement document when the workspace contains durable scope, behavior, acceptance criteria, or product constraints. Keep raw discussion in `llm-wiki` unless it has been reviewed or explicitly accepted.
