# OpenCode PrivateGPT Bridge Architecture and Technical Requirements

## Purpose

Specify the target Python architecture that implements the canonical OpenCode PrivateGPT Bridge requirement. This appendix turns the architecture idea into implementable boundaries without treating unverified PrivateGPT internals or optional libraries as fixed facts.

## Appendix Relationship

This is the technical-architecture appendix of [the canonical requirement](./00.requirement.md). It may allocate responsibilities and state technical constraints needed for the parent requirement, but it cannot add business scope, change the formal MVP API surface, or treat design-input examples as verified runtime facts.

## Architecture Profile

The MVP is a **Developer Local Bridge Architecture**: a Python modular monolith, running as one local process on a developer workstation. It combines a console runtime with an embedded FastAPI/ASGI server, an OpenAI-compatible translation boundary, and a PrivateGPT session/Agent integration.

It is not a shared enterprise gateway, a multi-tenant platform, a distributed deployment, or a second coding agent.

| Decision | Requirement |
|---|---|
| Runtime | Python process, local bind `127.0.0.1:4100` by default. |
| Client | OpenCode CLI is the sole formal MVP client. |
| API surface | `/health`, `/v1/models`, `/v1/chat/completions`. No `/v1/responses`, WebSocket, image, or upload contract is committed in MVP. |
| Model | Expose only `gemini-2.5-pro`; resolve/configure its PrivateGPT model ID without fallback. |
| Agent | PrivateGPT Agent API is mandatory. One cached Agent binding per workspace is the published MVP strategy. |
| Stream | PrivateGPT SSE is parsed and relayed as OpenAI-compatible SSE. |
| Tool ownership | OpenCode owns filesystem, command execution and approval; adapter executes none. |

## Container Boundary

```text
Developer + OpenCode CLI
  -> local HTTP /v1
OpenCode PrivateGPT Bridge (one Python process)
  ├─ Console runtime: login, start, status, doctor, agent commands
  ├─ Embedded API server: health, validation, SSE lifecycle
  ├─ Gateway pipeline: validate, enrich, route, execute, relay, finalize
  ├─ Agent service: model resolution and workspace Agent synchronization
  ├─ Provider SPI / PrivateGPTProvider: provider mapping and upstream SSE parsing
  ├─ Token manager + secure store adapter
  └─ Workspace Agent metadata cache
  -> authenticated HTTPS
PrivateGPT Model, Agent and Conversation APIs
```

All internal collaboration inside the Python process is by module/function call, never internal HTTP or IPC. Provider-specific request schemas remain below the Provider SPI boundary.

## Python Component Responsibilities

| Component | Owns | Must not own |
|---|---|---|
| CLI runtime | Process lifecycle, login/logout, status, doctor and configuration hints. | Chat proxying, workspace operations, credentials in output. |
| FastAPI/ASGI layer | Routes, request validation, response framing, HTTP/SSE cancellation lifecycle. | PrivateGPT endpoint schemas or direct concrete-provider calls. |
| Gateway pipeline | Validation, correlation IDs, resilience policy, provider invocation and response relay. | A separately deployed service or tool execution. |
| Agent service | Workspace key, instruction hash, model resolution and Agent create/update decision. | Inline fallback when Agent synchronization fails. |
| Provider registry/SPI | Provider-qualified model resolution and abstraction. | Business policy or endpoint-specific HTTP details. |
| `PrivateGPTProvider` | Authenticated Model/Agent/Conversation mapping and upstream SSE parsing. | Hidden retries, circuit breaker, filesystem/shell tools. |
| Token manager | Login state, secure-store access, access-token memory cache and refresh. | Plaintext credential persistence or logging. |
| OpenAI mapper | OpenCode message rendering and compatible non-stream/SSE response mapping. | Executing model tool calls. |
| Redacted diagnostics | Request ID, route, latency, provider, status and resilience state. | Tokens, cookies, raw prompts or raw source code. |

## Core Request Flow

1. OpenCode calls `/v1/chat/completions` with the sole supported model alias.
2. API validation rejects unsupported models and unsupported tool payloads before provider work.
3. Token manager checks login state and refreshes once on a pre-request 401 path when supported by the validated auth flow.
4. Agent service resolves the internal model ID, then finds, creates, or updates the workspace-scoped Agent when its model or instruction hash differs.
5. The mapper produces a unified request; the provider builds the PrivateGPT conversation request with `metadata.agent_id` and `tools: []`.
6. For non-stream requests, the provider parses upstream SSE to visible content and the mapper emits an OpenAI-compatible response.
7. For `stream=true`, the mapper relays incremental OpenAI-compatible SSE chunks and ends with `[DONE]`.
8. OpenCode independently decides whether to read files, edit files, or run commands under its own approval policy.

## Agent and Conversation State

The workspace Agent cache stores only the metadata needed to avoid repeated create/update operations: workspace key/hash, `agent_id`, internal model ID, instruction hash and update timestamp. It does not persist full chat history, token values, or raw prompts.

`conversation_id` and `parent_message_id` are `null` in the MVP unless a separate session-mapping decision introduces them. A global shared conversation is prohibited. The idea document's per-request Agent option is a future alternative, not the selected MVP contract.

## Authentication and Local Security

The adapter starts login in a browser and uses an approved local secure store for refresh/session material; access tokens are memory-cached. The exact callback, browser-session or cookie mechanism must be selected only after runtime evidence confirms the PrivateGPT flow. Reading browser profiles is not an MVP requirement.

Security requirements:

- Bind only to localhost by default; never bind to LAN interfaces without explicit approval.
- Do not hardcode, return, or log tokens, cookies, passwords, raw prompts, or raw source code.
- Use a local adapter API key only when OpenCode configuration needs one; it is not a substitute for PrivateGPT authentication.
- Return clear login-required errors when session refresh cannot recover.

## Streaming, Cancellation and Resilience

SSE is the required streaming transport. The parser normalizes upstream events; the OpenAI mapper owns client-facing chunk framing.

| Event | Required behavior |
|---|---|
| Connection/429/5xx/timeout before first relayed chunk | Resilience layer may retry under configured, bounded policy. |
| First chunk relayed | Mark request non-retriable. |
| Upstream failure after first chunk | Finish the stream with a safe error path; never replay the request. |
| Client disconnect/cancel | Cancel generator, close upstream response, release connection and log redacted state. |
| Repeated upstream failures | Circuit breaker may fail fast for a bounded cooldown; it is local protection, not a distributed control plane. |

An implementation must use bounded buffering or backpressure for stream events. Exact timeout values, buffer size and breaker thresholds are configuration decisions to validate during pilot, not protocol constants in this appendix.

## Technology Constraints

| Concern | Required characteristic | Not prescribed yet |
|---|---|---|
| HTTP API | FastAPI/ASGI-compatible embedded server. | A particular ASGI runner package. |
| Upstream client | Shared async HTTP client with connection/time-out configuration. | Exact library or connection limits. |
| Validation | Typed request/response models. | A specific validation library version. |
| Logging | Structured, redacted logs with correlation ID. | External log platform. |
| Credentials | OS-backed secure storage or another approved local secure store. | Plaintext file or browser-profile scraping. |
| Resilience | Pre-stream retry and local circuit breaker boundary. | A third-party retry/breaker library. |

## Project Layout Guidance

```text
src/
  api/          # FastAPI routes, validation, response/SSE framing
  services/     # gateway and Agent lifecycle policy
  providers/    # SPI, registry, PrivateGPT implementation
  clients/      # reusable async HTTP transport
  auth/         # token manager and secure-store adapter
  schemas/      # OpenAI-compatible and unified contracts
  resilience/   # retry, cancellation and circuit-breaker policy
  config/       # local configuration loading and validation
  diagnostics/  # redacted health/status/doctor support
```

This is a role boundary, not a claim that every package already exists. The current source skeleton is expected to trail this target architecture.

## Verification and Runtime Evidence

| Area | Minimum evidence |
|---|---|
| Login | Browser/session flow, secure-store decision, redacted logs and refresh behaviour. |
| Agent | Model discovery, Agent create/update response, cache metadata and `metadata.agent_id` request capture. |
| API | Direct `/v1/models` and `/v1/chat/completions` smoke checks using redacted payloads. |
| SSE | Captured upstream event shape, mapper contract test, cancellation and no-retry-after-first-chunk test. |
| Boundary | Code review confirming no direct provider import from API/Agent layer and no tool executor. |
| Pilot | OpenCode non-destructive analysis plus a proposed small fix subject to OpenCode approval. |

Until those captures exist, internal PrivateGPT JSON schemas, exact event names, token flow details and timing defaults remain validation conditions rather than implementation facts.
