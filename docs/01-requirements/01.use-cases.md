# Use Cases — OpenCode PrivateGPT Bridge MVP

## Purpose

Define the actor goals and observable outcomes for the OpenCode PrivateGPT Bridge MVP. These use cases refine the canonical requirement into testable interactions while retaining OpenCode as the owner of workspace access, tool execution and approval.

## Context

The use cases cover the required local Bridge lifecycle: startup, browser login, workspace Agent synchronization, OpenCode chat completion, direct smoke test and pilot workflow. They do not add multi-model routing, LiteLLM, a central gateway, Bridge-side tool execution, IntelliJ runtime behavior or public API routes outside the MVP requirement.

| ID | Primary actor | Goal | Trigger | Success outcome | Requirement source |
|---|---|---|---|---|---|

| UC-002 | Developer | Authenticate or recover access to PrivateGPT. | `privategpt-adapter login` or login-required error. | Refresh token is stored securely; access token is cached in memory; login status is available. | FR002–FR003, §8.2, §10.2 |

## Main Flow

### UC-001 — Start local Bridge

**Preconditions:** local configuration and Python runtime exist; the Bridge can access its configured secure token store. Requirement does not specify the installation/provisioning workflow in detail.

1. Developer runs `privategpt-adapter start`.
2. Bridge loads configuration and checks token-store access, authentication status, model resolution and Agent status.
3. Bridge resolves the internal model ID for the only supported public alias, `gemini-2.5-pro`.
4. Bridge ensures the workspace Agent is synchronized as specified in UC-003.
5. Bridge starts the local OpenAI-compatible server at the configured localhost endpoint and prints the OpenCode provider endpoint.

**Success outcome:** OpenCode can use the local `/v1` endpoint. The Bridge has not accessed files, run commands or executed tools.

**Acceptance criteria:** starts without IntelliJ; exposes `/health`; defaults to loopback binding; rejects unsupported models; reports clear status/doctor results.

### UC-002 — Authenticate or recover PrivateGPT access

**Preconditions:** developer can invoke the Bridge login command; PrivateGPT browser login flow is reachable.

1. Developer runs `privategpt-adapter login` or receives a login-required/token-refresh-failed response.
2. Bridge opens browser login and receives an OAuth callback at localhost.
3. Bridge exchanges the authorization code for access and refresh tokens.
4. Bridge stores refresh token securely and retains access token in memory.
5. Bridge verifies PrivateGPT API reachability and reports login status.

**Success outcome:** Bridge can authenticate future PrivateGPT calls without printing token values.

**Acceptance criteria:** no manual token copy is required; refresh token is not plaintext; access token is absent from logs; a 401 causes at most one forced refresh/retry.

### UC-003 — Synchronize workspace Agent

**Preconditions:** valid login; internal Gemini 2.5 Pro model ID is resolved; Bridge knows the workspace context required to derive its workspace hash.

1. Developer runs `privategpt-adapter agent sync`, or Bridge reaches its Agent-sync stage during start/chat.
2. Bridge builds the OpenCode-compatible Agent instruction and computes its instruction hash.
3. Bridge lists current user Agents through PrivateGPT Agent API.
4. If the workspace Agent is missing, Bridge creates it. If the model ID or instruction hash changed, Bridge updates it. Otherwise it reuses the cached/current binding.
5. Bridge stores `agent_id`, model ID, instruction hash and update time in local metadata cache.

**Success outcome:** a synchronized Agent is available for conversation requests and the Bridge does not recreate it for every request.

**Acceptance criteria:** Agent instruction does not use IntelliJ ToolRegistry or ask for `<tool_call>`; Agent sync failure is explicit; inline-instruction fallback is prohibited.

### UC-004 — Obtain PrivateGPT chat completion through OpenCode

**Preconditions:** Bridge is running; login is valid; workspace Agent is synchronized; OpenCode is configured for `privategpt/gemini-2.5-pro`.

1. OpenCode sends `POST /v1/chat/completions` to Bridge with the supported model, messages and `stream` choice.
2. Bridge validates request/model, checks token state and ensures `agent_id` remains available.
3. Bridge preserves OpenCode system/developer messages, renders the request and builds PrivateGPT ChatRequest with `metadata.agent_id`, internal model ID and `tools: []`.
4. Bridge calls PrivateGPT Conversation API using Bearer authentication.
5. PrivateGPT returns an SSE response.
6. For `stream=false`, Bridge collects visible assistant content and returns an OpenAI-compatible response. For `stream=true`, Bridge maps and relays OpenAI-compatible chunks followed by `[DONE]`.
7. OpenCode receives the response and retains responsibility for any file, command or approval action.

**Success outcome:** OpenCode receives compatible assistant content from PrivateGPT without Bridge-side tool execution.

**Acceptance criteria:** every conversation has `metadata.agent_id`; unsupported model is rejected without fallback; plugin-style tool calls are not executed; client cancellation closes the upstream stream; a request is not retried after its first relayed chunk.

### UC-005 — Run direct HTTP smoke test

**Preconditions:** UC-001 and UC-002 have succeeded; developer has local adapter endpoint and local API key/configuration required by the smoke sample.

1. Developer sends the documented simple request to `/v1/chat/completions` with `gemini-2.5-pro` and `stream=false`.
2. Bridge validates the request and executes the UC-004 interaction.
3. Developer inspects the compatible response to the simple prompt.

**Success outcome:** direct HTTP path confirms Bridge can call PrivateGPT and return assistant content independently from a full OpenCode pilot.

**Acceptance criteria:** direct request succeeds for a simple prompt; the request is routed through the synchronized Agent; no non-OpenCode-client-specific logic is required in MVP.

### UC-006 — Run controlled OpenCode pilot

**Preconditions:** UC-001 through UC-004 have succeeded; OpenCode provider configuration is loaded and its permission policy is configured.

1. Developer selects `privategpt/gemini-2.5-pro` in OpenCode.
2. Developer asks OpenCode to analyze repository structure without editing.
3. OpenCode calls Bridge via UC-004 and returns analysis to developer.
4. Developer asks OpenCode to propose a small code fix.
5. OpenCode asks for approval before edit and before test/command execution according to its permission policy.
6. Developer reviews and approves or declines those OpenCode actions.

**Success outcome:** the MVP proves useful for a non-destructive analysis and a proposed small fix; Bridge remains an AI integration boundary rather than a coding agent.

**Acceptance criteria:** OpenCode can load provider config and obtain response; no unsupported provider feature is needed; Bridge does not execute tools; edit/bash permission remains with OpenCode.

## Alternate Flows

| Use case | Condition | Required behavior |
|---|---|---|
| UC-001 / UC-002 | Login is absent, expired or token refresh fails. | Bridge reports login-required/token-refresh-failed; developer performs login. |
| UC-001 / UC-003 | Internal Gemini 2.5 Pro model ID cannot be resolved. | Bridge reports model error and does not substitute another model. |
| UC-003 | Agent creation/update/list fails. | Bridge fails clearly; does not use inline instruction fallback. |
| UC-004 | Request model is unsupported. | Reject as `model_not_supported`. |
| UC-004 | OpenCode sends unsupported tool requirement. | Reject clearly when it cannot be safely ignored; Bridge never executes the tool. |
| UC-004 | PrivateGPT Agent emits plugin-style `<tool_call>`. | Return `privategpt_agent_instruction_mismatch`; do not execute it; Agent can be reset/synchronized. |
| UC-004 | PrivateGPT transient failure occurs before first relayed response chunk. | Bounded retry may occur before output begins. |
| UC-004 | Failure occurs after first relayed chunk. | Do not retry whole request; return/end stream through mapped error path. |
| UC-004 | OpenCode/client cancels or disconnects. | Close upstream stream and release connection; do not leave background work running. |
| UC-006 | Developer declines OpenCode edit or command approval. | OpenCode does not perform the declined action; Bridge has no alternate tool action. |

## Các điểm cần làm rõ

1. OpenCode request shape thực tế có bắt buộc native tool calls, `stream=true`, hoặc tool result messages không?
2. PrivateGPT Agent “ready” status, model-discovery schema, Agent API schema, ChatRequest schema và SSE event schema cụ thể là gì?
3. Cơ chế secure token store nào được phê duyệt cho môi trường developer mục tiêu?
4. Điều kiện, ngưỡng và thông điệp người dùng mong muốn cho timeout, upstream unreachable, rate limit, circuit breaker và stream failure sau chunk đầu tiên là gì?
5. Chính sách gửi source code nội bộ tới PrivateGPT, retention và audit/logging là gì?
6. Các flow `install`, `stop`, `restart`, `logout` và `agent reset` cần được chấp nhận như use case độc lập hay chỉ là runbook/alternate flow?
