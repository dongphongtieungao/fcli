# Use Cases

## Purpose

Provide a reusable framework for turning requirements into actor goals, flows, acceptance criteria, and verification targets.

## Context

Use cases bridge requirements and behavior. They should be generic until the project has real actors and workflows, but the structure below should remain stable.

## Use Case Principles

| Principle | Rule |
|---|---|
| Actor-centered | Each use case starts with a user, system, operator, or external actor. |
| Goal-oriented | The goal must describe the outcome, not an implementation detail. |
| Flow-aware | Happy path, alternative paths, and error paths should be separated. |
| Acceptance-linked | Each meaningful flow should map to acceptance criteria or a verification gap. |
| Traceable | Use cases should link back to requirements and forward to BPMN/specs/tests. |

## Use Case Record Shape

| Field | Description |
|---|---|
| ID | Stable use case identifier. |
| Actor | Primary actor or system role. |
| Goal | Outcome the actor needs. |
| Preconditions | Required state before the flow starts. |
| Main flow | Ordered steps for the normal path. |
| Alternate/error paths | Branches, validation failures, recovery, or escalation. |
| Acceptance criteria | Observable checks or expected evidence. |
| Related artifacts | Requirement, BPMN, contract, QA, or test links. |

## Promotion Rules

Promote a use case when it clarifies accepted behavior, expected workflow, or acceptance criteria. If the flow is still uncertain, record it as draft and link open questions instead of inventing missing behavior.
