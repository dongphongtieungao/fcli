# Review Checklists — OpenCode PrivateGPT Bridge

## Purpose

Provide actionable PR and design-review checklists that trace changes back to specification, test evidence and quality gates.

## Context

Apply sections according to the change-risk mapping in the Quality Matrix. A checked box means the reviewer inspected evidence; it is not a substitute for a test result.

## PR intake

- [ ] Objective, affected behavior and risk are stated.
- [ ] Files in/out of scope are clear; unrelated user changes are preserved.
- [ ] Requirement, Application Spec/OpenAPI and BDD sources are linked.
- [ ] Applicable QA rows and quality gates are listed.
- [ ] Open questions are separated from confirmed behavior.

## Specification and traceability review

- [ ] Behavior maps to a requirement/use case and BPMN flow.
- [ ] Main-flow changes have a happy-path scenario.
- [ ] Critical failure/control changes have Given/When/Then coverage.
- [ ] Application Spec and OpenAPI agree on routes, model, auth, response mode and limitations.
- [ ] No reference plugin observation was promoted as product truth without approval.
- [ ] Manifest/source catalog changes are included when canonical role/status/path changed.

## Architecture and Python boundary review

- [ ] Local modular-monolith boundaries match C4/standards.
- [ ] API/services depend on Provider SPI, not raw PrivateGPT clients.
- [ ] Async network/stream paths do not block the event loop.
- [ ] No IntelliJ runtime, AgentLoopRunner or ToolRegistry dependency entered the Bridge.
- [ ] No unrequested gateway, provider/model routing or new public endpoint was added.
- [ ] Configuration and dependency changes are justified and reversible.

## API and contract review

- [ ] Only `/health`, `/v1/models` and `/v1/chat/completions` are exposed unless scope was approved.
- [ ] Only `gemini-2.5-pro` is public; internal model ID is explicit; no fallback.
- [ ] Request validation occurs before upstream calls.
- [ ] Every conversation has synchronized `metadata.agent_id` and upstream `tools: []` for confirmed MVP.
- [ ] System/developer/user content is preserved according to the application spec.
- [ ] Non-stream and SSE responses match OpenAPI; SSE terminates `[DONE]`.
- [ ] Error fields/statuses are not treated as stable where contract marks them open.
- [ ] Breaking compatibility and migration impact are documented.

## Agent and tool-boundary review

- [ ] Agent sync is workspace-scoped and idempotent.
- [ ] Agent create/update/reuse behavior and cache metadata are tested.
- [ ] Instructions do not request plugin-style `<tool_call>`.
- [ ] Bridge cannot execute read/edit/bash or other tools.
- [ ] Unsupported tool payload/output is rejected or handled only as approved.
- [ ] OpenCode remains the owner of permissions and approvals.

## File/context review

- [ ] Text file content supplied by OpenCode is treated as message content.
- [ ] Bridge does not open a selected workspace path itself.
- [ ] Text already in messages is not uploaded to PrivateGPT.
- [ ] `metadata.attachments` remains empty for the confirmed text flow.
- [ ] Binary/structured attachments remain rejected/unsupported unless requirement, OpenAPI, BDD and data controls were approved together.
- [ ] Any future upload path defines consent, MIME/size limits, conversation lifecycle, retention and cleanup.

## Authentication, security and data review

- [ ] Refresh token uses approved secure storage; access token is memory-only.
- [ ] Local API key/token/cookie is absent from code, examples and logs.
- [ ] Default bind is loopback, not `0.0.0.0`.
- [ ] A 401 causes at most one forced refresh/retry.
- [ ] Logs/errors redact raw prompts, source code, credentials and internal stack details.
- [ ] Fixtures are synthetic/redacted and approved for repository use.
- [ ] Source-code transmission/retention risk is recorded when context handling changes.

## Streaming and resilience review

- [ ] Retry can occur only before first relayed chunk.
- [ ] Failure after first chunk does not restart the whole request.
- [ ] Client cancellation closes upstream stream/connection.
- [ ] Timeout, response-size and buffering limits are enforced/configurable.
- [ ] Reasoning markers are filtered without losing visible content.
- [ ] Stream completion/error behavior is tested with approved fixtures.

## Test and evidence review

- [ ] Applicable QA-F/QA-N rows have tests or approved manual evidence.
- [ ] Exact commands, exit codes, warnings/errors and artifacts are recorded.
- [ ] Contract/YAML/Markdown parsers ran for changed specs.
- [ ] Unit tests cover validation, mapping, Agent policy, errors and redaction as affected.
- [ ] Integration tests cover affected PrivateGPT boundary calls.
- [ ] Runtime/OpenCode compatibility claims include redacted captures.
- [ ] Skipped/flaky/missing checks lower the verdict appropriately.

## Documentation and KMS review

- [ ] Frontmatter, exactly one H1, Purpose/Context and related links are valid.
- [ ] Requirement, BPMN, BDD, contract, QA and review chain remains navigable.
- [ ] Durable decisions are in official docs; observations remain workspace/reference evidence.
- [ ] Manifest, YAML registry and sources catalog agree.
- [ ] No stale placeholder or contradictory claim remains in the changed surface.

## Review decision

| Decision | Condition |
|---|---|
| `APPROVE` | All applicable blocking gates pass with sufficient evidence. |
| `APPROVE_WITH_GAPS` | Only explicit non-blocking gaps remain with owner/next action. |
| `REQUEST_CHANGES` | A test, contract, security, boundary or traceability gate fails. |
| `BLOCKED` | Required source, environment, schema or decision is missing. |
| `NEEDS_HUMAN_REVIEW` | Business/security/data/release approval is required. |

Reviewer records: applicable checklist sections, QA/gate IDs, evidence links, unresolved findings and final decision.
