# Review Checklists

## Purpose

Provide reusable review checklists for documentation, code, contracts, runtime behavior, data safety, and release readiness.

## Context

Use these checklists until the new project adds module-specific review criteria. Scale the checklist to the risk of the task; do not require release checks for tiny edits.

## General Review

- [ ] Scope is clear and matches the request.
- [ ] User changes were preserved.
- [ ] No secrets or sensitive data were introduced.
- [ ] Tests or verification are appropriate for the change.
- [ ] Documentation impact was considered.
- [ ] Residual risks are reported.

## Documentation Review

- [ ] Frontmatter is present where expected.
- [ ] Related links are real.
- [ ] Source-of-truth claims are supported.
- [ ] Original meaning changed only when intended.

## Contract And Runtime Review

- [ ] API/application/schema/config/output changes are reflected in the correct contract.
- [ ] Runtime command, environment, and configuration assumptions are documented.
- [ ] Warnings, errors, generated artifacts, and logs were inspected when checks ran.
- [ ] Backward compatibility or migration impact is recorded.

## Data And Security Review

- [ ] No secrets, credentials, tokens, or sensitive data were added.
- [ ] Fixtures and examples are synthetic, anonymized, or approved.
- [ ] Logs and release evidence avoid sensitive values.
- [ ] Access, retention, or deletion impact is recorded when data handling changes.

## Release Review

- [ ] Package/build scope is clear.
- [ ] Install or upgrade path is documented.
- [ ] Smoke test evidence is available or the gap is explicit.
- [ ] Known issues, rollback, and handover owner are recorded.
- [ ] DoD verdict is supported by evidence.

## Promotion Rules

Promote new checklist items when reviewers need them repeatedly. Keep task-specific review comments in task notes unless they become reusable policy.
