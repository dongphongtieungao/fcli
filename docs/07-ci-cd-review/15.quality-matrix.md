# Quality Matrix — CI and PR Gates

## Purpose

Translate the QA matrix into enforceable CI/PR gates so changes cannot bypass the specification, security boundaries or required evidence.

## Context

The repository does not yet demonstrate a complete automated CI pipeline or application test harness. This matrix defines required gates and their promotion criteria. `automation pending` means the gate is known but must not be reported as passing automatically.

## Quality gates

| Gate | Requirement | Applies when | Check/evidence | Blocking condition | Automation |
|---|---|---|---|---|---|
| QG-01 Traceability | Changed behavior maps requirement → spec → test. | Docs, contract, behavior changes. | IDs/links and affected QA rows. | Missing source or acceptance mapping. | Review available. |
| QG-02 Documentation/KMS | Official docs and registries remain valid. | Docs/registry changes. | Frontmatter, one H1, links, YAML/source IDs. | Parse error, orphan or stale registry. | Partial. |
| QG-03 Contract integrity | Application/OpenAPI schemas parse and match supported scope. | API/config/interface changes. | OpenAPI parser/linter/contract tests. | Parse failure or undocumented breaking change. | Parser available; linter pending. |
| QG-04 Unit behavior | Validation, mapping, policy, SSE and redaction are tested. | Python behavior changes. | Targeted unit results. | Applicable test failure/missing critical test. | Pending implementation. |
| QG-05 Integration | Provider auth/model/Agent/conversation behavior is verified. | Provider/service changes. | Approved fixtures/mocks and results. | Missing Agent/auth/stream evidence. | Pending. |
| QG-06 Security/data | No secret/raw source leak or Bridge-side tool/file action. | Every PR; stronger for auth/data. | Secret scan, dependency/log review. | Secret, unsafe file/tool path, unredacted output. | Pending. |
| QG-07 Runtime resilience | Startup, stream, retry and cancellation work. | Runtime/stream changes. | Smoke/log/cancellation evidence. | Retry-after-chunk, orphan stream or unsafe bind. | Pending. |
| QG-08 BDD acceptance | Affected scenarios remain covered. | Business behavior changes. | Feature mapping and automated/manual evidence. | Main flow without happy path or critical scenario gap. | Specs exist; runner pending. |
| QG-09 OpenCode compatibility | Real client provider/model/request is compatible. | Provider/request/tool/file changes. | Redacted OpenCode capture/pilot. | Claimed compatibility without evidence. | Manual/pending. |
| QG-10 Review/DoD | Checklist, warnings, risks and verdict are complete. | Every PR/handover. | Reviewer checklist and evidence pack. | Unresolved blocking finding or unsupported DONE claim. | Review available. |

## Required gates by change type

| Change type | Minimum gates |
|---|---|
| Markdown wording/link only | QG-01, QG-02, QG-10 |
| Requirement/BPMN/BDD change | QG-01, QG-02, QG-08, QG-10 |
| Application/OpenAPI/config contract | QG-01, QG-02, QG-03, QG-08, QG-10 |
| Python validation/mapping | QG-01, QG-04, QG-08, QG-10 |
| Auth/token/security | QG-01, QG-04, QG-05, QG-06, QG-10 |
| Agent/provider integration | QG-01, QG-04, QG-05, QG-08, QG-10 |
| SSE/retry/cancellation | QG-01, QG-04, QG-05, QG-07, QG-08, QG-10 |
| OpenCode tools/file compatibility | QG-01, QG-03, QG-05, QG-06, QG-08, QG-09, QG-10 |
| Release/handover | All applicable gates plus DoD/runtime smoke/rollback evidence |

## Proposed CI stages

| Stage | Inputs | Required checks | Output artifact |
|---|---|---|---|
| `spec` | Markdown, manifests, YAML, features | Structure, links, YAML/OpenAPI parse, trace IDs. | Spec validation log. |
| `static` | Python/config | Formatting/lint/import/dependency/secret checks once configured. | Static analysis report. |
| `unit` | Python modules | Validation, mapping, Agent policy, SSE, resilience, redaction. | Unit test report. |
| `contract` | OpenAPI/application/provider fixtures | Schema and request/response contract tests. | Contract report/fixtures. |
| `integration` | Approved mocks/test environment | Auth/model/Agent/conversation and cancellation. | Integration report/redacted logs. |
| `acceptance` | BDD and runtime | Scenario runner/direct smoke/OpenCode pilot as appropriate. | Acceptance evidence pack. |
| `review` | Diff and prior artifacts | Checklist, risk, warnings, DoD verdict. | PR review decision. |

Stage names are a target contract, not proof that workflows currently exist.

## Gate outcomes

| Outcome | CI/PR action |
|---|---|
| `PASS` | Required evidence is complete for the scoped gate. |
| `PASS_WITH_GAP` | Non-blocking limitation recorded with owner/next action. |
| `FAIL` | Merge blocked until evidence passes. |
| `BLOCKED` | Merge blocked pending source/environment/decision. |
| `NEEDS_HUMAN_REVIEW` | Authorized reviewer decision required. |

## Blocking rules

Always block merge for: exposed secrets; model fallback; missing Agent binding; Bridge tool execution; unapproved filesystem/upload behavior; retry after first chunk; cancellation leak; broken contract parser; missing critical BDD mapping; or full compatibility claims without runtime evidence.

## Promotion criteria

A pending gate becomes automated only after its command, environment, fixtures, expected warnings and output artifact are committed and referenced by the QA matrix. Adding a workflow name without executable evidence does not satisfy the gate.
