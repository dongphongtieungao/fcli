# Quality Matrix

## Purpose

Define reusable quality gates for project changes, reviews, CI, release, and documentation promotion.

## Context

The repository may start without active application code. These gates describe what should be evaluated as code, contracts, runtime workflows, and release artifacts appear.

## Quality Gates

| Gate | Required When | Method | Evidence |
|---|---|---|---|
| Source traceability | Any durable docs, behavior, or contract change | Source/related link review | Source IDs and related docs |
| Secret and data safety | Any config, fixture, log, prompt, release evidence, or example data | Review or scanner | No exposed secrets/PII |
| Contract integrity | API, application, schema, output, or config contract changes | Parser, linter, or contract test | Exact command/result |
| Behavior verification | Code or runtime behavior changes | Smallest useful test or smoke check | Exit code, logs, artifacts |
| Documentation quality | Official Markdown changes | Metadata/link/structure review | Frontmatter, one H1, Purpose/Context |
| Release readiness | Package, install, deploy, or handover changes | Build/install/smoke/rollback review | Evidence pack and known issues |
| Review completeness | Risky or user-facing changes | Checklist review | Reviewer notes or open gaps |

## Risk Scaling

| Risk | Examples | Required Gate Set |
|---|---|---|
| Low | Typo, small docs cleanup, comments | Traceability and structure check |
| Medium | Contract, config, prompt, docs registry, non-runtime code | Traceability, parser/lint, targeted review |
| High | Business behavior, security, data, runtime entry point, release | QA matrix, tests/smoke, review checklist, DoD |

## Promotion Rules

Promote a gate into this file when it should apply across tasks or releases. Keep one-off verification details in `verification-notes.md` unless they become reusable policy.
