# Non-Functional Requirements

## Purpose

Provide a reusable non-functional requirements framework for quality attributes, constraints, and verification expectations.

## Context

No project-specific NFRs are confirmed yet. This file defines the categories and evidence shape that should be used when the project introduces real constraints.

## NFR Categories

| Category | Reusable Question | Evidence |
|---|---|---|---|
| Security | What must be protected, redacted, authenticated, or authorized? | Threat note, review checklist, security test |
| Privacy | What personal or sensitive data can appear in storage, logs, fixtures, or release evidence? | Data governance, redaction rule |
| Reliability | What failures must be tolerated or recovered? | Runbook, smoke test, retry policy |
| Performance | What latency, throughput, memory, or data-size expectations matter? | Benchmark, budget, monitoring |
| Operability | How will the system be configured, observed, started, stopped, and debugged? | Runbook, logs, health check |
| Maintainability | What code, documentation, and review constraints keep the system changeable? | Standards, review checklist |
| Portability | What runtime, OS, deployment, or dependency constraints matter? | Environment setup, package docs |
| Compliance | What legal, license, audit, or retention expectations apply? | Governance, release evidence |

## NFR Record Shape

| Field | Description |
|---|---|
| ID | Stable NFR identifier. |
| Category | Quality category from this file or project-specific extension. |
| Statement | Testable constraint or quality expectation. |
| Rationale | Why the constraint matters. |
| Verification | Command, review, test, audit, or human approval method. |
| Status | Draft, reviewed, published, deprecated. |

## Promotion Rules

Promote an NFR when it constrains architecture, data governance, implementation, release, or operations. If it cannot be verified yet, record the verification gap and link it to the QA matrix or DoD.
