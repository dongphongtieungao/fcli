# Non-Functional Requirements — OpenCode PrivateGPT Bridge MVP

## Purpose

Define the quality constraints that govern the local Python bridge. These constraints complement the functional requirement; they do not assert that PrivateGPT runtime schemas, timing limits or login mechanics have already been verified.

## NFR Register

| ID | Category | Requirement | Verification evidence | Status |
|---|---|---|---|---|
| NFR-SEC-001 | Local exposure | Bind to `127.0.0.1` by default. LAN exposure requires an explicit security decision. | Startup/config test and listener inspection. | published |
| NFR-SEC-002 | Credential protection | Do not hardcode, return or log tokens, cookies, passwords, raw prompts or raw source code. Store refresh/session material only in an approved secure local store; cache access tokens in memory. | Secret/log review, secure-store test and redacted-log inspection. | published |
| NFR-SEC-003 | Tool isolation | Adapter must not read/write workspace files, run shell commands or execute model/plugin tool calls. | Dependency/code review and OpenCode pilot evidence. | published |
| NFR-REL-001 | Agent integrity | Every conversation uses a synchronized workspace Agent binding. Agent-sync failure is explicit; inline-instruction or model fallback is prohibited. | Agent create/update capture and `metadata.agent_id` request evidence. | published |
| NFR-REL-002 | Streaming safety | Relay SSE incrementally. Retry a conversation only before the first client-facing chunk; after that point, do not replay the request. | Stream mapper test, failure-before/after-first-chunk test. | published |
| NFR-REL-003 | Cancellation | Client cancellation/disconnect closes the upstream response and releases resources without detached work. | Cancellation integration test and redacted diagnostic log. | published |
| NFR-REL-004 | Failure containment | Use bounded, configurable retry and a local circuit-breaker boundary for transient upstream failures. Do not hide auth, validation or Agent errors. | Resilience policy test and error-mapping review. | published |
| NFR-PERF-001 | Resource use | Use asynchronous upstream I/O and bounded stream buffering/backpressure. Timeout, buffer and connection values are configuration validated during pilot, not fixed protocol facts. | Load/smoke observation and configuration review. | published |
| NFR-OPS-001 | Operability | Provide health, login/status, doctor and redacted diagnostics sufficient to distinguish local configuration, session, model, Agent and upstream failures. | CLI/API smoke checks and operator review. | published |
| NFR-MAIN-001 | Changeability | API/Agent layers depend on Provider SPI, not a concrete provider or raw PrivateGPT HTTP client. Keep Python modules separated by C4 ownership. | Import/dependency review and targeted tests. | published |
| NFR-VER-001 | Evidence discipline | A PrivateGPT model ID, Agent schema, conversation schema, SSE event shape or OpenCode payload is a validation condition until captured redacted evidence confirms it. | Evidence pack reviewed before pilot/release claim. | published |

## Quality Boundaries

- The MVP has one formal client (OpenCode), one public model alias (`gemini-2.5-pro`) and one implemented provider (`PrivateGPTProvider`).
- No availability, throughput or latency numeric SLO is published yet because the upstream model, network and streaming behavior need pilot evidence.
- WebSocket, public Agent routes, `/v1/responses`, distributed queues and centralized deployment are not NFR commitments for this MVP.

## Verification Use

For a code or configuration change, select the relevant NFR rows and record the smallest useful evidence in the QA Matrix. A passing unit test does not verify a runtime condition that needs a PrivateGPT or OpenCode capture.
