# Glossary

## Purpose

Define reusable rules for maintaining shared architecture and technical terminology.

## Context

Add terms only when they affect requirements, architecture, code, contracts, operations, or verification. Keep business-only wording in the business glossary unless it changes architecture or implementation.

## Term Rules

| Rule | Description |
|---|---|---|
| Define once | Prefer one canonical definition and link related aliases. |
| Cite source | Link terms to requirement, ADR, BPMN, code boundary, or user decision. |
| Separate domains | Keep technical terms here and business workflow terms in business glossary. |
| Mark status | Draft terms must not be used as accepted constraints. |
| Avoid jargon drift | Rename or deprecate terms when they stop matching implementation or docs. |

## Term Record Shape

| Field | Description |
|---|---|
| Term | Canonical name. |
| Definition | Clear meaning in this project. |
| Source | Requirement, ADR, code boundary, decision, or external standard. |
| Related | Synonyms, business terms, modules, contracts, or runbook links. |
| Status | Draft, reviewed, published, deprecated. |

## Alias And Boundary Rules

- `PrivateGPT Adapter`, `OpenCode PrivateGPT Bridge`, and `PA` refer to the same local adapter service; **PrivateGPT Adapter** is the canonical technical name.
- `OpenAI-compatible API` must not be called `PrivateGPT API`; the latter refers only to internal endpoints.
- `Agent Mode` does not grant the adapter permission to edit files or run commands.
- `Gemini 2.5 Pro` identifies the MVP model alias; it must not imply a fallback to another model.

## Promotion Rules

Promote a term when misunderstanding it could change code, architecture, tests, operations, or release behavior. Keep exploratory wording in task notes until reviewed.
