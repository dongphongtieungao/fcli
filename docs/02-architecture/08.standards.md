# Engineering Standards — OpenCode PrivateGPT Bridge MVP

## Purpose

Define the coding, security, contract and verification standards that preserve the architecture described by C4, NFR and ADR. They apply to implementation and review work for this MVP.

## Standards Register

| ID | Standard | Verification |
|---|---|---|
| STD-001 | Implement the adapter in Python using explicit module boundaries from C4: API, services/Agent policy, Provider SPI, provider implementation, auth, resilience and diagnostics. | Dependency/import review. |
| STD-002 | Keep API routes and Agent services dependent on Provider SPI abstractions; do not import `PrivateGPTProvider` or raw PrivateGPT HTTP details directly above the provider boundary. | Dependency review and targeted tests. |
| STD-003 | Use asynchronous I/O for PrivateGPT calls and streaming. Do not block the event loop with synchronous network/file operations on request paths. | Code review and async test/smoke evidence. |
| STD-004 | Validate the public model whitelist and OpenAI-compatible request before external calls. Only `gemini-2.5-pro` is supported; no model fallback. | Unit/contract test. |
| STD-005 | Agent synchronization is idempotent per workspace. Store only safe metadata (`agent_id`, internal model ID, instruction hash, timestamps); do not persist full chat history by default. | Agent-sync test and storage review. |
| STD-006 | Use typed, redacted error mapping. Never leak credentials, cookies, raw prompts, raw source code or internal stack details through API responses or logs. | Error/log review and test. |
| STD-007 | Store refresh/session material only in an approved secure local store and keep access tokens memory-only. Do not hardcode secrets or commit `.env` values. | Secret scan and secure-store test. |
| STD-008 | For SSE, frame client output in the OpenAI-compatible mapper; retry only before first relay and close upstream stream on cancellation. | Stream mapper and cancellation tests. |
| STD-009 | Add correlation/request IDs, route, provider, status, latency and resilience state to diagnostics. Log safe metadata only. | Log inspection. |
| STD-010 | Treat unknown PrivateGPT/OpenCode schema details as validation conditions. Add redacted runtime capture and tests before declaring a capability implemented. | Evidence pack review. |
| STD-011 | Every behavior or contract change updates the relevant requirement, C4/NFR/ADR, tests and source registry when its canonical role changes. | Traceability and registry review. |
| STD-012 | Keep changes scoped and preserve unrelated user work; use reversible edits and inspect diff before handoff. | Diff review. |

## Review Gates

Before implementation review, confirm: required NFR rows map to tests/evidence; no API surface has expanded without parent-requirement approval; no tool executor or direct workspace capability entered the adapter; and no draft observation is used as a verified contract.
