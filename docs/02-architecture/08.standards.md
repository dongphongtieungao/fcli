# Standards

## Purpose

Define reusable engineering, documentation, security, verification, and release standards.

## Context

Project-specific standards should be added when scope and stack are known. Until then, these standards define the minimum expected behavior for agent and developer work.

## Standards

| ID | Standard | Verification |
|---|---|---|
| STD-001 | Do not commit secrets or sensitive data. | Review or scan |
| STD-002 | Keep official docs linked to real sources. | Documentation review |
| STD-003 | Add verification evidence for behavior changes. | Test/evidence review |
| STD-004 | Keep changes scoped to the requested task and preserve unrelated user edits. | Diff review |
| STD-005 | Keep contracts parseable when API, application, schema, or config files change. | Parser or lint |
| STD-006 | Record promotion status when workspace knowledge becomes official. | `promotion.md` review |
| STD-007 | Do not replace reusable registry docs with empty placeholders. | Manifest/source review |

## Promotion Rules

Promote a standard when it should apply across future implementation, review, release, or documentation tasks. Link new standards to QA, review checklist, and DoD when they create a gate.
