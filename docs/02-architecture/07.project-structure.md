# Project Structure

## Purpose

Define reusable repository layout, ownership, and boundary rules.

## Context

The current repository is a starter structure. Paths may be added or removed by the new project, but each durable path should have a clear role.

## Structure

| Path | Role |
|---|---|
| `src/` | Application source |
| `tests/` | Tests and executable examples |
| `docs/` | Official project knowledge |
| `llm-wiki/` | Workspace-only agent notes |
| `specs/` | Machine-readable contracts |
| `tools/` | Helper scripts |

## Boundary Rules

| Rule | Description |
|---|---|
| Docs are official | `docs/` stores source-of-truth artifacts and reusable methodology. |
| Workspace is temporary | `llm-wiki/` stores observations and task memory; it does not override docs. |
| Specs are machine-readable | `specs/` stores contracts and schemas that can be parsed or linted. |
| Generated output is controlled | Generated artifacts should have retention, evidence, or ignore rules. |
| Ownership is explicit | New top-level paths should be documented here or in architecture docs. |

## Promotion Rules

Update this file when a new durable top-level folder, module category, artifact class, or ownership boundary is introduced.
