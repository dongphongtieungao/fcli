# Architecture C4

## Purpose

Provide a reusable C4 architecture framework for system context, containers, components, boundaries, and diagrams.

## Context

The project-specific architecture is not defined yet. Keep this file as the architecture entrypoint and store generated diagrams under `docs/02-architecture/diagrams/`.

## C4 Levels

| Level | Question | Evidence |
|---|---|---|
| Context | Who uses the system and what external systems matter? | Vision, use cases, contracts |
| Container | What deployable/runtime parts exist? | Architecture decision, project structure |
| Component | What internal components own important responsibilities? | Code boundary, module docs, ADR |
| Code | Which code areas are important enough to document? | File role snapshot, tests |

## Boundary Rules

| Rule | Description |
|---|---|
| Name owners | Each important container/component should have an owner or `unknown`. |
| Show dependencies | External systems and critical internal dependencies should be explicit. |
| Link decisions | Architectural trade-offs should link to ADRs. |
| Link quality | NFRs should constrain relevant components or runtime paths. |
| Avoid diagram drift | Update diagrams when boundary or dependency meaning changes. |

## Diagram Policy

`docs/02-architecture/diagrams/` is intentionally kept as a scaffold. Add diagrams only when they represent current architecture. Generated images should not be committed unless they are intended review artifacts.

## Promotion Rules

Promote architecture updates when a boundary, dependency, runtime mode, data flow, deployment shape, or integration becomes durable project knowledge.
