# Architecture Decision Records — OpenCode PrivateGPT Bridge MVP

## Purpose

Record the durable decisions already made by the canonical requirement and published appendices. This index is not permission to add dependencies, deployment modes or provider capabilities that the parent requirement has not approved.

## Decision Index

| ADR | Decision | Status | Source |
|---|---|---|---|
| ADR-001 | Run the adapter as a local Python modular monolith with embedded FastAPI/ASGI server. | accepted | Vision; Architecture-design appendix; C4 |
| ADR-002 | Keep OpenCode as the sole coding-agent runtime; adapter has no filesystem, shell, approval or tool executor. | accepted | Primary requirement; Business requirements; C4 |
| ADR-003 | Use PrivateGPT Agent API and cache one Agent binding per workspace; include `metadata.agent_id` on conversation requests. | accepted | Primary requirement FR005–FR008; Architecture v2 |
| ADR-004 | Expose only `gemini-2.5-pro` and do not fallback to another model. | accepted | Primary requirement FR004; Business requirements |
| ADR-005 | Preserve Provider SPI; implement only `PrivateGPTProvider` in the MVP. | accepted | Provider SPI appendix; C4 |
| ADR-006 | Use OpenAI-compatible `/v1/models` and `/v1/chat/completions` plus health as the MVP surface; do not commit public Agent routes, `/v1/responses` or WebSocket. | accepted | Primary requirement; Architecture-design appendix |
| ADR-007 | Use SSE relay; retry only before the first relayed chunk and cancel/close upstream work on client disconnect. | accepted | Primary requirement FR009–FR010; Provider SPI appendix |
| ADR-008 | Use browser login with approved local secure storage; no plaintext credential persistence or browser-profile scraping in MVP. | accepted | Primary requirement FR002–FR003; Architecture-design appendix |

## Consequences and Follow-up

| Decision area | Consequence | Required follow-up |
|---|---|---|
| PrivateGPT integration | Internal model, Agent and SSE schemas cannot be invented from these ADRs. | Capture redacted runtime evidence before implementation/pilot acceptance. |
| Provider extension | A new provider or capability changes the contract surface. | Add a new ADR, capability contract and contract tests. |
| Public API expansion | New OpenAI routes can affect OpenCode compatibility and security. | Change parent requirement and API contract first. |
| Credentials | Secure-store option depends on developer platform and approved login flow. | Make and verify a platform-specific implementation decision before pilot. |

## Decision Procedure

Create a new ADR when a future change modifies a row above or introduces a new deployment, dependency, persistence, security, client or provider contract. Link it to C4, NFR, Standards and verification evidence. Do not use this document to promote an unverified PrivateGPT observation into an accepted decision.
