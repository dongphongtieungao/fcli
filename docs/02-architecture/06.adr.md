# Architecture Decision Records

## Purpose

Provide a reusable ADR index and decision-recording framework.

## Context

No project-specific architecture decisions are approved yet. Use this file to decide when a decision should become an ADR and how it should be linked.

## ADR Triggers

| Trigger | Example |
|---|---|---|
| Boundary change | New container, component, package, service, or ownership split |
| Dependency choice | New framework, database, package, service, or integration |
| Data/storage decision | Persistence, retention, migration, schema, or sensitive data handling |
| Runtime decision | Deployment, job orchestration, config, logging, or recovery behavior |
| Quality trade-off | Security, performance, reliability, maintainability, or cost |

## ADR Record Shape

| Field | Description |
|---|---|
| ID | Stable ADR identifier. |
| Context | Problem and constraints. |
| Decision | Chosen option. |
| Alternatives | Options considered. |
| Consequences | Benefits, risks, and follow-up work. |
| Status | Draft, accepted, superseded, deprecated. |
| Links | C4, standards, contracts, runbook, tests, or release evidence. |

## Decision Index

| ADR | Decision | Status |
|---|---|---|
| ADR-000 | ADR framework active; project-specific decisions not recorded yet | draft |

## Promotion Rules

Promote a decision into ADR when it changes future implementation choices or review criteria. Do not hide durable architecture decisions only in chat or task notes.
