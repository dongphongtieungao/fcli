# Business Glossary

## Purpose

Define reusable rules for business terminology, actor language, workflow vocabulary, and acceptance wording.

## Context

The published domain is the OpenCode PrivateGPT Bridge MVP. This glossary prevents actor, MVP, pilot, approval, and acceptance language from leaking ambiguously into requirements, contracts, tests, or code.

## Term Rules

| Rule | Description |
|---|---|---|
| Use business language | Define terms as stakeholders use them before translating to technical terms. |
| Link behavior | Terms that affect workflow or validation should link to BPMN or requirements. |
| Separate aliases | Record synonyms and deprecated labels to avoid duplicate concepts. |
| Record ownership | Important terms should have a business owner or `unknown`. |
| Avoid premature certainty | Draft terms must not be used as accepted business rules. |

## Term Record Shape

| Field | Description |
|---|---|
| Term | Canonical business term. |
| Definition | Meaning in stakeholder language. |
| Owner | Business owner or `unknown`. |
| Related behavior | Requirement, use case, BPMN, contract, or test. |
| Status | Draft, reviewed, published, deprecated. |

## OpenCode PrivateGPT Bridge Business Terms

| Term | Definition | Owner | Related behavior | Status |
|---|---|---|---|---|
| Developer | The person who runs the local adapter and OpenCode in a source repository. | unknown | Browser login, local configuration, OpenCode pilot | published |
| OpenCode user | A developer using OpenCode as the MVP coding client through the configured PrivateGPT provider. | unknown | Model selection, repository analysis, proposed code fix | published |
| OpenCode PrivateGPT Bridge MVP | The first deliverable limited to a localhost Python adapter, one supported model, Agent API use, and OpenCode compatibility. | unknown | In-scope and out-of-scope definitions | published |
| Pilot | The controlled validation of the adapter with OpenCode before wider use on real code changes. | unknown | OpenCode pilot tests, request capture spike | published |
| Browser login | The user-initiated authentication journey that opens a browser, completes identity-provider login, and establishes secure local token use. | unknown | FR002, token refresh | published |
| Agent synchronization | The user-visible readiness step that ensures an OpenCode-compatible PrivateGPT Agent is created or updated for the workspace. | unknown | FR005, `privategpt-adapter agent sync` | published |
| Supported model | The only model the MVP may expose and accept: `gemini-2.5-pro`. | unknown | FR004, model whitelist | published |
| Unsupported model request | A request for any model other than the supported model; it must be rejected rather than silently redirected. | unknown | FR004, error handling | published |
| Non-destructive analysis task | An OpenCode task that reads or explains repository information without editing files or running unapproved commands. | unknown | Pilot acceptance criterion | published |
| Approval | The user decision required by OpenCode before an edit or test command is performed. The adapter does not grant or execute that approval. | unknown | Tool execution boundary, pilot tests | published |
| Acceptance criterion | An observable condition for accepting the MVP, such as successful login, agent sync, model discovery, request mapping, or OpenCode connectivity. | unknown | Requirement §§12–13 | published |

## Promotion Rules

Promote terms when they clarify accepted requirements, workflow decisions, validations, or user-facing language. Keep uncertain vocabulary in task notes until reviewed.
