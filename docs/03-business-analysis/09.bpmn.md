# BPMN

## Purpose

Provide a reusable business process framework for mapping use cases into workflows, decisions, exceptions, and acceptance evidence.

## Context

No project-specific business process is defined yet. This file defines how future processes should be captured without inventing domain behavior.

## Process Modeling Rules

| Rule | Description |
|---|---|
| Start from use cases | BPMN should clarify accepted use cases, not replace requirements. |
| Name actors and systems | Lanes or roles should identify who performs each step. |
| Separate decisions | Branching logic should be explicit and linked to acceptance criteria. |
| Capture exceptions | Error, retry, cancellation, timeout, and approval paths should be visible. |
| Link evidence | Important flows should map to tests, QA checks, contracts, or runbook steps. |

## Process Record Shape

| Field | Description |
|---|---|
| Process ID | Stable workflow identifier. |
| Trigger | Event or request that starts the process. |
| Actors | Human, system, service, or external party involved. |
| Main flow | Ordered happy-path steps. |
| Decisions | Branches, conditions, and owners. |
| Exception paths | Failure, fallback, escalation, rollback, or manual handling. |
| Evidence | Acceptance criteria, tests, logs, output artifacts, or review checks. |

## Promotion Rules

Promote a workflow when a use case or business behavior becomes durable project knowledge. If the flow is still under discovery, keep the draft in task workspace and promote only open questions or accepted decisions.

## Starter Process

| Process ID | Trigger | Status | Notes |
|---|---|---|---|
| BP-000 | Project-specific workflow discovery | draft | Use this framework when the first workflow is known. |
