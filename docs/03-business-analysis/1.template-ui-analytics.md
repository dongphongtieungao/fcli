# UI Analytics Notes

## Purpose

Provide a reusable analysis framework for UI, dashboard, analytics, reporting, and user-facing metric work.

## Context

Use only if the new project includes UI, dashboards, analytics, reporting, or user-facing metrics. This file should frame product questions, not prescribe a visual design.

## Analysis Areas

| Area | Question | Evidence |
|---|---|---|
| User goal | What decision or action should the UI/metric support? | Requirement, use case |
| Audience | Who reads or operates the view? | Actor, persona, role |
| Metric definition | How is the metric calculated and constrained? | Contract, glossary, data governance |
| Freshness | How current must the data be? | NFR, runbook, QA |
| Interaction | What filters, drilldowns, exports, or actions are needed? | UI spec, use case |
| Failure state | What should happen when data is missing or invalid? | Acceptance criteria |
| Evidence | How will correctness be verified? | Test, fixture, screenshot, review |

## Promotion Rules

Promote UI/analytics analysis when it defines user-facing behavior, metric meaning, output contracts, or acceptance criteria. Keep exploratory layout ideas in design docs or task workspace until accepted.
