# Spec Acceptance Mapping Skill

## Purpose

Map request hoặc code change sang requirement, business flow, acceptance criteria, contract, test và evidence.

Skill này đảm bảo agent không sửa behavior dựa trên suy đoán. Mọi thay đổi nghiệp vụ, interface, generated output, runtime flow hoặc user visible behavior phải có mapping rõ từ source chính thức tới acceptance và verification.

## Context

Skill này là Spec First gate cho behavior change. Trong repo này, nó đặc biệt quan trọng với application/API behavior, output artifacts, module-specific behavior, data contracts, runbook commands, and QA evidence.

## When to use

Dùng bắt buộc khi task có một trong các dấu hiệu:

1. Đổi business rule.
2. Đổi application, API, UI, data contract hoặc output format.
3. Đổi runtime workflow, pipeline, job, report, batch hoặc rendering behavior.
4. Thêm feature mới.
5. Fix bug có ảnh hưởng user visible behavior.
6. Review xem code có đồng bộ với nghiệp vụ không.
7. Chuẩn bị test, QA matrix, DoD hoặc release evidence.

Không bắt buộc dùng cho docs wording, comment, metadata hoặc refactor không đổi behavior. Nếu không chắc behavior có đổi không, dùng skill này ở mức light mapping.

## Inputs

1. Task Brief.
2. Context Pack.
3. Prompt Routing Result nếu có.
4. Source paths liên quan.
5. Existing tests hoặc logs nếu có.

## Valid mapping sources

Nguồn hợp lệ theo độ ưu tiên:

1. Requirement, use case, PRD, spec chính thức.
2. Vision, glossary và domain model.
3. BPMN hoặc business process flow.
4. Gherkin feature, BDD hoặc acceptance scenario.
5. application, API, UI, event, data hoặc report contract.
6. Output contract, runbook, artifact expectation, record/page/item count expectation khi có.
7. QA matrix, review checklist, DoD.
8. Existing test hoặc verification script.
9. Workspace notes chỉ là observation, không phải source chốt acceptance.

## Procedure

### Step 1. Identify behavior under change

Mô tả behavior hoặc contract đang bị tác động:

1. Người dùng hoặc actor nào bị ảnh hưởng.
2. Flow bắt đầu và kết thúc ở đâu.
3. Input là gì.
4. Output là gì.
5. Rule hoặc validation nào liên quan.
6. Error path hoặc edge case nào liên quan.
7. Behavior hiện tại và expected behavior khác nhau thế nào.

### Step 2. Find upstream business source

Tìm nguồn upstream:

```powershell
rg -n --heading --smart-case "<business term>|<feature>|<flow>|<rule>" docs/01-requirements docs/02-architecture docs/03-business-analysis -g "*.md"
```

Nếu không có business source, ghi `NOT_MAPPED` hoặc `NEEDS_CLARIFICATION`.

### Step 3. Find executable specification

Tìm Gherkin, BDD hoặc test case:

```powershell
rg -n --heading --smart-case "<feature>|<scenario>|Given|When|Then|<business term>" tests/features tests/bdd docs/04-spec-test -g "*.feature" -g "*.md" -g "*.py"
```

Nếu chưa có executable spec, ghi gap `missing executable specification`.

### Step 4. Find contract or interface spec

Tùy loại hệ thống, tìm:

1. application spec.
2. OpenAPI spec.
3. UI spec.
4. Data contract.
5. Report contract.
6. Event contract.
7. Runbook command shape.
8. QA matrix validation lane.

```powershell
rg -n --heading --smart-case "<command>|<endpoint>|<screen>|<schema>|<output>|<exit code>|<field>" docs/04-spec-test docs/02-architecture -g "*.md" -g "*.yaml" -g "*.yml" -g "*.json"
```

Với report generation hoặc application task, đọc thêm:

```powershell
rg -n --heading --smart-case "<command>|<mode>|<config>|debug|validation|warning|error|artifact" docs/04-spec-test docs/06-quality-assurance docs/09-runbook -g "*.md"
```

### Step 5. Extract acceptance criteria

Tạo AC rõ, có ID. Không bịa rule. Nếu thiếu thông tin, ghi `Open Question`.

Mẫu ID:

```text
AC_001: <observable condition>
AC_002: <observable condition>
AC_003: <error or edge condition>
```

AC tốt phải:

1. Quan sát được.
2. Kiểm được bằng test, command, log hoặc artifact inspection.
3. Có input và expected output rõ.
4. Không dùng từ mơ hồ như nhanh, tốt, đẹp, hợp lý nếu không có threshold.
5. Có link tới requirement hoặc contract khi có.

### Step 6. Map AC to tests and evidence

Với mỗi AC, map:

1. Upstream source.
2. Contract field hoặc behavior.
3. Existing test hoặc test cần tạo.
4. Verification command.
5. Evidence artifact.
6. Mapping status.

Mapping status:

1. `MAPPED` khi có source và evidence phù hợp.
2. `PARTIALLY_MAPPED` khi có source nhưng thiếu test hoặc thiếu evidence.
3. `NOT_MAPPED` khi không có source phù hợp.
4. `NOT_APPLICABLE` khi task không đổi behavior.
5. `NEEDS_CLARIFICATION` khi thiếu quyết định nghiệp vụ.

### Step 7. Decide implementation gate

Chỉ cho phép implementation khi:

1. Behavior đã map được tối thiểu tới source chính thức.
2. AC quan trọng có verification method.
3. Missing tests hoặc missing evidence đã được ghi rõ.
4. Open questions không chặn scope hiện tại.

Nếu không đạt, trả về `ANALYSIS_ONLY`, `BLOCKED` hoặc `NEEDS_HUMAN_REVIEW`.

## Output contract

```text
Spec Acceptance Map
Affected behavior:
Affected actor or user:
Upstream business sources:
Contract or interface sources:
Existing executable specs:
Runbook or validation sources:
Acceptance criteria:
AC to evidence map:
Mapping status:
Missing tests:
Missing sources:
Open questions:
Implementation gate:
Recommended next skill:
```

## Usage and verification

Use this skill before implementation for behavior, contract, application, data-backed logic, or user-visible output changes. Verify completion by checking that each important AC has a source, contract or runbook link when available, test/evidence mapping, status, and explicit gap if any source is missing.

## AC to evidence map template

```text
AC_001
Statement:
Source:
Contract:
Test or scenario:
Verification command:
Evidence expected:
Status:
Gap:
```

## Failure modes

1. Agent tự tạo AC không dựa trên source.
2. Chỉ map code với test mà không map tới requirement hoặc contract.
3. Dùng workspace note làm source chốt acceptance.
4. Gọi test pass là business verified khi chưa có AC.
5. Bỏ qua error path, empty state, permission hoặc rollback.
6. Không ghi gap khi thiếu Gherkin hoặc test.

## Done criteria

Skill hoàn tất khi mỗi behavior quan trọng có mapping status rõ và implementation gate được quyết định bằng evidence, không bằng cảm tính.
