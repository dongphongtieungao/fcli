# KMS Sync and Drift Control Skill

## Purpose

Quyết định sau mỗi task cần cập nhật `llm-wiki/`, `docs/`, `manifest`, `sources`, `prompts`, taxonomy, template hoặc không cần cập nhật gì.

Skill này giúp framework học có kiểm soát, tránh tri thức quan trọng bị kẹt trong chat, đồng thời tránh promote nhầm observation thành source of truth.

## Context

Skill này là phase sync sau verification. Nó quyết định knowledge nào thuộc `llm-wiki/`, knowledge nào thuộc official `docs/`, và registry nào phải cập nhật để tránh drift.

## When to use

Dùng sau một task khi có một trong các điều kiện:

1. Phát hiện source mới.
2. Tạo hoặc sửa artifact chính thức trong `docs/`.
3. Tạo hoặc sửa prompt contract.
4. Tạo hoặc sửa skill.
5. Phát hiện contradiction giữa docs, tests, code và workspace.
6. Có assumption hoặc open question đáng lưu.
7. Có lesson learned từ debug, verification hoặc RCA.
8. Có module knowledge bền vững cần lưu vào `llm-wiki/module-notes/`.
9. Có change ảnh hưởng manifest, sources, prompts hoặc KMS index.

Không dùng để ghi mọi task note tạm thời. Chỉ sync tri thức có giá trị tái sử dụng hoặc cần audit.

## Inputs

1. Task Brief.
2. Context Pack.
3. Diff hoặc danh sách file thay đổi.
4. Evidence Pack.
5. Contradictions, assumptions, open questions nếu có.
6. New or updated official artifacts nếu có.

## Workspace and source of truth rules

1. `docs/` là official KMS và source of truth.
2. `llm-wiki/` là controlled workspace, không phải source of truth.
3. `src/` là current implementation, không tự động là policy.
4. `tests/` là executable hoặc spec adjacent evidence khi có.
5. Workspace observation chỉ được promote sang `docs/` khi có decision hoặc evidence chính thức.
6. Nếu official artifact mới được thêm, phải cập nhật registry liên quan.

## Sync decision tree

### Case 1. Không có tri thức bền vững mới

Không cập nhật KMS. Final report ghi:

```text
Knowledge sync: not needed
Reason: no durable observation or official source change
```

### Case 2. Có observation về code hoặc module

Ghi vào `llm-wiki/module-notes/<module>.md` hoặc `llm-wiki/codebase-map.md` nếu observation có giá trị tái sử dụng.

Không ghi nếu chỉ là chi tiết tạm thời của task.

### Case 3. Có assumption

Ghi vào `llm-wiki/assumptions.md` hoặc module note liên quan khi assumption ảnh hưởng plan, implementation hoặc verification.

Assumption phải có:

1. Statement.
2. Context.
3. Source hoặc lý do thiếu source.
4. Risk nếu sai.
5. Owner cần confirm.
6. Expiry hoặc next action.

### Case 4. Có open question

Ghi vào `llm-wiki/open-questions.md` khi thiếu quyết định chặn spec, implementation, verification hoặc release.

Open question phải có:

1. Question.
2. Impact.
3. Required decision.
4. Suggested owner.
5. Blocking level.

### Case 5. Có contradiction

Ghi vào `llm-wiki/contradiction-log.md` khi docs, tests, code, prompt, AGENTS hoặc workspace notes mâu thuẫn.

Contradiction phải có:

1. Source A.
2. Source B.
3. Conflict.
4. Impact.
5. Proposed resolution.
6. Status.

### Case 6. Official docs thay đổi

Nếu sửa hoặc thêm official artifact trong `docs/`, kiểm tra:

1. Frontmatter hợp lệ.
2. `source_of_truth` đúng.
3. `related` đúng.
4. Link không orphan.
5. Manifest có cần cập nhật không.
6. Sources catalog có cần cập nhật không.
7. Prompts registry có cần cập nhật không.
8. Taxonomy hoặc metadata schema có cần cập nhật không.

### Case 7. Prompt hoặc skill mới được thêm

Nếu thêm file trong `docs/grounding/prompts/` hoặc `docs/grounding/skills/`, cần kiểm tra:

1. File có ID, title, owner, intended use hoặc purpose.
2. File có required sources.
3. File có scope và forbidden actions nếu là prompt execution.
4. File có output contract.
5. File có verification hoặc failure policy.
6. `docs/grounding/prompts.md` hoặc skill index tương ứng có cần cập nhật không.
7. `manifest.md`, `manifest.yaml`, và `sources.md` có cần catalog entry không nếu file là official grounding artifact.
8. Nếu chưa có skill index riêng, catalog directory-level `docs/grounding/skills/` trong manifest/source catalog để tránh phải thêm một source ID cho từng file nhỏ.

## Procedure

### Step 1. Inspect task outcome

Xác định task đã tạo ra loại thay đổi nào:

1. Docs only.
2. Code or test change.
3. Contract or spec change.
4. Prompt or skill change.
5. KMS registry change.
6. Workspace observation.
7. Verification result only.

### Step 2. Decide sync target

Chọn một hoặc nhiều target:

1. `no-sync`.
2. `llm-wiki/open-questions.md`.
3. `llm-wiki/assumptions.md`.
4. `llm-wiki/contradiction-log.md`.
5. `llm-wiki/lessons-learned.md`.
6. `llm-wiki/codebase-map.md`.
7. `llm-wiki/module-notes/<module>.md`.
8. `docs/` official artifact.
9. `docs/grounding/manifest.md`.
10. `docs/grounding/manifest.yaml`.
11. `docs/grounding/sources.md`.
12. `docs/grounding/prompts.md`.
13. `docs/00-governance/taxonomy.md` hoặc metadata schema.

### Step 3. Apply minimum necessary update

Chỉ cập nhật phần cần thiết. Không rewrite toàn bộ tài liệu nếu không cần.

Mọi update phải giữ nguyên ý nghĩa gốc, không bịa source, không bịa owner, không gọi published nếu chưa có review.

### Step 4. Validate KMS consistency

Chạy hoặc đề xuất lệnh:

```powershell
rg -n --heading --smart-case "<new artifact id>|<new prompt id>|<module>|<feature>" docs llm-wiki
rg -n --heading --smart-case "source_of_truth:|related:|status:|truth_level:|source_policy:" docs llm-wiki -g "*.md"
```

Nếu có helper:

```powershell
.\tools\kb.ps1 audit
.\tools\kb.ps1 audit-metadata
.\tools\kb.ps1 missing-links
.\tools\kb.ps1 contradictions
```

### Step 5. Report sync result

Báo rõ đã sync gì, không sync gì, vì sao.

## Output contract

```text
KMS Sync Report
Task outcome type:
Sync needed:
Sync targets:
Official docs changed:
Workspace notes changed:
Manifest update needed:
Sources update needed:
Prompts registry update needed:
Taxonomy or metadata update needed:
Contradictions recorded:
Open questions recorded:
Assumptions recorded:
Lessons learned recorded:
Commands or checks:
Remaining drift risk:
Final KMS status:
```

## Usage and verification

Use this skill after verification to decide whether to update workspace notes, official docs, manifest, source catalog, prompt governance, or no KMS surface. Verify completion by checking that every official artifact addition or status/path change has a manifest/source catalog decision and that no workspace observation was promoted without evidence.

## Final KMS status

Dùng một trong các trạng thái:

1. `SYNCED` khi changes được phản ánh vào đúng artifact.
2. `PARTIALLY_SYNCED` khi có update nhưng còn missing registry, missing link hoặc pending owner decision.
3. `NOT_SYNCED` khi phát hiện drift nhưng chưa cập nhật.
4. `NOT_NEEDED` khi task không tạo durable knowledge.
5. `BLOCKED` khi thiếu permission, source hoặc quyết định để sync an toàn.

## Failure modes

1. Ghi mọi thứ vào `docs/` làm official hóa observation.
2. Ghi mọi task note vào `llm-wiki/` gây nhiễu.
3. Không cập nhật manifest khi thêm official artifact.
4. Không cập nhật prompts registry khi thêm prompt mới.
5. Không ghi contradiction dù phát hiện docs và code lệch nhau.
6. Không ghi open question khi thiếu AC hoặc source.
7. Rewrite tài liệu lớn ngoài scope.
8. Tạo link tới file không tồn tại.

## Done criteria

Skill hoàn tất khi knowledge sync decision rõ, target đúng, drift còn lại được ghi, và final report nói chính xác KMS đã sync hay chưa.
