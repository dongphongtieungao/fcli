# Evidence Verification Skill

## Purpose

Chuẩn hóa việc kiểm chứng kết quả agent bằng evidence có thể kiểm tra lại.

Skill này ngăn agent báo `done` hoặc `VERIFIED` chỉ dựa trên lời khẳng định. Mọi verdict phải dựa trên command, exit code, log, warning, output artifact, test result, manual inspection hoặc limitation được ghi rõ.

## Context

Skill này triển khai Verification Model cho SDD. Với repo này, validation/report generation phải lấy canonical command từ application spec, QA matrix, runbook hoặc task-specific source; không dùng ví dụ trong skill hoặc `AGENTS.md` như nguồn canonical khi nguồn chính thức khác tồn tại.

## When to use

Dùng sau khi có một trong các kết quả:

1. Code change.
2. Test change.
3. Contract change.
4. Docs official change.
5. Runtime behavior change.
6. Report, file, API response, application output hoặc UI artifact.
7. Prompt hoặc KMS artifact thay đổi.
8. Agent claim đã hoàn thành task.

Dùng ở mức light cho docs only. Dùng đầy đủ cho runtime, business behavior, contract, data, security hoặc release.

## Inputs

1. Task Brief.
2. Context Pack.
3. Prompt Routing Result nếu có.
4. Spec Acceptance Map nếu có.
5. Code Boundary Result nếu có.
6. Diff hoặc danh sách file thay đổi.
7. Commands đã chạy hoặc có thể chạy.
8. Logs, outputs, artifacts nếu có.

## Verification sources

Lấy command hoặc checklist từ nguồn canonical theo thứ tự:

1. Contract hoặc spec, đặc biệt `docs/04-spec-test/application-spec.md`.
2. QA matrix, đặc biệt `docs/06-quality-assurance/14.qa-matrix.md`.
3. DoD.
4. Runbook hoặc DoD liên quan, đặc biệt `docs/09-runbook/18.dod.md` khi completion evidence cần được chốt.
5. CI workflow.
6. Existing test documentation.
7. Task specific instruction từ human.
8. Fallback command do agent đề xuất, nhưng phải ghi là non canonical nếu chưa có source.

## Procedure

### Step 1. Identify verification scope

Xác định task thuộc loại nào:

1. Docs only.
2. Comment or header only.
3. Code change hẹp.
4. Business behavior change.
5. Contract or interface change.
6. Data, persistence or integration change.
7. Runtime, report or artifact generation.
8. Release or handover.

### Step 2. Select evidence level

Chọn mức evidence tối thiểu có ý nghĩa:

1. `structure-check` cho docs only.
2. `syntax-or-import-check` cho comment hoặc code touch nhẹ.
3. `targeted-test` cho code change hẹp.
4. `acceptance-or-contract-test` cho behavior hoặc interface change.
5. `runtime-check` cho application, API, UI, batch, job hoặc generated output.
6. `log-and-artifact-inspection` cho runtime hoặc user facing output.
7. `regression-check` cho shared module hoặc high risk.
8. `release-readiness-check` cho release hoặc handover.

### Step 3. Resolve commands

Ví dụ command lookup:

```powershell
rg -n --heading --smart-case "pytest|test|lint|verify|run|smoke|contract|spectral|ajv|eval|DoD|done" docs tests .github tools verify
```

Nếu có `tools/kb.ps1`:

```powershell
.\tools\kb.ps1 specs "<feature or module>"
.\tools\kb.ps1 ready
.\tools\kb.ps1 audit-ci
```

No validation shortcut:

1. Không chạy validation chỉ vì command xuất hiện trong skill này.
2. Resolve command từ canonical source trước khi chạy.
3. Với report generation, mặc định command phải có `--debug` trừ khi canonical source nói rõ không cần.
4. Nếu `AGENTS.md`, application spec, QA matrix và runbook khác nhau, dùng application spec/runbook cho command shape và QA matrix cho lane status/caveat.

### Step 4. Run or record verification

Nếu agent có thể chạy command, ghi:

1. Command.
2. Command source.
3. Exit code.
4. Stdout summary.
5. Stderr summary.
6. Warning và error.
7. Artifact path hoặc output metadata.

Nếu không thể chạy, ghi rõ:

1. Vì sao không chạy được.
2. Evidence thay thế là gì.
3. Verdict bị hạ cấp ra sao.

### Step 5. Inspect warnings and errors

Không chỉ ghi command pass. Phải đọc warning và error:

1. Warning expected hay unexpected.
2. Error relevant hay unrelated.
3. Có security, data, performance hoặc contract risk không.
4. Có flaky, skipped, xfailed hoặc partial test không.
5. Có TODO hoặc known failure không.

### Step 6. Inspect artifacts

Nếu task tạo output artifact, kiểm tra:

1. Path tồn tại.
2. Size hoặc count hợp lý.
3. Page, record, response, status, checksum, count, size hoặc metadata nếu phù hợp.
4. Nội dung chính có khớp AC hoặc contract không.
5. Có log hoặc warning liên quan artifact không.

Với rendered/generated output, ghi thêm:

1. Output artifact path.
2. Artifact-specific metadata nếu có thể đọc, ví dụ page count, item count, record count, checksum hoặc rendered object count.
3. Warning/error trong log.
4. Nếu có warning kiểu `file not found`, `0 points`, missing data, skipped source hoặc partial render, không gọi full `VERIFIED`; tối đa ghi file generation passed hoặc `PARTIALLY VERIFIED` trừ khi QA source nói warning đó acceptable.

### Step 7. Assign verdict

Chỉ dùng một trong các verdict:

1. `VERIFIED` khi evidence đủ với task type và risk.
2. `PARTIALLY VERIFIED` khi có evidence hữu ích nhưng còn thiếu gate, command, environment hoặc artifact inspection.
3. `FAILED` khi test, runtime, log, artifact hoặc diff vi phạm yêu cầu.
4. `BLOCKED` khi thiếu source, test data, dependency hoặc environment bắt buộc.
5. `NEEDS HUMAN REVIEW` khi cần business, architecture, security, compliance hoặc release decision.

Không dùng `VERIFIED` nếu warning hoặc error chưa được đọc.

## Output contract

Compact evidence pack:

```text
Evidence Pack
Task summary:
Task type:
Affected files:
Affected module:
Grounding sources:
Verification performed:
Command source:
Command:
Exit code:
Warnings:
Errors:
Artifacts:
Output artifact path:
Artifact metadata:
Manual inspection:
Skipped or not run:
Reason for skipped verification:
Remaining risks:
Final status:
```

## Usage

Use this skill after any code, test, official docs, prompt, KMS, runtime, report, or artifact-producing change. For documentation-only work, use structure/link/traceability checks. For runtime/report behavior, use canonical command resolution plus log and artifact inspection.

Extended evidence pack cho behavior hoặc runtime:

```text
Extended Evidence Pack
Affected business or runtime flow:
Acceptance mapping:
Tests or automation:
Runtime command:
Runtime output:
Log inspection:
Artifact inspection:
Regression scope:
Rollback or mitigation note:
Knowledge sync needed:
Final status:
```

## Failure modes

1. Agent ghi test pass nhưng không ghi command.
2. Agent ghi command nhưng không ghi exit code.
3. Agent không đọc warning hoặc skipped tests.
4. Agent dùng một test unrelated để claim full verification.
5. Agent không kiểm artifact output.
6. Agent claim verified khi environment không chạy được.
7. Agent không hạ cấp verdict khi thiếu evidence.

## Done criteria

Skill hoàn tất khi final status được gán theo evidence thật, limitation được ghi rõ và reviewer có thể kiểm lại bằng command hoặc artifact path.
