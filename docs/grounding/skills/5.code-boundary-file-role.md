# Code Boundary and File Role Skill

## Purpose

Giúp agent hiểu vai trò file, module boundary, caller, callee, risk và verification surface trước khi sửa code.

Skill này chống sửa mù trong codebase lớn. Nó không tạo file note cho mọi source file. Nó chỉ tạo File Role Snapshot trong task context, và chỉ cập nhật `llm-wiki/module-notes/` khi tri thức mới có giá trị bền vững.

## Context

Skill này là code-intelligence gate trước implementation. Trong repo này, nó phải nhận diện đúng entry points, shared surfaces, adapters, domain modules, factories, validators, and output boundaries before code changes.

## When to use

Dùng trước mọi code change có một trong các điều kiện:

1. File là entry point, orchestration service, controller, command, pipeline, adapter, repository, renderer, factory hoặc validation harness.
2. File thuộc shared module hoặc hotspot.
3. File có nhiều caller hoặc ảnh hưởng nhiều flow.
4. Task đổi behavior, contract, generated output, data access hoặc runtime path.
5. Agent chưa biết file thuộc module nào.
6. Có dấu hiệu mâu thuẫn giữa docs, tests, `llm-wiki/` và `src/`.

Không cần dùng đầy đủ cho typo, comment hoặc docs only. Có thể dùng light snapshot nếu chỉ sửa test nhỏ.

## Inputs

1. Task Brief.
2. Context Pack.
3. Spec Acceptance Map nếu task đổi behavior.
4. Target file hoặc candidate files.
5. Codebase map hoặc module note nếu có.

## Required sources

1. `AGENTS.md`.
2. Module-local `AGENTS.md` khi subtree liên quan có local policy.
3. `docs/02-architecture/04.architecture-c4.md` hoặc architecture artifact tương đương nếu có.
4. `docs/02-architecture/06.adr.md` nếu có architectural decision liên quan.
5. `docs/02-architecture/08.standards.md` nếu có coding boundary hoặc style.
6. `docs/04-spec-test/` nếu file liên quan contract.
7. `docs/06-quality-assurance/14.qa-matrix.md` nếu cần verification surface.
8. `llm-wiki/codebase-map.md` và module notes nếu có.
9. Target file, direct callers, direct callees, tests liên quan.

## Procedure

### Step 1. Identify target files

Từ task và context pack, xác định file cần đọc hoặc sửa.

Nếu chưa biết target file, dùng `rg`:

```powershell
rg -n --heading --smart-case "<feature>|<command>|<class>|<function>|<business term>" src tests
```

### Step 2. Locate module boundary

Tìm module qua path, package, namespace, imports, folder structure và architecture docs.

```powershell
rg -n --heading --smart-case "<module>|<bounded context>|<component>|<service>|<pipeline>" docs llm-wiki src tests
```

Ghi rõ module boundary:

1. Module này chịu trách nhiệm gì.
2. Module này không chịu trách nhiệm gì.
3. Module upstream nào gọi nó.
4. Module downstream nào nó gọi.
5. Contract nào nó phải tuân thủ.

### Step 3. Identify domain/runtime boundary

Nếu path thuộc domain/runtime model, ghi rõ:

1. Shared module: code used by multiple features or bounded contexts.
2. Feature module: code owned by one feature or domain area.
3. Adapter/output module: integration, persistence, rendering, export, or external boundary behavior.
4. `shared`: `src/common/`, `src/db/`, `src/exporters/`, `src/app/`, capture, CSV provider, aggregation shared code.

Với model-specific code, đọc tối thiểu:

1. Local `AGENTS.md`.
2. `interfaces.py`.
3. Factory, registry, router, or dependency injection files when present.
4. Contract, schema, or adapter files when present.
5. Relevant query, schema, contract, or data-access files.

Không chạm model directory khác trong cùng change nếu scope không phải shared và user chưa cho phép.

### Step 4. Read workspace notes as observations

Đọc nếu tồn tại:

1. `llm-wiki/codebase-map.md`.
2. `llm-wiki/module-notes/<module>.md`.
3. `llm-wiki/contradiction-log.md`.
4. `llm-wiki/open-questions.md`.
5. `llm-wiki/lessons-learned.md`.

Không dùng workspace notes để override official docs.

### Step 5. Find callers and callees

Tìm trực tiếp bằng symbol hoặc filename:

```powershell
rg -n --heading --smart-case "<function>|<class>|<method>|<command>|<service>" src tests
```

Phân loại:

1. Direct callers.
2. Direct callees.
3. Test callers.
4. Runtime entry points.
5. Config hoặc wiring references.

### Step 6. Find related tests and validation

```powershell
rg -n --heading --smart-case "<module>|<function>|<class>|<behavior>|<command>" tests verify docs/06-quality-assurance docs/09-runbook
```

Nếu không có test, ghi rõ `no targeted test found`.

### Step 7. Create File Role Snapshot

Tạo snapshot trong task context, không nhất thiết ghi vào repo.

```text
File Role Snapshot
File:
Current role:
Module:
Impacted domain/runtime model:
Local AGENTS policy:
Bounded context:
Official sources read:
Workspace notes read:
Callers:
Callees:
Related tests:
Contracts affected:
Runtime or user impact:
Change risk:
Planned change:
Out of scope:
Verification surface:
Open questions:
```

### Step 8. Decide whether to update Code Intelligence Header

Chỉ thêm header khi file là hotspot hoặc entry point quan trọng và task đang chạm file đó.

Không thêm header vào DTO, enum, generated file, simple utility hoặc file không quan trọng.

Header không được chứa business rule, secret, SQL, assumption chưa ghi, hoặc claim verified không có evidence.

### Step 9. Decide whether to update module note

Cập nhật `llm-wiki/module-notes/<module>.md` chỉ khi có durable knowledge mới:

1. Module responsibility rõ hơn.
2. Main code flow được xác nhận.
3. Known contradiction mới.
4. Open question mới.
5. Verification surface mới.
6. Hotspot mới.

Nếu chỉ là task note tạm thời, không cập nhật module note.

```text
Module:

```

## Usage and verification

Use this skill before touching important code files. Verify completion by checking that the output names the file role, impacted domain/runtime model, local policy, callers, callees, tests, contracts affected, risk, out-of-scope areas, and implementation readiness.

## Implementation readiness

Gán một trong các trạng thái:

1. `READY_FOR_SMALL_CHANGE` khi scope hẹp, caller, callee và test surface rõ.
2. `NEEDS_SPEC_MAPPING` khi behavior hoặc contract chưa map.
3. `NEEDS_ARCHITECT_REVIEW` khi chạm shared architecture, dependency hoặc boundary.
4. `BLOCKED` khi không xác định được file role hoặc thiếu source nghiêm trọng.

## Failure modes

1. Sửa file chỉ vì tên file có vẻ liên quan.
2. Không đọc caller/callee.
3. Không đọc test surface.
4. Coi source code là source of truth nghiệp vụ.
5. Thêm header vào mọi file gây nhiễu.
6. Tạo file note cho mọi file làm KMS stale.
7. Refactor vượt scope khi chưa có plan.

## Done criteria

Skill hoàn tất khi file role, module boundary, caller, callee, tests và risk đã rõ đủ để quyết định có được sửa code hay không.
