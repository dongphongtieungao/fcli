# Create Current Task Skill

## Purpose

Create a fresh `llm-wiki/tasks/current/` workspace from a user-provided task description.

This skill analyzes the intended task and writes task-specific workspace files. It must not copy blank template files. It uses `docs/methodology/Task.md` as the task workspace methodology.

## Context

Task history is part of the agent request lifecycle. It helps future agents trace why requirements, architecture decisions, business rules, implementation choices, verification notes, or documentation promotions happened.

`llm-wiki/tasks/` remains workspace-only and must not override official `docs/`, executable specs, contracts, ADRs, QA matrix, runbooks, or current user instructions.

## When to use

Use this skill when:

1. The user starts a new task and provides a task description.
2. The user asks to create a new current task.
3. `current` was archived and a fresh task workspace is needed.
4. The task likely needs traceable intake, decisions, verification, or promotion notes.

If `llm-wiki/tasks/current/` already has useful content, use Skill 8 first to archive it into `llm-wiki/tasks/archive/` and clean `current`.

## Inputs

1. User-provided task description.
2. Current date.
3. Existing `llm-wiki/tasks/current/` state.
4. Relevant archived task history under `llm-wiki/tasks/archive/` if the new task appears related to previous work.

## Procedure

### Step 1. Read guidance

Read:

```text
docs/methodology/Task.md
docs/grounding/skills/8.archive-current-task.md
```

Use the task workspace methodology as guidance. Do not copy blank template files.

### Step 2. Protect existing current task

Inspect:

```text
llm-wiki/tasks/current/
```

If it has meaningful content, archive it with Skill 8 before creating the new task. After Skill 8, `current` should be empty unless Skill 9 is immediately invoked to populate it.

### Step 3. Analyze the new task

From the user description, infer conservatively:

1. Task objective.
2. Task type.
3. Risk level.
4. Impacted module or bounded context: module name, `shared`, `n/a`, or `unknown`.
5. Artifact layer: requirement, ADR, BPMN, contract spec, QA matrix, runbook, methodology, workspace-only, or none.
6. Expected output.
7. Initial open questions.
8. Likely grounding sources to read.
9. Likely verification expectation.
10. Whether task history search is needed.

Do not invent business rules. If unclear, mark the field as `unknown` and record an open question.

### Step 4. Search task history when useful

Search archived task folders only when the new task appears connected to prior work:

```powershell
rg -n --heading --smart-case "<keyword>" llm-wiki/tasks/archive
```

Record relevant archived task paths in `TASK.md` and `intake.md`.

Do not scan `llm-wiki/tasks/archive/` for every task by default. If no history search is needed, record that it was skipped and why.

### Step 5. Create `current`

Create or refresh:

```text
llm-wiki/tasks/current/
```

Generate task-specific files. Recommended minimum:

```text
TASK.md
raw.md
intake.md
decision-log.md
promotion.md
verification-notes.md
```

Create optional files only when useful:

```text
analysis.md
clarification-log.md
contradictions.md
```

### Step 6. Populate task-specific content

`TASK.md` should summarize:

1. Original task description.
2. Task objective.
3. Task type.
4. Artifact layer.
5. Impacted model.
6. Initial source plan.
7. Related archived tasks, if any.
8. Ready status.
9. Next skill.

`raw.md` should preserve the original task description.

`intake.md` should contain a structured task brief.

`promotion.md` should say whether promotion is currently expected, not perform promotion.

`verification-notes.md` should define expected verification, not claim verification has passed.

### Step 7. Report

Return a concise create-task report:

```text
Create Current Task Report
Task objective:
Task type:
Impacted domain/runtime model:
Artifact layer:
Files created:
Related archived tasks checked:
Open questions:
Next skill:
Final status:
```

## Safety rules

1. Do not treat task workspace notes as source of truth.
2. Do not promote to `docs/` during task creation.
3. Do not delete archived task history.
4. Do not overwrite meaningful `current` content without archiving first.
5. Do not claim task readiness if grounding, acceptance, or verification sources are missing.
6. Do not create file names from unsafe user text; use conservative kebab-case titles when needed.

## Done criteria

1. `llm-wiki/tasks/current/` exists.
2. `TASK.md`, `raw.md`, and `intake.md` are task-specific.
3. The workspace records whether archived task history was checked.
4. Open questions and promotion expectations are explicit.
5. The next skill or next action is clear.
