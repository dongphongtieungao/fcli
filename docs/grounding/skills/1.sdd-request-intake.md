# SDD Request Intake Skill

## Purpose

Chuẩn hóa request thô của user thành một task brief có thể giao cho agent mà không làm mất kiểm soát scope, source, risk và verification.

Skill này là cổng đầu tiên của agentic SDD. Nó không sửa code, không sửa tài liệu chính thức và không ra quyết định nghiệp vụ thay human. Nó chỉ phân loại request, xác định nguồn cần đọc, xác định điều kiện thiếu và tạo task brief đủ rõ để chuyển sang retrieval, prompt routing, planning hoặc escalation.

## Context

Skill này triển khai bước intake trong workflow `grounding -> spec -> code -> verify -> sync docs`. Nó phải tuân thủ root `AGENTS.md`, local `docs/AGENTS.md` khi task chạm tài liệu, và local policy của module/path khi có.

## When to use

Dùng skill này trước mọi task không tầm thường, đặc biệt khi request có một trong các dấu hiệu sau:

1. User yêu cầu sửa code, sửa bug, refactor, thêm feature hoặc đổi behavior.
2. User yêu cầu tạo spec, review spec, tạo plan, tạo prompt hoặc cập nhật KMS.
3. Request chạm tới business rule, contract, application, API, UI, data, security, runtime, release hoặc observability.
4. Request có thể cần nhiều agent role như architect, coder, reviewer, QA, SRE hoặc KMS curator.
5. Request chưa nêu rõ acceptance criteria, file liên quan, source of truth hoặc verification.

Không cần dùng skill này cho câu hỏi giải thích ngắn, hỏi khái niệm chung, hoặc việc không tác động repo.

## Inputs

1. User request hiện tại.
2. Bối cảnh ngắn do user cung cấp nếu có.
3. Danh sách file hoặc module user nhắc tới nếu có.
4. Bằng chứng hiện có như log, screenshot, command output, issue, spec, ADR hoặc runbook nếu có.

## Required sources

Agent phải ưu tiên kiểm tra theo thứ tự:

1. User request hiện tại.
2. `AGENTS.md`.
3. `docs/AGENTS.md` nếu request chạm `docs/`.
4. Local `AGENTS.md` nếu request chạm model-specific code:
   `AGENTS.md` gần nhất trong subtree liên quan nếu tồn tại.
5. `docs/grounding/manifest.md`.
6. `docs/grounding/manifest.yaml` nếu cần registry máy đọc.
7. `docs/grounding/sources.md` nếu cần source ID, owner, coverage.
8. `docs/grounding/prompts.md` nếu cần route lane.
9. Artifact chính thức trong `docs/` liên quan tới request.
10. `tests/features/` hoặc `tests/bdd/` nếu task có acceptance hoặc regression.
11. `llm-wiki/` chỉ để xem observation, assumption, contradiction, lesson learned.
12. `src/` để xác nhận implementation hiện tại, không dùng làm source of truth nghiệp vụ.

## Procedure

### Step 1. Restate request

Tóm tắt request thành 3 đến 5 ý:

1. User muốn đạt kết quả gì.
2. Hệ thống, module hoặc artifact nào có thể bị ảnh hưởng.
3. Output mong muốn là analysis, spec, plan, code change, review, verification hay KMS update.
4. Có yếu tố rủi ro nào được nêu trực tiếp.
5. Có bằng chứng nào user đã cung cấp.

### Step 2. Classify task type

Chọn một task type chính:

1. `analysis` khi chỉ cần phân tích hoặc giải thích.
2. `documentation` khi chỉ tạo hoặc sửa Markdown, prompt, runbook, KMS note.
3. `spec` khi cần tạo hoặc sửa FR, NFR, AC, contract, BPMN, Gherkin.
4. `planning` khi cần milestone, task checklist, agent workflow.
5. `implementation` khi cần sửa code hoặc test.
6. `review` khi cần đánh giá code, docs, spec, KMS hoặc prompt.
7. `verification` khi cần chạy hoặc thiết kế bằng chứng kiểm chứng.
8. `kms-maintenance` khi cần cập nhật manifest, sources, prompts, taxonomy, templates.
9. `release-handover` khi chạm release note, DoD, rollback, handover.

Nếu request chứa nhiều task type, chọn task type rủi ro cao nhất làm primary và ghi các secondary task type.

### Step 3. Classify risk

Gán risk level:

1. `low` cho wording, metadata, comment, link, docs only không đổi nghĩa.
2. `medium` cho code path cục bộ, test only, config không production, docs chính thức có ảnh hưởng nhỏ.
3. `high` cho business rule, shared module, contract, data path, persistence/query behavior, generated output, runtime pipeline.
4. `security-data` cho secret, PII, auth, permission, production data, credential, destructive command.
5. `release` cho artifact customer facing, deployment, rollback, operational readiness.

Nếu không chắc risk, chọn mức cao hơn và ghi lý do.

### Step 4. Identify grounding sources

Liệt kê nguồn cần đọc, không bịa path. Nếu chưa biết source cụ thể, ghi nhóm nguồn cần tìm:

1. Requirement hoặc use case.
2. Vision, glossary, C4, NFR, ADR, standards.
3. BPMN hoặc business flow.
4. Feature, BDD, test case.
5. application, API, UI, data contract.
6. QA matrix, review checklist, DoR, DoD, runbook.
7. Prompt lane hoặc prompt module.
8. Module note hoặc contradiction log trong `llm-wiki/` nếu có.

### Step 5. Identify impacted domain/runtime model and local policy

Với repo này, mọi task code/runtime phải ghi rõ `Impacted domain/runtime model`:

1. `shared` khi chạm module dùng chung hoặc hành vi cross-cutting.
2. Tên module cụ thể khi task chỉ chạm một bounded context.
3. `n/a` khi task chỉ là workspace, docs intake, hoặc cleanup không chạm runtime behavior.
4. `shared` khi chạm `src/common/`, `src/db/`, `src/exporters/`, `src/app/`, capture, CSV provider hoặc aggregation shared surface.
5. `n/a` cho docs-only hoặc analysis không chạm domain/runtime model.

Nếu module-specific code bị ảnh hưởng, ghi local policy cần đọc và tránh trộn business logic giữa các bounded context trừ khi user giao scope shared rõ ràng.

### Step 6. Define scope boundary

Xác định:

1. Files hoặc folders có thể đọc.
2. Files hoặc folders có thể sửa nếu task về sau cho phép implementation.
3. Files hoặc folders out of scope.
4. Business behavior nào được phép đổi.
5. Contract nào không được đổi.
6. Dependency nào không được thêm nếu chưa approve.

### Step 7. Define forbidden actions

Luôn ghi rõ các hành động cấm phù hợp với risk:

1. Không tự tạo business rule ngoài spec.
2. Không đổi contract nếu request không yêu cầu.
3. Không thêm dependency nếu chưa có dependency review.
4. Không chạy DDL, DML, migration, delete, cleanup hoặc destructive command nếu chưa được phép.
5. Không in secret, token, private key, DSN, password hoặc PII vào output.
6. Không gọi `VERIFIED` nếu chưa có evidence.
7. Không promote nội dung `llm-wiki/` thành source of truth nếu chưa có quyết định chính thức.
8. Không sửa ngoài files in scope.
9. Không mix logic giữa các bounded context khi không có scope shared rõ ràng.
10. Không hardcode query, prompt, hoặc contract lớn vào code khi repo đã có vị trí artifact riêng.

### Step 8. Decide readiness

Gán một trong các trạng thái:

1. `READY_FOR_RETRIEVAL` khi đủ rõ để tìm context.
2. `READY_FOR_PLANNING` khi đã biết source chính và scope đủ rõ.
3. `READY_FOR_IMPLEMENTATION` chỉ khi spec, AC, impacted model, local policy, scope, files và verification đủ rõ.
4. `ANALYSIS_ONLY` khi thiếu spec, AC, source hoặc approval.
5. `BLOCKED` khi thiếu dữ liệu bắt buộc và không có fallback an toàn.
6. `NEEDS_HUMAN_REVIEW` khi cần business, architecture, security hoặc release decision.

## Output contract

Agent phải xuất task brief theo mẫu:

```text
Task Brief
Request summary:
Task type:
Secondary task types:
Risk level:
Impacted domain/runtime model:
Local AGENTS policy:
Affected domain or module:
Expected output:
Known facts:
Unknowns:
Grounding sources to read:
Files in scope:
Files out of scope:
Forbidden actions:
Assumptions:
Open questions:
Verification expectation:
Ready status:
Next skill:
```

## Usage and verification

Use this skill by producing the Task Brief before retrieval or implementation. Verify completion by checking that the brief includes task type, risk, impacted model, sources to read, files in/out of scope, forbidden actions, verification expectation, readiness, and next skill.

## Escalation rules

Escalate khi gặp một trong các điều kiện:

1. Thiếu business rule hoặc acceptance criteria cho behavior change.
2. Có mâu thuẫn giữa docs, tests, source code và workspace notes.
3. Task chạm security, secret, auth, permission, production data.
4. Task đổi architecture, shared module hoặc dependency.
5. User muốn release, rollback hoặc customer facing output nhưng thiếu DoD.

## Failure modes

1. Agent nhảy thẳng sang code khi chưa có source.
2. Agent coi request mơ hồ là spec đã chốt.
3. Agent không phân loại risk.
4. Agent không ghi forbidden actions.
5. Agent không xác định verification trước khi làm.
6. Agent hỏi quá nhiều câu lẻ tẻ thay vì gom open questions theo scope.

## Done criteria

Skill hoàn tất khi có task brief rõ, có trạng thái readiness và có next skill được đề xuất.
