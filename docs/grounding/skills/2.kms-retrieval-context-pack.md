# KMS Retrieval and Context Pack Skill

## Purpose

Dựng context pack có thể tái lập trước khi agent planning, coding, reviewing hoặc verifying.

Skill này giúp agent không đọc repo theo cảm tính. Nó buộc agent truy xuất theo source precedence, ghi command hoặc query đã dùng, phân biệt official docs, tests, workspace notes và source code. Kết quả là một context pack đủ gọn để làm việc, nhưng đủ bằng chứng để reviewer kiểm lại.

## Context

Skill này là retrieval layer sau intake. Nó phải tạo context có thể tái lập bằng `rg` hoặc `tools/kb.ps1`, không dựa vào chat history, và phải giữ `docs/` cao hơn `llm-wiki/` và `src/`.

## When to use

Dùng sau `1.sdd-request-intake.md` khi ready status là `READY_FOR_RETRIEVAL`, `READY_FOR_PLANNING`, `ANALYSIS_ONLY` hoặc `NEEDS_HUMAN_REVIEW`.

Dùng bắt buộc cho:

1. Task code change.
2. Task đổi business behavior, contract, data hoặc runtime.
3. Task review code, spec, prompt, KMS hoặc architecture.
4. Task cần kiểm đồng bộ docs, tests, `llm-wiki/`, `src/`.
5. Task cần tạo plan hoặc verify evidence.

## Inputs

1. Task Brief từ SDD Request Intake Skill.
2. Keyword, feature, module, command, file path hoặc business term liên quan.
3. Danh sách source cần đọc nếu task brief đã xác định.
4. Repo root có `rg` và nếu có thì `tools/kb.ps1`.

## Retrieval precedence

Luôn truy xuất theo thứ tự:

1. `AGENTS.md`, local `AGENTS.md` nếu scope yêu cầu, và grounding manifest.
2. Official docs trong `docs/`.
3. Executable specs trong `tests/features/` hoặc `tests/bdd/`.
4. Workspace notes trong `llm-wiki/`.
5. Source implementation trong `src/`.
6. Agent inference sau cùng.

Không dùng `llm-wiki/` hoặc `src/` để override official docs. Nếu mâu thuẫn, ghi contradiction candidate.

## Procedure

### Step 1. Resolve grounding entrypoint

Đọc hoặc tìm các file nền:

```powershell
rg -n --heading --smart-case "Grounding|Manifest|Source|Prompt|Precedence" AGENTS.md docs/grounding
```

Nếu có `tools/kb.ps1`, ưu tiên:

```powershell
.\tools\kb.ps1 status
.\tools\kb.ps1 grounding
```

### Step 2. Search official docs

Tìm theo keyword chính, synonym, module hoặc feature:

```powershell
.\tools\kb.ps1 docs "<keyword>"
```

Nếu không có helper:

```powershell
rg -n --heading --smart-case "<keyword>|<synonym>|<module>|<feature>" docs -g "*.md"
```

Mục tiêu là tìm:

1. Requirement hoặc use case.
2. Vision hoặc glossary.
3. C4, NFR, ADR, standards.
4. BPMN hoặc business flow.
5. application, API, UI, data contract.
6. QA matrix, review checklist, DoR, DoD, runbook.
7. Prompt lane hoặc prompt module.

### Step 3. Search tests and executable specs

```powershell
.\tools\kb.ps1 specs "<keyword>"
```

hoặc:

```powershell
rg -n --heading --smart-case "<keyword>|Feature:|Scenario:|Given|When|Then|pytest|assert" tests verify -g "*.feature" -g "*.py" -g "*.md"
```

Mục tiêu là tìm acceptance scenario, regression surface, validation harness hoặc command source.

### Step 4. Search workspace notes

```powershell
.\tools\kb.ps1 workspace "<keyword>"
```

hoặc:

```powershell
rg -n --heading --smart-case "<keyword>|assumption|open question|contradiction|lesson|verified" llm-wiki -g "*.md"
```

Phân loại kết quả workspace thành:

1. Observation.
2. Assumption.
3. Contradiction.
4. Lesson learned.
5. Verification note.

### Step 5. Search source code

```powershell
.\tools\kb.ps1 code "<keyword>"
```

hoặc:

```powershell
rg -n --heading --smart-case "<keyword>|<class>|<function>|<command>|<module>" src tests
```

Khi task có code impact, tìm thêm entrypoint, caller, callee:

```powershell
rg -n --heading --smart-case "def main|if __name__|Controller|router|command|pipeline|factory|provider|adapter|repository|service" src
```

### Step 6. Search blockers and contradictions

```powershell
.\tools\kb.ps1 blockers
.\tools\kb.ps1 contradictions
```

hoặc:

```powershell
rg -n --heading --smart-case "BLOCKED|TBD|TODO|FIXME|unclear|assumption|open question|contradiction|deprecated|legacy" docs llm-wiki tests src
```

### Step 7. Build context pack

Không copy toàn bộ tài liệu. Chỉ ghi các nguồn có liên quan thật sự.

Mỗi item trong context pack phải có:

1. Path.
2. Vai trò.
3. Vì sao liên quan.
4. Độ tin cậy.
5. Trích yếu ngắn.
6. Có cần đọc sâu tiếp không.

Với task report/code trong repo này, context pack phải ghi thêm:

1. Impacted domain/runtime model.
2. Local `AGENTS.md` đã đọc hoặc cần đọc.
3. Model or component contract files cần đọc, such as interfaces, factories, registries, adapters, schemas, or specs.
4. SQL/spec/runbook surface liên quan.

## Output contract

```text
Context Pack
Request keyword:
Task type:
Risk level:
Commands or searches used:
Official docs found:
Tests or executable specs found:
Workspace notes found:
Source code locations found:
Blockers or contradictions:
Missing sources:
Impacted domain/runtime model:
Local policy needed:
Context confidence:
Recommended next skill:
```

## Usage and verification

Use this skill by running the smallest repeatable search set and recording each command or query. Verify completion by checking that the context pack distinguishes official docs, tests, workspace notes, source code, blockers, missing sources, confidence, and recommended next skill.

## Context confidence

Dùng một trong các mức:

1. `high` khi có official docs, contract hoặc tests khớp request.
2. `medium` khi có docs chính thức nhưng thiếu test hoặc thiếu contract.
3. `low` khi chủ yếu dựa vào source code hoặc workspace notes.
4. `insufficient` khi không tìm được source đủ để plan hoặc implement an toàn.

## Failure modes

1. Search `src/` trước khi search `docs/`.
2. Copy quá nhiều context làm nhiễu prompt.
3. Không ghi command đã dùng nên không tái lập được.
4. Coi `llm-wiki/` là official source.
5. Không tìm blocker hoặc contradiction trước khi planning.
6. Không tìm test surface trước khi implementation.

## Done criteria

Skill hoàn tất khi có context pack đủ để chuyển sang prompt routing, spec mapping, code boundary analysis, planning hoặc escalation.
