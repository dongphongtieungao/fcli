# Prompt Lane Router Skill

## Purpose

Chọn đúng prompt lane và prompt contract cho request hiện tại, sau đó tạo execution prompt có role, source, scope, forbidden actions, output contract và verification expectation.

Skill này không thay thế `AGENTS.md` và không thay thế spec. Nó chỉ dùng prompt governance để điều phối agent vào đúng workflow.

## Context

Skill này nối Context Pack với prompt governance. Nó được dùng để chọn lane đã được registry hoặc `prompts.md` công nhận, không tạo lane/prompt mới theo cảm tính.

## When to use

Dùng sau khi đã có Task Brief và Context Pack.

Dùng bắt buộc khi:

1. Request cần agent thực thi theo lane cụ thể.
2. Có nhiều prompt module trong `docs/grounding/prompts/`.
3. Task có thể là analysis, planning, implementation, review, testing, documentation, KMS maintenance hoặc workspace update.
4. Cần đóng gói prompt cuối cùng để đưa cho Codex, Claude Code, Gemini coding agent hoặc agent tương đương.

## Inputs

1. Task Brief.
2. Context Pack.
3. `docs/grounding/prompts.md`.
4. Prompt module liên quan trong `docs/grounding/prompts/` nếu có.
5. Source registry trong `manifest.yaml` và `sources.md` nếu cần source ID.

## Lane taxonomy

Chọn một primary lane:

1. `analysis` dùng để hiểu yêu cầu, docs, code, gap, contradiction.
2. `planning` dùng để tạo milestone, tasks, risk, verification method.
3. `implementation` dùng để sửa code, test hoặc config khi gate đủ.
4. `review` dùng để review code, docs, spec, KMS, prompt, traceability.
5. `refactoring` dùng khi behavior được bảo vệ và scope refactor rõ.
6. `testing` dùng để xác định hoặc chạy validation, QA, eval, regression.
7. `documentation` dùng khi tạo hoặc sửa tài liệu chính thức.
8. `kms-maintenance` dùng khi cập nhật manifest, sources, prompts, taxonomy, templates.
9. `workspace-update` dùng khi cập nhật `llm-wiki/` sau task.
10. `release-handover` dùng khi chuẩn bị DoD, handover, release note, rollback.

Nếu có nhiều lane, dùng thứ tự an toàn:

1. `analysis` trước khi thiếu source.
2. `planning` trước khi implementation.
3. `testing` hoặc `verification` trước khi claim done.
4. `kms-maintenance` sau khi official artifact thay đổi.
5. `workspace-update` sau khi có durable observation.

## Procedure

### Step 1. Read prompt governance

Đọc `docs/grounding/prompts.md` để xác định lane và prompt registry.

Nếu file không tồn tại, không bịa. Tạo routing dựa trên `AGENTS.md` và ghi gap `missing prompts.md`.

### Step 2. Match request to lane

Dựa vào Task Brief:

1. Nếu expected output là findings hoặc explanation, chọn `analysis`.
2. Nếu expected output là plan, chọn `planning`.
3. Nếu có file changes trong `src/` hoặc `tests/`, chọn `implementation` sau khi readiness đủ.
4. Nếu cần đánh giá output của agent, chọn `review`.
5. Nếu cần command, test, log, artifact, chọn `testing` hoặc verification workflow.
6. Nếu chạm docs, chọn `documentation`.
7. Nếu chạm manifest, sources, prompts, taxonomy, chọn `kms-maintenance`.
8. Nếu chạm module notes, assumptions, contradictions, lessons learned, chọn `workspace-update`.

### Step 3. Select prompt contracts

Chọn prompt module theo thứ tự:

1. Prompt role level như architect, coder, reviewer, debugger, verifier.
2. Prompt module level như application runtime, application logging, renderer, capture.
3. Prompt validation hoặc QA nếu task cần evidence.
4. Prompt KMS update nếu có artifact chính thức mới.

Nếu không có prompt module cụ thể, dùng lane contract trong `prompts.md` và ghi gap `no module prompt found`.

Không tạo prompt lane hoặc prompt file mới ngoài registry trong lúc routing. Nếu cần lane mới, trả về `NEEDS_HUMAN_REVIEW` hoặc chuyển sang KMS maintenance với đề xuất cập nhật `docs/grounding/prompts.md`, `manifest.yaml`, và `sources.md`.

### Step 4. Resolve required references

Từ prompt contract, lấy:

1. Required references.
2. Constraints.
3. Forbidden actions.
4. Response policy.
5. Verification expectation.

Nếu references là source ID, resolve qua `manifest.yaml` hoặc `sources.md`. Nếu không resolve được, ghi `unresolved source id`.

### Step 5. Build execution prompt

Execution prompt cuối cùng phải có các phần:

1. Role.
2. Task.
3. Required sources.
4. Context summary.
5. Scope.
6. Files in scope.
7. Files out of scope.
8. Forbidden actions.
9. Workflow.
10. Output contract.
11. Verification method.
12. Failure policy.

### Step 6. Check prompt safety

Trước khi dùng execution prompt, kiểm tra:

1. Có role chưa.
2. Có source chưa.
3. Có scope chưa.
4. Có forbidden actions chưa.
5. Có verification method chưa.
6. Có output contract chưa.
7. Có xử lý insufficient grounding chưa.
8. Có cấm overclaim verified chưa.

Nếu thiếu, không chuyển sang implementation. Giữ `analysis` hoặc `planning` lane.

## Execution prompt template

```text
Bạn là <execution role>.

Task:
<task summary>

Required sources:
<resolved source paths>

Context:
<context pack summary>

Scope:
<allowed scope>

Forbidden actions:
<forbidden actions>

Workflow:
1. Đọc required sources.
2. Tóm tắt current state.
3. Nêu assumptions và open questions.
4. Thực hiện đúng lane.
5. Verify theo method được chỉ định.
6. Báo final report đúng output contract.

Output contract:
<sections>

Failure policy:
Nếu thiếu source, thiếu AC, thiếu test data hoặc thiếu environment, không claim done. Trả về BLOCKED, PARTIALLY VERIFIED hoặc NEEDS HUMAN REVIEW tùy trường hợp.
```

## Usage and verification

Use this skill by producing a Prompt Routing Result before agent execution. Verify completion by checking that every selected prompt file exists or is explicitly reported unresolved, every source ID resolves to a path, and no new lane is invented outside `docs/grounding/prompts.md`.

## Failure modes

1. Chọn coder prompt cho task chưa qua spec clarity.
2. Dùng prompt module nhưng không đọc references.
3. Prompt module claim hiện trạng không được repo chứng minh.
4. Quên forbidden actions.
5. Không có verification expectation.
6. Không cập nhật `prompts.md` khi thêm prompt mới.

## Done criteria

Skill hoàn tất khi agent biết chính xác lane nào dùng, prompt contract nào dùng, source nào phải đọc và output phải có bằng chứng gì.
