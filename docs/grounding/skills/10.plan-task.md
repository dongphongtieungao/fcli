# Skill 10: Plan Task

## Purpose

Create a concise implementation or review plan for a repo task by orchestrating the other SDD skills. Use when the user asks to plan, analyze before doing, break down work, identify modules/files to read or change, define verification, or decide which SDD skills should run before implementation.

## Context

This skill is the thin planning orchestrator. It does not duplicate the detailed SDD procedures in Skills 1–9; it calls them when their trigger applies. The output is a structured plan that a human or agent can approve before implementation begins.

## When To Use

- User asks to plan, scope, or break down work.
- Task is non-trivial and touches code, docs, contracts, or architecture.
- User asks "what should we do before implementing X?"
- Agent needs to determine which SDD skills apply to a request.
- Task crosses module boundaries or affects multiple layers (docs + code + tests).

## Skill Orchestration Order

1. **Intake** — Use `sdd-request-intake` (Skill 1) for task type, risk, impacted domain/runtime model, scope, forbidden actions, and readiness.
2. **Retrieval** — Use `kms-retrieval-context-pack` (Skill 2) when context must be gathered from policy, grounding, docs, tests, `llm-wiki/`, or source.
3. **Routing** — Use `prompt-lane-router` (Skill 3) when a lane or prompt contract is needed.
4. **Acceptance** — Use `spec-acceptance-mapping` (Skill 4) when behavior, generated output, application, data contract, runtime flow, or business rules may change.
5. **Boundary** — Use `code-boundary-file-role` (Skill 5) before touching important source files or model-specific code.
6. **Verification** — Use `evidence-verification` (Skill 6) to define the verification plan and later verify the result.
7. **Sync** — Use `kms-sync-drift-control` (Skill 7) when durable observations, official docs, prompts, skills, manifest, sources, or drift may need sync.

Not all skills are needed for every task. Select only the ones triggered by the task scope and risk.

## Planning Rules

1. Read `AGENTS.md` first for non-trivial tasks.
2. If the task touches `docs/`, read `docs/AGENTS.md`.
3. If the task touches a subtree with local policy, identify the impacted module and read the local `AGENTS.md`.
4. Keep the plan scoped; list files out of scope as well as in scope.
5. Do not propose implementation until the readiness gate and verification method are clear.
6. Do not skip risk classification. Low-risk tasks may need fewer skills; high-risk tasks need all applicable skills.
7. If the plan reveals a blocking gap (missing spec, contradiction, missing source), report `BLOCKED` with the gap and do not proceed to implementation.

## Output Template

```text
Plan Task
Task summary:
Task type:
Risk level:
Impacted domain/runtime model:
Required skills:
Grounding sources to read:
Files to inspect:
Files likely to change:
Files out of scope:
Implementation steps:
Verification plan:
KMS sync plan:
Blockers or open questions:
Ready status:
```

## Common Mistakes

1. Skipping intake and jumping directly to implementation planning.
2. Including files out of scope in the change plan.
3. Not checking for contradictions or open questions before proposing implementation.
4. Not defining verification plan before starting work.
5. Using all skills for a trivial task (over-engineering the planning phase).
6. Not reading `AGENTS.md` or local policy before planning.

## Verification

Use this skill before non-trivial implementation. Verify completion by checking that the plan has: clear scope, risk classification, identified grounding sources, files in/out of scope, implementation steps, verification method, and KMS sync decision. The plan must not contain unresolved blocking gaps.
