# JSON Schema Governance Rules

## Purpose

Define reusable governance expectations for JSON Schema, config schemas, tool schemas, and machine-readable data contracts.

## Context

This starter repository currently has example schemas only. Add project-specific schema rules when real contracts exist, but keep these baseline controls.

## Rules

| Rule | Guidance |
|---|---|
| Stable IDs | Schemas should have stable `$id` or documented path identity when used by tooling. |
| No secrets | Examples must not include credentials, tokens, or sensitive data. |
| Explicit required fields | Required fields should match the contract, not implementation convenience. |
| Versioning | Breaking schema changes should be documented. |
| Parseability | Schemas committed to the repo should parse in the project toolchain. |
| Traceability | Schemas should link to requirement, API/application contract, or data governance source. |
| Compatibility | Breaking changes should name migration or compatibility impact. |

## Schema Review Checklist

- [ ] Schema parses as valid JSON or YAML.
- [ ] Required fields match accepted behavior.
- [ ] Examples are synthetic and non-sensitive.
- [ ] Breaking changes are documented.
- [ ] Tests or validation commands are linked when available.

## Promotion Rules

Promote schema rules when repeated contract changes need consistent validation. Keep one-off schema notes in task verification unless they become reusable.
