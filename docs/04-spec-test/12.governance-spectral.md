# API Lint Governance

## Purpose

Define reusable lint and review expectations for OpenAPI or comparable API contracts.

## Context

Use this document when the project introduces OpenAPI or another API contract format.

## Starter Rules

| Area | Guidance |
|---|---|
| Info metadata | API contracts should have title, version, and description. |
| Operation clarity | Public operations should have summaries and clear responses. |
| Error model | Error responses should be consistent. |
| No sensitive examples | Examples must not include secrets or private data. |
| Operation identity | Operations should have stable identifiers when used by tooling or tests. |
| Schema reuse | Shared request/response shapes should be modeled consistently. |
| Compatibility | Breaking changes should name migration or versioning impact. |

## API Contract Checklist

- [ ] Contract parses.
- [ ] Public operations have summary, responses, and error behavior.
- [ ] Request/response schemas are linked or embedded consistently.
- [ ] Examples are synthetic and non-sensitive.
- [ ] QA matrix names the validation command once available.

## Promotion Rules

Promote API lint rules when they should apply across endpoints, services, or releases. Keep endpoint-specific findings with the contract or task evidence.
