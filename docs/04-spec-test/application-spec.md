# Application Spec — OpenCode PrivateGPT Bridge MVP

## Purpose

Define the observable application contract for installing, configuring, operating and calling the local OpenCode PrivateGPT Bridge. This document connects the canonical requirement, business flow, executable acceptance scenarios and machine-readable OpenAPI contract.

## Context

The Bridge is a Python modular monolith running on the developer machine. OpenCode CLI is the only MVP coding client. The Bridge exposes an OpenAI-compatible local boundary and translates requests to PrivateGPT Model, Agent and Conversation APIs. OpenCode retains ownership of workspace access, file reading/editing, command execution and user approval.

This specification is `draft` because the exact payload generated by the pilot version of `@ai-sdk/openai-compatible`, the current PrivateGPT SSE wire schema and complete HTTP error mapping still require redacted runtime evidence. Confirmed behavior may be implemented; unresolved wire details must not be guessed.

## Scope

### In scope

1. Local installation and configuration of the Python Bridge.
2. Browser login, secure refresh-token storage and in-memory access-token management.
3. Local server lifecycle and diagnostics.
4. Public alias `gemini-2.5-pro` with no model fallback.
5. Workspace-scoped PrivateGPT Agent synchronization.
6. OpenAI-compatible model discovery and chat completion for OpenCode.
7. Non-streaming JSON and streaming SSE responses.
8. Pre-stream retry boundary, cancellation propagation and safe error mapping.
9. OpenCode provider configuration generation.
10. Text context supplied by OpenCode, including file content already rendered into messages.

### Out of scope

1. LiteLLM as a mandatory runtime, multi-provider/model routing or a central gateway.
2. `/v1/responses`, WebSocket, image input or other unconfirmed OpenAI surfaces.
3. Bridge-side file access, workspace mutation, command execution or approval.
4. IntelliJ runtime, ToolRegistry and plugin-style `<tool_call>` execution.
5. Full chat-history persistence.
6. PrivateGPT file upload for text content already supplied in OpenCode messages.
7. Native tool calling and binary/multimodal attachments until the compatibility spike confirms their wire contract.

## Actors and boundaries

| Actor / system | Responsibility | Must not do |
|---|---|---|
| Developer | Install, login, operate and diagnose the Bridge; approve OpenCode actions. | Supply secrets in logs or committed configuration. |
| OpenCode CLI | Build coding context, call the local provider, manage tools, workspace actions and permissions. | Delegate workspace/tool execution to the Bridge. |
| Bridge CLI | Manage lifecycle, configuration, login, Agent synchronization and diagnostics. | Implement an independent coding workflow. |
| Bridge local API | Validate and map OpenAI-compatible requests/responses. | Expose PrivateGPT internal APIs directly. |
| PrivateGPTProvider | Implement Provider SPI using PrivateGPT Model, Agent and Conversation APIs. | Leak upstream wire details above the provider boundary. |
| Secure token store | Persist approved refresh/session material. | Store access tokens or secrets in plaintext files. |

## CLI contract

| Command | Required behavior | Observable result | Known gap/failure |
|---|---|---|---|
| `install` | Create runtime directory, install dependencies, generate config/API key and validate Python. | Runtime/config become available. | Exit codes and partial-install rollback unspecified. |
| `login` | Browser login, localhost callback, token exchange, secure refresh-token save and reachability check. | Login status available; access token memory-only. | Report login/token failure without token values. |
| `logout` | Clear locally retained authentication material. | Authenticated operations require login. | Exact output/exit code unspecified. |
| `start` | Validate config/auth/model/Agent, start loopback server and print provider endpoint. | Local endpoints become reachable. | Stop on auth/model/Agent/port/config failure. |
| `stop` | Stop local process. | Endpoints cease accepting requests. | Exact process semantics unspecified. |
| `restart` | Stop then start with current config. | Readiness checks rerun. | Must not report ready after failed checks. |
| `status` | Report process/application readiness. | Safe status only. | Status schema unspecified. |
| `doctor` | Check Python, config, port, token store, PrivateGPT, model, Agent, OpenCode hint and redaction. | Diagnostic findings. | Exit-code policy open. |
| `logs` | Show/access logs. | Redacted diagnostics only. | No tokens, raw prompts or source code. |
| `agent sync` | Resolve model, build/hash instructions and create/update/reuse Agent. | Cache Agent metadata. | Explicit failure; no inline fallback. |
| `agent status` | Report Agent binding/readiness. | Safe Agent metadata. | Output schema unspecified. |
| `agent reset` | Reset local binding for resynchronization. | Old binding not silently reused. | Remote deletion behavior open. |
| `opencode config` | Produce provider configuration. | Env-based API key and model alias. | Never print actual key value. |

Stable CLI exit codes and JSON output are not defined. Automation must not depend on them until approved.

## Local HTTP contract

The machine-readable contract is [10.openapi.yaml](../../specs/10.openapi.yaml). Default server: `http://127.0.0.1:4100`; host and port are configurable, and default binding must not be `0.0.0.0`.

| Operation | Required input | Successful result | Rules |
|---|---|---|---|
| `GET /health` | None defined. | Successful reachability response. | Body schema remains open. |
| `GET /v1/models` | Local authentication per OpenAPI. | List containing only `gemini-2.5-pro`. | No alternate model or fallback. |
| `POST /v1/chat/completions` | Bearer local API key; `model`, `messages`; optional `stream`. | JSON or SSE ending `[DONE]`. | Validate first; require Agent; never execute tools. |

### OpenCode provider profile

```json
{
  "provider": {
    "privategpt": {
      "npm": "@ai-sdk/openai-compatible",
      "options": {
        "baseURL": "http://127.0.0.1:4100/v1",
        "apiKey": "{env:PRIVATEGPT_ADAPTER_API_KEY}"
      },
      "models": {
        "gemini-2.5-pro": { "name": "PrivateGPT Gemini 2.5 Pro" }
      }
    }
  },
  "model": "privategpt/gemini-2.5-pro"
}
```

The actual installed provider request must be captured before fields beyond the confirmed subset are declared supported.

## Core application flows

### Startup

1. Load/validate configuration and secure-token-store access.
2. Check authentication.
3. Resolve internal model ID for `gemini-2.5-pro`.
4. Synchronize workspace Agent.
5. Bind loopback host/port.
6. Report ready/provider endpoint only after all checks succeed.

### Authentication

1. Browser login returns through localhost OAuth callback.
2. Refresh token is stored securely; access token remains memory-only.
3. Refresh before expiry.
4. Upstream HTTP 401 triggers one forced refresh and retry.
5. Further failure maps `login_required` or `token_refresh_failed`.

### Agent synchronization

1. Derive workspace hash/name.
2. Resolve and explicitly set internal Gemini model ID; never inherit the plugin's legacy default.
3. Build/hash OpenCode-compatible instructions.
4. List, then create/update/reuse Agent as required.
5. Cache safe metadata only.
6. Fail explicitly without inline-instruction fallback.

### Chat completion

1. Validate API key, public model and supported request subset.
2. Ensure upstream token and workspace Agent.
3. Preserve caller-supplied system/developer/user content.
4. Render messages into Agent Mode question.
5. Build ChatRequest with explicit internal model ID, `metadata.agent_id`, `tools: []`, and null conversation/parent IDs for MVP.
6. Call PrivateGPT Conversation API using Bearer auth and SSE.
7. Parse visible content, filter reasoning and capture IDs when present.
8. Return JSON for non-stream or mapped chunks plus `[DONE]` for stream.
9. Leave workspace/tool/approval actions to OpenCode.

## File and context handling

OpenCode can select files using `@file` or `--file`, but that UI does not prove a separate provider upload request. MVP rules:

1. File content supplied as text in `messages` is rendered into the PrivateGPT question.
2. Bridge never reads the workspace path itself.
3. Bridge does not call PrivateGPT file-upload API for text already present in messages.
4. Upstream `metadata.attachments` remains empty for this flow.
5. Binary/image/PDF, structured content parts and separate upload calls remain unsupported pending a redacted request capture.
6. Future upload support requires conversation-scoped lifecycle, consent, MIME/size limits, retention and cleanup; Agent persistence is not the default.

## Request mapping contract

| Public field/context | PrivateGPT mapping | Rule |
|---|---|---|
| `model` | Internal `model_id` | Only `gemini-2.5-pro`; explicit resolution; no fallback. |
| `messages` | Rendered `question` | Preserve system/developer content. |
| `stream=false` | Collect upstream SSE | Return one JSON response. |
| `stream=true` | Relay mapped chunks | End `[DONE]`; no retry after first relayed chunk. |
| Agent binding | `metadata.agent_id` | Mandatory. |
| Text file content | Rendered `question` | No upload and no filesystem access. |
| `tools`/`tool_choice`/tool results | Unconfirmed | Safely ignore only when proven safe; otherwise reject; never execute. |
| Session IDs | `conversation_id`, `parent_message_id` | `null` unless explicit session mapping is implemented. |

## State and persistence

| State | Storage/lifetime | Prohibited content |
|---|---|---|
| Refresh/session credential | Approved secure local store until logout/revocation. | Plaintext config/log copies. |
| Access token | Process memory until expiry/stop. | Disk persistence and logs. |
| Adapter API key | Approved local secret/config mechanism. | Examples, logs and source control. |
| Model resolution | Local cache; TTL not fixed. | Alternate-model fallback. |
| Agent metadata | Workspace cache until reset/change. | Prompts, chat history or source code. |
| Request diagnostics | Safe request metadata under operational retention. | Raw messages, source code, tokens/cookies. |
| Conversation history | Not persisted by Bridge by default. | Global cross-workspace conversation reuse. |

| Section | Confirmed behavior |
|---|---|
| `server` | Host `127.0.0.1`, port `4100`, configurable. |
| `privategpt` | Base URL and request timeout. |
| `model` | Public alias, optional explicit internal ID, discovery control. |
| `auth` | Browser login, token-store selection, refresh-before-expiry. |
| `agent` | Enabled, workspace name template, instruction sources/migration. |
| `openai_compat` | Local API-key mode and unsupported model/tool rejection. |

| `streaming` | Enabled and bounded buffering. |
| `resilience` | Pre-stream-only retry and circuit-breaker enablement. |

Configuration precedence, broader environment-variable names, schema validation and secure-store variants remain open.

## Validation and business rules

| ID | Rule | Observable result |
|---|---|---|
| APP-RULE-001 | Loopback-only default. | No default `0.0.0.0`. |
| APP-RULE-002 | Public model equals `gemini-2.5-pro`. | `model_not_supported`; no upstream fallback. |
| APP-RULE-003 | Internal model must resolve/configure. | `model_not_available`; no substitution. |
| APP-RULE-004 | Workspace Agent mandatory. | Every conversation includes `metadata.agent_id`. |
| APP-RULE-005 | Bridge never executes tools. | Unsupported requirement rejected or proven-safe ignored. |
| APP-RULE-006 | Plugin `<tool_call>` is invalid output. | `privategpt_agent_instruction_mismatch`; no execution. |
| APP-RULE-007 | Text file content is caller context. | No Bridge file access/upload. |
| APP-RULE-008 | Retry only before client-visible output. | No whole-request retry after first chunk. |
| APP-RULE-009 | Cancellation propagates. | Upstream closes; no background request. |
| APP-RULE-010 | Sensitive/raw content redacted. | No token/cookie/raw prompt/source in logs/errors. |

## Error contract

| Condition | Required behavior | Identifier/gap |
|---|---|---|
| Missing/expired login | Refresh if permitted; otherwise request login. | `login_required` |
| Refresh failure | Stop after allowed retry. | `token_refresh_failed` or `login_required` |
| Unsupported model | Reject before upstream. | `model_not_supported` |
| Internal model unavailable | No fallback. | `model_not_available` |
| Agent failure | Explicit failure; no inline fallback. | Status/code open. |
| Unsupported tool payload | Reject when not safely ignorable. | Status/code open. |
| Plugin-style tool output | Do not execute; reset/sync mitigation. | `privategpt_agent_instruction_mismatch` |
| Upstream timeout/rate/unreachable | Bounded pre-relay handling; mapped error. | Mapping open. |
| Failure after first chunk | No whole-request retry. | Stream error framing open. |
| Client cancellation | Cancel/release upstream. | Client status open/not applicable. |
| Oversized response | Enforce limit. | Threshold/code open. |

OpenAPI intentionally uses a generic error schema until statuses and payload fields are verified.

## Streaming and resilience

1. Upstream uses `Accept: text/event-stream`.
2. Plugin evidence observes `data`, `mes_info`, `conv`, `hint`, `DONE`; runtime capture is still required.
3. Reasoning/thinking markers are filtered.
4. Non-stream mode accumulates bounded visible content.
5. Stream mode relays mapped chunks and `[DONE]`.
6. Retry is bounded and pre-first-chunk only.
7. Disconnect cancels upstream work.
8. Timeout, buffer and circuit-breaker thresholds remain configurable/open where not approved.

## Security, data and observability

1. Network exposure defaults to loopback.
2. Local `/v1` calls use the generated adapter API key; upstream calls use OAuth Bearer tokens.
3. Refresh token is securely stored; access token is memory-only.
4. Bridge does not open workspace files or infer permissions from text paths.
5. Source text received from OpenCode is transmitted to configured PrivateGPT; organizational retention/approval policy remains open.
6. Diagnostics should include timestamp, level, correlation ID, component, operation/route, provider, status, latency and resilience state.
7. Diagnostics must exclude secrets, raw messages, raw source code, cookies and full upstream bodies.

Exact log schema, metrics and retention remain unspecified; tests should validate redaction and lifecycle signals, not unapproved formatting.

## Open questions and implementation gates

1. Exact OpenCode 1.17.9 payload for basic chat, tools, results and `@file`/`--file`?
2. Does MVP require native tool calling, or is text-only output sufficient?
3. Approved HTTP statuses and error JSON fields?
4. Current PrivateGPT model, Agent and SSE runtime schemas?
5. Approved timeout, maximum response, retry and circuit-breaker values?
6. Approved secure token store for each target OS?
7. Retention/audit policy for internal source text sent to PrivateGPT?
8. Stable CLI exit codes/machine-readable outputs required?
9. Is Agent reset local-only or remote?

These gaps do not block the confirmed server, model whitelist, text-message mapping, Agent requirement and no-tool-execution boundary. They block claims of complete OpenCode wire compatibility and binary/structured attachment support.
