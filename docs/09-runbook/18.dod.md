# Definition Of Done

## Purpose

Define reusable completion evidence for analysis, implementation, documentation, package, release, and KMS tasks.

## Context

This DoD is generic by design. It defines evidence expectations before a task can be called done, even when the new project has not established full CI or release automation.

## Done Verdicts

| Verdict | Meaning |
|---|---|
| `DONE` | Requested scope is complete and evidence supports completion. |
| `PARTIAL` | Useful work is complete but verification, promotion, or approval remains. |
| `BLOCKED` | Completion is blocked by missing source, environment, decision, or failing evidence. |
| `NEEDS_HUMAN_REVIEW` | Human approval is required before acceptance or release. |

## Done Checklist

- [ ] Requested change is complete.
- [ ] Verification was run or explicitly reported as not available.
- [ ] Documentation and grounding impact were considered.
- [ ] No unrelated user changes were reverted.
- [ ] Final report includes changed areas, checks, and residual risk.
- [ ] Warnings/errors were read and classified.
- [ ] Promotion status is recorded when docs/specs/runbooks changed.
- [ ] Release/package tasks include install, smoke, rollback, known issues, and evidence paths when applicable.

## Evidence Pack

| Evidence | Required When |
|---|---|
| Changed files | Any task that edits files |
| Source-of-truth files read | Documentation, behavior, architecture, QA, or release tasks |
| Exact command and exit code | Tests, parsers, builds, smoke checks, or runtime checks |
| Output artifact path/metadata | Generated files, packages, reports, screenshots, or release evidence |
| Warning/error review | Any command or log-producing check |
| Residual risk | Partial verification, known gap, blocked path, or human review |

## Promotion Rules

If task completion creates durable knowledge, update official docs or record `SYNC_NEEDED` in `promotion.md`. Do not mark a task fully done when the implementation and official docs knowingly drift.
