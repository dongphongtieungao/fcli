# Definition of Done — OpenCode PrivateGPT Bridge Handover

## Purpose

Define when a scoped task or MVP increment is complete enough for handover. Done requires implemented behavior, passing applicable gates, inspected evidence, synchronized knowledge and a reusable handover packet—not an agent assertion or completion of documentation alone.

## Context

This DoD closes the artifact chain established by DoR. The same requirement/spec/scenario/QA/gate IDs selected at readiness must appear in implementation, tests, verification and handover evidence. Missing runtime automation or compatibility captures lower the verdict; they cannot be hidden behind a documentation pass.

## Done verdicts

| Verdict | Meaning | Handover action |
|---|---|---|
| `DONE_FOR_HANDOVER` | All applicable blocking gates pass and the evidence packet is complete. | Handover/merge/release within approved scope. |
| `DONE_WITH_ACCEPTED_GAPS` | Only approved non-blocking gaps remain with owner and next action. | Handover with explicit limitations; no broader claim. |
| `PARTIALLY_DONE` | Useful outputs exist but required implementation/test/runtime evidence is incomplete. | Do not represent scope as complete. |
| `NEEDS_HUMAN_REVIEW` | Business, security, data or release approval remains. | Handover decision waits for authorized reviewer. |
| `BLOCKED` | A required gate failed or prerequisite is unavailable. | Stop handover/merge/release. |
| `FAILED` | Evidence contradicts required behavior. | Correct implementation/spec/test before reassessment. |

## Universal done checklist

### Scope and implementation

- [ ] Delivered behavior matches the approved task packet and plan.
- [ ] No out-of-scope feature, provider/model, endpoint or dependency was added.
- [ ] Unrelated user changes were preserved.
- [ ] Code follows C4/ADR/standards and Provider SPI dependency direction.
- [ ] Configuration/defaults and backward-compatibility impact are documented.
- [ ] Rollback/recovery action is available for risky changes.

### Required Bridge boundaries

- [ ] Default bind remains loopback-only.
- [ ] Only `gemini-2.5-pro` is exposed and no model fallback exists.
- [ ] Internal model ID is explicit; no plugin legacy default is inherited.
- [ ] Workspace Agent is synchronized/idempotent and every conversation has `metadata.agent_id`.
- [ ] Bridge cannot execute tools, commands, edits or plugin `<tool_call>`.
- [ ] OpenCode remains responsible for workspace permission/approval.
- [ ] Text file content supplied in messages causes no Bridge filesystem read or PrivateGPT upload.
- [ ] Binary/structured attachments remain unsupported unless separately approved and specified.

### Auth, security and data

- [ ] Refresh/session material uses approved secure storage.
- [ ] Access token remains memory-only.
- [ ] No secret, API key, token, cookie or PII is committed or printed.
- [ ] Logs/errors redact raw prompts, source code and internal stack details.
- [ ] Synthetic/approved fixtures are used.
- [ ] Source transmission, retention and attachment impact are documented where affected.

### Streaming and resilience

- [ ] Non-stream response maps correctly.
- [ ] Streaming response emits mapped chunks and `[DONE]`.
- [ ] Retry occurs only before first client-visible chunk.
- [ ] Failure after first chunk does not restart the whole request.
- [ ] Client cancellation closes/releases upstream resources.
- [ ] Timeout, buffering and response-size controls are tested where affected.

### Tests and quality gates

- [ ] Every affected main flow has a happy-path scenario.
- [ ] Critical alternate/error flows have scenarios/tests.
- [ ] Applicable QA-F/QA-N rows have evidence.
- [ ] Applicable QG gates passed or have an approved non-blocking disposition.
- [ ] Unit, contract, integration, runtime and pilot levels were run as required by risk.
- [ ] Skipped/flaky/missing checks are visible and reflected in verdict.
- [ ] Review checklist is completed with reviewer decision.

### Documentation and KMS

- [ ] Requirement/application/OpenAPI/BDD remain aligned with delivered behavior.
- [ ] Architecture/ADR/standards were updated when decisions/boundaries changed.
- [ ] QA, quality and review artifacts reflect new reusable gates/evidence.
- [ ] Manifest, machine registry and source catalog are synchronized when required.
- [ ] Observations/contradictions/open questions are stored in the correct official/workspace layer.
- [ ] No stale placeholder, broken link or unsupported compatibility claim remains in the changed surface.

## Evidence pack contract

Every handover must include:

```text
Task ID and objective:
Final scope and changed files:
Requirement / architecture / spec IDs:
BDD scenarios:
PG / QA / QG IDs:
Implementation summary:
Commands/checks and command source:
Exit codes and result summary:
Warnings/errors/skips and classification:
Test reports, fixtures and artifact paths:
Runtime/log/redacted capture evidence:
Review checklist decision:
Security/data review result:
KMS sync result:
Rollback/recovery note:
Known limitations and accepted gaps:
Final verdict:
Reviewer/approver and date where required:
```

Claims without command/result or approved manual evidence cannot receive `DONE_FOR_HANDOVER`.

## Minimum evidence by task type

| Task type | Minimum evidence for done |
|---|---|
| Documentation/KMS | Structure/frontmatter/link/registry checks, diff review and source traceability. |
| Requirement/BPMN/BDD | Scenario/flow consistency, traceability and human confirmation where behavior changed. |
| Application/OpenAPI contract | YAML/parser/linter or justified substitute, examples, compatibility impact and QA mapping. |
| Python unit behavior | Targeted tests, import/static check and affected scenario mapping. |
| Auth/security | Unit/integration tests, secret scan, redaction evidence and secure-store decision. |
| Agent/provider | Provider fixtures, create/update/reuse/idempotency evidence and no-inline-fallback check. |
| SSE/resilience | Stream fixtures, first-chunk retry, cancellation and resource-release evidence. |
| OpenCode compatibility | Real client/provider/model run plus redacted request/response capture. |
| File/attachment | `@file` capture, approved mapping, no-unapproved-upload proof and data lifecycle decision. |
| Release/handover | Install/start/health/chat smoke, OpenCode pilot, artifact/version/checksum if packaged, rollback and approval. |

## Agent completion contracts

| Role | Done output before next role |
|---|---|
| Planner | Approved file-level plan, risk/gap map, tests and verification sequence tied to IDs. |
| Coder | Scoped implementation, tests and contract/docs sync; no unsupported claims. |
| Tester | Executed/defined tests mapped to QA rows with results, fixtures and missing coverage. |
| Verifier | Independent evidence pack, warnings/artifact inspection and final evidence verdict. |
| Reviewer | Completed checklist, blocking findings disposition and approval decision. |
| Handover owner | Consolidated package, limitations, rollback and next actions understandable without chat history. |

## Handover packet

The final packet must allow a new agent or human to continue without private conversation context:

1. Task packet/readiness record from DoR.
2. Final diff or changed-file inventory.
3. Links to requirement, architecture, application/OpenAPI and BDD.
4. QA row and quality-gate results.
5. Test/runtime/compatibility evidence artifacts.
6. Review decision and unresolved findings.
7. Configuration/install/run/rollback information affected by the task.
8. Known limitations, owners and next actions.
9. Final DoD verdict with scope-qualified wording.

## Blocking handover conditions

Handover is blocked by exposed secrets; model fallback; missing Agent binding; Bridge tool execution; unapproved file/upload behavior; retry after output; cancellation/resource leak; broken contract; missing critical scenario; failed blocking QA/QG row; source conflict; or runtime compatibility claims without evidence.

## Current project caveat

The documentation artifact chain is suitable for agent task handoff, but this does not make the application implementation `DONE_FOR_HANDOVER`. Until code, automated tests, runtime smoke, PrivateGPT fixtures/captures and the real OpenCode pilot exist, the overall product verdict is at most `PARTIALLY_DONE`.

## Done decision procedure

1. Load the DoR task packet and delivered diff.
2. Confirm scope and requirement/spec/scenario mapping did not drift.
3. Resolve and run applicable QA/QG checks.
4. Inspect warnings, logs and artifacts—not only exit codes.
5. Complete review checklist and security/data disposition.
6. Synchronize docs/tests/registry/workspace notes as required.
7. Assign one scope-qualified verdict and produce the handover packet.
