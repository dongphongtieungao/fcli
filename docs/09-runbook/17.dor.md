# Definition of Ready — OpenCode PrivateGPT Bridge

## Purpose

Define when a task is ready to be handed to planner, coder, tester and verifier agents without additional free-form chat. A ready task has bounded scope, authoritative sources, acceptance criteria, applicable quality gates and an explicit treatment of unresolved gaps.

## Context

The specification chain exists from requirement through architecture, BPMN, BDD, application/OpenAPI, guardrails, QA and review gates. Some wire-level details remain unverified. Readiness therefore applies per task and behavior, not to the repository as a single binary state.

## Readiness verdicts

| Verdict | Meaning | Allowed action |
|---|---|---|
| `READY_FOR_IMPLEMENTATION` | Scope, sources, contract, acceptance and verification are complete for the task. | Planner/coder/tester/verifier may execute their role contracts. |
| `READY_WITH_GAPS` | Gaps are explicit and do not affect the scoped behavior. | Proceed only inside confirmed boundaries; preserve gaps. |
| `DISCOVERY_ONLY` | Evidence is sufficient for investigation but not behavior implementation. | Capture payload/schema, analyze or draft options only. |
| `NEEDS_HUMAN_REVIEW` | Business, security, data or architecture decision is required. | Prepare decision material; do not choose silently. |
| `BLOCKED` | Source conflict, missing critical contract, unsafe action or failed prerequisite exists. | Stop implementation until resolved. |

## Required artifact baseline

| Layer | Required artifact | Role in handoff | Ready condition |
|---|---|---|---|
| Requirement | `00.requirement.md`, `01.use-cases.md` | Scope, actors, FR and acceptance. | Affected FR/UC identified; no unresolved conflict. |
| Architecture | Vision, C4, NFR, ADR, Standards | Boundaries, decisions and Python rules. | Affected components/decisions identified. |
| Business flow | `09.bpmn.md` | Main, alternate and error sequence. | Task maps to a flow/gateway/error. |
| Acceptance | `tests/features/*.feature`, BDD traceability | Observable behavior. | Happy path and critical scenario identified or explicitly added. |
| Application contract | `application-spec.md` | CLI/runtime/state/error/file rules. | Operation and side effects identified. |
| HTTP contract | `specs/10.openapi.yaml` | Request/response/auth/SSE schema. | Affected operation/schema known; draft gaps do not invalidate scope. |
| AI guardrails | `13.prompt-guardrails.md` | Forbidden behavior and evidence claims. | Applicable PG IDs selected. |
| QA | `14.qa-matrix.md` | Tests/fixtures/evidence. | Applicable QA-F/QA-N rows selected. |
| CI/PR | `15.quality-matrix.md`, `16.review-checklists.md` | Merge gates and review criteria. | Applicable QG IDs/checklist sections selected. |
| Completion | `18.dod.md` | Final evidence and handover. | Expected DoD evidence named before work starts. |

## Task handoff packet

Every implementation task must provide this structured packet:

```text
Task ID:
Objective:
Task type and risk:
Affected actor/flow:
Requirement IDs:
Architecture/ADR/standard IDs:
Application/OpenAPI operations:
BDD scenarios:
Files in scope:
Files out of scope:
Allowed behavior change:
Forbidden actions:
Applicable prompt guardrails:
Applicable QA rows:
Applicable quality gates:
Expected tests/fixtures:
Verification commands or command-resolution source:
Expected evidence artifacts:
Known gaps and assumptions:
Rollback/recovery expectation:
Readiness verdict:
```

Blank values must be written as `not applicable` or an explicit gap; they must not be silently omitted.

## Universal ready checklist

### Scope and source

- [ ] Objective and observable outcome are defined.
- [ ] Files and behaviors in/out of scope are listed.
- [ ] Affected FR/UC and business flow are identified.
- [ ] No higher-priority source conflict remains unresolved.
- [ ] Reference plugin/code observations are not treated as product requirements.

### Architecture and contract

- [ ] Affected C4 component and dependency direction are known.
- [ ] Applicable ADR/NFR/standards are identified.
- [ ] CLI/application operation or HTTP path is identified.
- [ ] Input, output, state changes, side effects and error paths are known.
- [ ] Model, Agent, tool, file and workspace boundaries are explicit.
- [ ] Backward-compatibility or migration impact is known.

### Acceptance and QA

- [ ] At least one happy-path scenario covers each affected main flow.
- [ ] Critical alternate/error scenario is defined.
- [ ] Applicable QA-F/QA-N rows and QG gates are selected.
- [ ] Expected fixtures, mocks, runtime environment and sensitive-data policy are known.
- [ ] Smallest meaningful verification and expected artifact are named.

### Security and operations

- [ ] Secret/token/logging impact is classified.
- [ ] File/source-content transmission and retention impact is classified.
- [ ] Retry, cancellation, timeout and cleanup impact is known where applicable.
- [ ] Destructive/external actions are absent or explicitly authorized.
- [ ] Rollback/recovery path is stated for risky changes.

### Agent execution

- [ ] Planner/coder/tester/verifier role outputs are defined.
- [ ] Forbidden actions include no model fallback, Bridge tool execution or unapproved upload.
- [ ] Unverified wire behavior is marked as a capture/decision gate.
- [ ] Expected DoD verdict and handover evidence are known.

## Readiness by role

| Role | Required inputs | Required output | Must not proceed when |
|---|---|---|---|
| Planner | Task packet, architecture, spec, BDD, QA/gates. | Bounded implementation plan, file list, dependencies, risks and verification sequence. | Source/contract conflict or unknown boundary changes scope. |
| Coder | Approved plan, contracts, scenarios, PG/QA/QG IDs. | Minimal implementation plus scoped tests/docs updates. | Behavior requires guessing schema/rule or unsafe external action. |
| Tester | Acceptance scenarios, contracts, implementation boundary, fixtures policy. | Test plan/cases/automation and evidence mapping. | Expected outcome or approved fixture is missing. |
| Verifier | QA rows, quality gates, commands, artifacts and diff. | Evidence pack and verdict with warnings/gaps. | Command source/environment/evidence is insufficient for the claim. |

## Task-type readiness additions

| Task type | Additional ready criteria |
|---|---|
| Documentation/spec | Source links, lifecycle status, registry impact and link checks known. |
| CLI/config | Command input/output, config precedence, secret handling and exit behavior known or scoped as open. |
| HTTP contract | OpenAPI operation/schema and compatibility impact identified. |
| Auth/security | Secure-store decision, synthetic fixtures and redaction assertions identified. |
| Agent/provider | Upstream fixture/schema source and Agent idempotency behavior identified. |
| SSE/resilience | Event fixtures, first-chunk boundary and cancellation assertion identified. |
| Tool compatibility | Actual OpenCode capture and human scope decision available. |
| File/attachment | Text-vs-binary behavior, consent, retention and cleanup decision available. |
| Release/handover | Install, runtime smoke, rollback, known issues and approval owner identified. |

## Current known gates

The following confirmed work may be `READY_WITH_GAPS`: local server skeleton, loopback/config validation, model whitelist, Agent abstraction/sync policy, text-message mapping, no-tool-execution boundary, OpenAPI models/non-stream shapes and unit tests based on confirmed rules.

The following remain `DISCOVERY_ONLY` or `BLOCKED` unless evidence/decision is provided:

1. Exact OpenCode native tool payload and response expectations.
2. Binary/structured `@file` attachment support or new upload API.
3. Complete PrivateGPT runtime Agent/SSE schemas beyond observed reference evidence.
4. Stable HTTP error statuses/payloads and CLI machine-output/exit codes.
5. Cross-platform secure-store implementation where no approved decision exists.
6. Release claims without runtime installation and OpenCode pilot evidence.

## Readiness decision procedure

1. Build the task packet from official artifacts.
2. Select affected PG, QA and QG IDs.
3. Check universal and task-type criteria.
4. Classify each gap as blocking or non-blocking for this scope.
5. Record one readiness verdict and rationale.
6. Hand the same packet to all agents; later roles append evidence rather than reinterpret scope.

## Ready record

```text
Task ID:
Readiness verdict:
Scope summary:
Sources resolved:
Acceptance scenarios:
PG / QA / QG IDs:
Blocking gaps:
Non-blocking gaps:
Human decisions required:
Expected verification/evidence:
Approved next role:
Recorded by / date:
```

## Transition condition

A task can move to implementation when the packet can be consumed without hidden chat context, the affected behavior maps through spec and acceptance, and every blocking gap is resolved. `READY_WITH_GAPS` must preserve the gaps in the plan, tests and final evidence pack.
