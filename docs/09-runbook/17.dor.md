# Definition Of Ready

## Purpose

Define reusable readiness criteria before starting analysis, implementation, package, release, or documentation promotion work.

## Context

This DoR is generic by design. It should be used to decide whether a task is ready, needs clarification, or should be limited to discovery.

## Readiness Levels

| Level | Meaning | Allowed Work |
|---|---|---|
| `READY` | Scope, sources, risk, and verification are clear enough. | Proceed with planned work. |
| `READY_WITH_GAPS` | Gaps are known and safe assumptions are recorded. | Proceed within explicit limits. |
| `DISCOVERY_ONLY` | Requirement/source is unclear. | Analyze, ask questions, draft options. |
| `BLOCKED` | Missing decision, source conflict, unsafe action, or environment blocker. | Do not implement until resolved. |

## Ready Checklist

- [ ] Objective and expected outcome are clear.
- [ ] Files in scope and out of scope are known.
- [ ] Source-of-truth docs or gaps are identified.
- [ ] Acceptance criteria or verification expectations are known.
- [ ] Risks, dependencies, and open questions are recorded.
- [ ] Promotion target is known when official docs may change.
- [ ] Human decision owner is known for business, security, compliance, or release decisions.

## Task-Type Additions

| Task Type | Additional Ready Criteria |
|---|---|
| Requirement analysis | Problem, stakeholder, scope, and open questions are captured. |
| Business behavior change | Current and expected behavior have source or explicit gap. |
| Code change | Relevant boundary, contract, caller/callee, and test surface are known. |
| Documentation/KMS | Artifact layer, related links, and source-of-truth impact are known. |
| Package/release | Artifact scope, install/run path, smoke check, rollback, and approval owner are known. |

## Promotion Rules

If a readiness rule becomes project-specific, promote it into this DoR or a project runbook. Do not hide readiness gaps in chat; record them in task intake or promotion notes.
