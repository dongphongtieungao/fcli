# Data Governance

## Purpose

Define a reusable data governance baseline for any project created from this repository.

## Context

The project domain is intentionally not assumed. This document defines the minimum governance decisions that should be made before a project stores, transforms, exports, logs, or releases data.

Use this file as the active governance baseline. Use [Template/00.data-governance.md](./Template/00.data-governance.md) when creating a project-specific data governance document.

## Governance Principles

| Principle | Rule | Evidence |
|---|---|---|
| Ownership | Every durable dataset, export, fixture, or generated artifact must have an owner or explicit `unknown`. | Owner list, RACI, or source catalog |
| Classification | Sensitivity must be known before data is stored, logged, shared, or committed. | Classification matrix or data inventory |
| Minimization | Store and expose only the fields needed for the stated purpose. | Data contract, schema, or review note |
| Traceability | Important data movement must be traceable from source to output. | Lineage note, runbook, test, or verification evidence |
| Quality | Data quality checks must align with acceptance criteria and operational risk. | QA matrix, validation test, or smoke check |
| Access | Least privilege is the default for systems, users, service accounts, and agents. | Access model or deployment runbook |
| Retention | Retention and deletion rules must be explicit for durable data and generated artifacts. | Retention policy or artifact lifecycle note |
| Reproducibility | Important transformations must be reproducible from documented inputs and commands. | Runbook, command evidence, or test fixture |
| Redaction | Secrets, credentials, PII, and sensitive values must not be exposed in logs, docs, prompts, fixtures, or release evidence. | Review checklist, redaction rule, or secret scan |

## Decision Matrix

Use this matrix when a task introduces or changes data.

| Question | Required Decision | Promotion Target |
|---|---|---|
| What data enters the system? | Source, owner, format, update frequency | `docs/grounding/sources.md`, requirement, contract |
| What data leaves the system? | Output contract, consumer, sensitivity | Contract/spec, QA matrix, runbook |
| What data is persisted? | Storage location, lifecycle, retention | Architecture, data governance, runbook |
| What data is logged? | Allowed fields, redaction, correlation ID | Logging standard, prompt guardrails, QA |
| What data is used for tests? | Fixture source, anonymization, validity | Tests, QA matrix, fixture README |
| What data is released? | Artifact owner, checksum, retention, known limits | Release evidence, DoD, runbook |

## Task Checklist

Before implementation or release, answer the items that apply:

- [ ] Data owner or `unknown` is recorded.
- [ ] Data classification is recorded.
- [ ] Sensitive values are excluded or redacted.
- [ ] Input and output contracts are linked.
- [ ] Retention and deletion expectations are known.
- [ ] Test fixtures are synthetic, anonymized, or approved.
- [ ] Data quality checks are mapped to acceptance criteria.
- [ ] Runtime or release evidence does not leak sensitive data.
- [ ] Required docs promotion is recorded in `promotion.md`.

## Promotion Rules

Promote data governance decisions when they become durable project knowledge:

| Change | Promote To |
|---|---|
| New data source | `docs/grounding/sources.md`, requirements, architecture |
| New data contract or schema | `docs/04-spec-test/`, `specs/` |
| New sensitivity or retention rule | This file or a project-specific data governance doc |
| New quality gate | `docs/06-quality-assurance/14.qa-matrix.md` |
| New install/runtime handling | `docs/09-runbook/` |
| New release evidence requirement | `docs/10-release/` and `docs/09-runbook/18.dod.md` |

Do not promote raw samples, secrets, production data, or unredacted logs into `docs/` or `llm-wiki/`.

## Open Questions For New Projects

- What data sources are in scope?
- Which data is customer-owned, regulated, confidential, or public?
- Which artifacts are durable, temporary, generated, or release-bound?
- What is the minimum fixture data needed for tests?
- Who approves retention, deletion, access, and release evidence?
