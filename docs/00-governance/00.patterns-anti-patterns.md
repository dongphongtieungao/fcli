# Patterns And Anti-Patterns

## Purpose

Capture reusable engineering, documentation, agent-workflow, verification, and cleanup patterns.

## Context

This file records practices worth reusing across projects. It should prefer behavior-shaping patterns over generic advice, and each anti-pattern should include a safer alternative.

## Patterns

| Pattern | Use When | Benefit |
|---|---|---|
| Spec before implementation | Behavior is user-visible or operationally important | Reduces ambiguity |
| Small scoped changes | The task touches shared files | Lowers regression risk |
| Source-linked docs | A doc controls code or verification | Improves traceability |
| Workspace before promotion | Findings are not yet official | Keeps docs clean |
| Registry preservation | Manifest, sources, prompts, or taxonomy are edited | Keeps agent routing usable |
| Evidence-first completion | A task claims done, verified, or release-ready | Avoids unsupported completion claims |
| Generic framework restore | Old domain docs are removed | Preserves reusable structure without old assumptions |

## Anti-Patterns

| Anti-pattern | Risk | Safer Alternative |
|---|---|---|
| Deleting by keyword only | Removes reusable scaffold | Classify by role first |
| Treating workspace notes as official | Creates drift | Promote with evidence |
| Claiming unrun verification | Misleads users | Report partial or pending checks |
| Creating orphan docs | Hard to maintain | Add real related links |
| Over-sanitizing registries | Makes manifest/sources useless | Keep source IDs, coverage, and usage rules |
| Leaving old terminology in skills | Reintroduces old project bias | Scan skills/prompts/methodology together |
| Replacing content with empty placeholders | Reduces reuse value | Convert into reusable framework/checklist |

## Reuse Review Questions

- Does this pattern help future agents or developers make a better decision?
- Is the pattern generic without becoming vague?
- Does the anti-pattern name a concrete failure mode?
- Is there a safer alternative that can be checked during review?

## Promotion Rules

Promote a lesson into this file when it is reusable across future tasks. Keep task-specific lessons in `llm-wiki/lessons-learned.md` until they repeat or become policy.
