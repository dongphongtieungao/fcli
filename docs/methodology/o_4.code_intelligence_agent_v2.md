# Code Intelligence Agent v2

## Purpose

Tài liệu này định nghĩa thiết kế tinh gọn cho **Code Intelligence Agent v2** trong mô hình Spec Driven Development, KMS, Grounding, Guardrails và agent workspace.

Mục tiêu là giúp agent hiểu nhanh codebase lớn, xác định đúng module và file quan trọng trước khi sửa code, đối chiếu code với nghiệp vụ/spec/test, nhưng không tạo thêm một hệ tài liệu song song khó bảo trì.

Trong [knowledge_base.md](./knowledge_base.md), tài liệu này thuộc trụ cột "Grounding, Retrieval & Traceability với `AGENTS.md`, manifest, `kb.ps1`, `rg`" và giải thích vai trò second brain/code intelligence.

## Context

Thiết kế này kế thừa các nguyên tắc từ:

1. `docs/` là KMS chính thức và source of truth cho nghiệp vụ, kiến trúc, spec, QA, runbook và methodology.
2. `llm-wiki/` là controlled workspace của agent, dùng cho observation, snapshot, contradiction, open question và lessons learned.
3. Source code là implementation hiện tại, không tự động là source of truth.
4. Header trong code chỉ là navigation hint, không phải business rule hoặc source of truth.
5. Mọi claim quan trọng phải được kiểm tra với grounding manifest, official docs, code thực tế và test phù hợp.

## Design Review Summary

Thiết kế ban đầu đúng hướng ở ba điểm:

1. Không tạo file note cho mọi source file.
2. Dùng module-level map thay vì mirror toàn bộ codebase.
3. Dùng Code Intelligence Header rất ngắn cho hotspot file.

Các điểm cần tối ưu:

1. Không cần tạo thêm `docs/02-architecture/09.module-boundary.md` và `docs/02-architecture/10.component-catalog.md`. Boundary và component catalog nên nằm trong C4 hiện có hoặc ADR/standards khi có quyết định kiến trúc thật sự.
2. Không nên bắt buộc nhiều map bền vững như `entrypoint-map.md`, `test-map.md`, `hotspot-map.md` cho mọi repo. Chúng chỉ nên được tạo khi module lớn hoặc có pain lặp lại.
3. Không nên thêm header vào code trong một task chỉ để “thiết lập intelligence layer” nếu task đó không sửa behavior. Header nên được thêm dần khi chạm vào hotspot hoặc entry point quan trọng.
4. Không nên coi `llm-wiki/module-notes/<module>.md` là artifact bắt buộc cho mọi module. Chỉ tạo cho bounded context, runtime model, pipeline, adapter, hoặc module thường xuyên bị sửa.
5. Không nên đưa policy dài vào `AGENTS.md`. `AGENTS.md` chỉ cần rule ngắn trỏ về tài liệu này.

## Architecture

Code Intelligence Agent v2 dùng 4 lớp thông tin, theo thứ tự tin cậy giảm dần:

```text
1. Official docs under docs/
   Source of truth for business, architecture, contracts, QA and methodology.

2. Grounding manifest
   Registry for which official sources should be read for a task.

3. Agent workspace under llm-wiki/
   Current observations, module notes, contradictions, open questions and lessons learned.

4. Source code hints
   Lightweight headers in important files, used only for navigation.
```

Thiết kế không tạo thêm architectural source of truth mới. Nếu thiếu boundary hoặc component detail, agent phải cập nhật C4/ADR/standards hiện có hoặc ghi contradiction/open question, không tạo `09.module-boundary.md` hay `10.component-catalog.md`.

## Minimal Artifact Set

### Required

```text
docs/
  grounding/
    manifest.yaml
    manifest.md
  methodology/
    o_4.code_intelligence_agent_v2.md

llm-wiki/
  codebase-map.md
  contradiction-log.md
  open-questions.md
  lessons-learned.md
  module-notes/
    <module>.md
```

### Optional

```text
llm-wiki/
  entrypoint-map.md
  test-map.md
  hotspot-map.md
  source-map.md
```

Optional artifacts are created only when they reduce repeated lookup cost. A small repo or narrow project can keep this information inside `codebase-map.md` and module notes.

### Not Recommended

```text
docs/02-architecture/09.module-boundary.md
docs/02-architecture/10.component-catalog.md
llm-wiki/file-notes/<file>.md
```

Reason:

1. Boundary and component knowledge belongs in C4, ADR, standards, or module notes depending on confidence and scope.
2. File notes become stale quickly and duplicate source code.
3. Extra canonical docs make SDD slower unless they represent a real source-of-truth decision.

## Artifact Roles

### `docs/02-architecture/04.architecture-c4.md`

C4 remains the official architecture map. It should describe containers, components, dependencies, external systems and architectural boundaries at the correct abstraction level.

C4 should not list every file. If C4 lacks a boundary needed for a code change, update C4 or record an open question instead of creating a parallel architecture document.

### `docs/02-architecture/06.adr.md` or `docs/02-architecture/adr/`

ADR records architectural decisions and trade-offs. Use ADR when a boundary, dependency direction, runtime model separation rule, or persistence pattern changes.

### `llm-wiki/codebase-map.md`

This is the durable workspace map for current implementation.

It should answer:

1. Which modules exist.
2. Where each module lives.
3. Which official docs each module depends on.
4. Which entry points are important.
5. Which tests or verification scripts are relevant.
6. Which module notes exist.
7. Which areas are known hotspots.

Keep this file compact. It is an index, not a generated tree dump.

### `llm-wiki/module-notes/<module>.md`

Create module notes only for meaningful modules.

A module note should include:

1. Module responsibility.
2. Source paths.
3. Official docs and grounding sources.
4. Main entry points.
5. Main code flow.
6. Relevant tests or validation scripts.
7. Known contradictions.
8. Open questions.
9. Confidence level.

Do not copy source code into module notes.

### `llm-wiki/contradiction-log.md`

Use this when code, docs, tests, AGENTS rules, or workspace notes disagree.

Each entry should include:

1. What conflicts.
2. Evidence paths.
3. Impact.
4. Proposed resolution.
5. Owner or follow-up.

### `llm-wiki/open-questions.md`

Use this when grounding is insufficient but work can continue with explicit assumptions, or when the task must pause for human clarification.

### `llm-wiki/lessons-learned.md`

Use this for stable operational lessons after debugging or verification, not for transient task notes.

## Code Intelligence Header

Code Intelligence Header is a short comment in important source files.

It is allowed only when the file is one of:

1. application entry point.
2. Orchestration service.
3. Runtime model factory, registry, adapter, renderer, or binding boundary.
4. Data access adapter.
5. Export/rendering boundary.
6. Validation or verification harness.
7. Known hotspot.

Do not add headers to every DTO, enum, mapper, simple utility, static config file, generated file, or single-purpose helper.

### Required Fields

```text
Code Intelligence Header
Role:
Module:
Model or bounded context:
Main flow:
Official docs:
Workspace note:
Change risk:
```

### Forbidden Header Content

1. Business rules that should live in specs.
2. SQL text or credentials.
3. Long implementation explanation.
4. Assumptions not recorded in `llm-wiki/open-questions.md`.
5. Status claims such as “fully verified” without evidence.

### Python Example

```python
# Code Intelligence Header
# Role: runtime dispatch entry point
# Module: application runtime
# Model or bounded context: all runtime modes
# Main flow: parse config -> resolve runtime mode -> run selected workflow
# Official docs: docs/04-spec-test/application-spec.md, docs/09-runbook/
# Workspace note: llm-wiki/module-notes/application-runtime.md
# Change risk: high, affects all runtime modes
```

## File Role Snapshot

Before editing an important source file, agent should create a short snapshot in the task context. This is not persisted unless the file becomes a recurring hotspot.

Template:

```text
File Role Snapshot
File:
Current role:
Module:
Official sources read:
Workspace notes read:
Callers:
Callees:
Tests or validation:
Risk:
Planned change:
```

This snapshot prevents blind edits without creating permanent file-level documentation.

## Main Code Flow

Main Code Flow belongs in module notes, not in source headers.

Recommended format:

```text
Flow: <business or runtime flow name>
Official source:
Entry point:
Steps:
  1. <component/function>
  2. <component/function>
  3. <component/function>
Validation:
Known gaps:
```

Use it to check whether implementation still follows business flow, application contract, reporting contract, or verification expectations.

## Agent Workflow

### When Creating the Layer

1. Read `AGENTS.md`.
2. Read `docs/grounding/manifest.yaml` and `docs/grounding/manifest.md`.
3. Read `docs/methodology/methodology.md`, `docs/methodology/o_2.kms_intergrate.md`, and `docs/methodology/o_3.kms_obsidian_rg_runbook.md`.
4. Identify existing official docs for architecture, specs, QA and runbook.
5. Create or update `llm-wiki/codebase-map.md` only if it does not already exist or is materially stale.
6. Create module notes only for main bounded contexts or high-change modules.
7. Record missing evidence in `llm-wiki/open-questions.md`.
8. Record contradictions in `llm-wiki/contradiction-log.md`.
9. Add Code Intelligence Headers only to selected important files, preferably when touching those files for real work.

### Before Editing Code

1. Resolve grounding sources from manifest and task-specific docs.
2. Read official docs first.
3. Read relevant `llm-wiki/` notes as workspace observations.
4. Search code with `rg`.
5. Read target file and direct callers/callees.
6. Create File Role Snapshot in the task context.
7. Make the smallest grounded change.
8. Run relevant validation from canonical spec/runbook/QA sources.
9. Update workspace notes only when durable knowledge changed.

## Recommended `rg` Queries

```powershell
rg -n --heading --smart-case "def main|if __name__|argparse|click|typer|FastAPI|router|Controller" src
rg -n --heading --smart-case "register|factory|provider|pipeline|orchestr|dispatch|mode" src
rg -n --heading --smart-case "DAO|Repository|Adapter|query|execute|read_sql|oracledb" src
rg -n --heading --smart-case "pytest|unittest|verify|assert|feature|Scenario" tests verify tools
rg -n --heading --smart-case "<businessKeyword>|<moduleName>|<flowName>" docs llm-wiki src tests
```

## AGENTS.md Policy Snippet

Use this shortened policy if `AGENTS.md` needs to reference Code Intelligence:

```markdown
## Code Intelligence Policy

- `docs/` remains the official KMS and source of truth.
- `llm-wiki/` is a controlled agent workspace, not a source of truth.
- For important code changes, read the relevant official docs, workspace module note, target file and direct callers/callees before editing.
- Use a File Role Snapshot in the task context before editing important files.
- Add Code Intelligence Header only to important entry point, orchestration, boundary, adapter, verification or hotspot files.
- Keep headers short and treat them as navigation hints only.
- Record doc/code/test conflicts in `llm-wiki/contradiction-log.md`.
- Record insufficient grounding in `llm-wiki/open-questions.md`.
```

## Definition of Done

A Code Intelligence rollout is good enough when:

1. `docs/` remains the only official KMS.
2. No extra architecture documents are introduced unless they represent a real source-of-truth decision.
3. `llm-wiki/codebase-map.md` identifies main modules and links to official sources.
4. Module notes exist only for meaningful or high-change modules.
5. Important flows are captured at module level.
6. Headers exist only in selected important source files.
7. Headers are short and contain no business rules.
8. Contradictions and open questions are recorded.
9. Validation commands are resolved from canonical docs before execution.
10. The rollout does not change runtime behavior unless a separate implementation task requires it.

## Anti Patterns

Avoid:

1. Creating file notes for every source file.
2. Copying source code into `llm-wiki/`.
3. Creating `09.module-boundary.md` or `10.component-catalog.md` as parallel architecture docs.
4. Treating header comments as source of truth.
5. Adding headers to every file.
6. Updating `docs/` from observations without official evidence.
7. Running validation from stale examples instead of canonical spec/runbook/QA sources.
8. Using `llm-wiki/` as the only evidence for a code change.

## Rollout Plan

### Phase 1. Baseline

Create or refresh `llm-wiki/codebase-map.md` with only main modules, paths, official docs, entry points, tests and known hotspots.

### Phase 2. Module Notes

Create module notes for the highest-value modules first. Do not require full coverage.

### Phase 3. Hotspot Headers

Add Code Intelligence Headers only when editing selected important files or when a repeated defect proves the file is a hotspot.

### Phase 4. Sync Loop

After debugging or verification, update contradiction, open question or lessons-learned notes when there is durable knowledge.

### Phase 5. Governance

Keep this document as the methodology reference. Add only the short policy snippet to `AGENTS.md` when needed.

## Review Notes

This optimized v2 deliberately removes the two proposed architecture documents:

1. `docs/02-architecture/09.module-boundary.md`
2. `docs/02-architecture/10.component-catalog.md`

The same information should be handled through existing C4, ADR, standards, `llm-wiki/codebase-map.md`, and module notes depending on whether the information is official design intent or workspace observation.
