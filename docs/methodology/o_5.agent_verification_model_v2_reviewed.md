# Mô Hình Thẩm Định Kết Quả Agent v2

## Mục Đích

Tài liệu này định nghĩa mô hình thẩm định kết quả làm việc của agent sau khi agent thực hiện một task. Mục tiêu là nghiệm thu bằng bằng chứng có thể kiểm tra lại, không nghiệm thu bằng lời khẳng định của agent.

Mô hình này được viết ở mức tổng quát để áp dụng cho nhiều dự án dùng Spec Driven Development, KMS, Code Intelligence và DoD. Nó không thay thế grounding manifest, QA matrix, runbook, DoD, hoặc các artifact chính thức của từng dự án.

Trong [knowledge_base.md](./knowledge_base.md), tài liệu này thuộc trụ cột "Evidence-Based Verification".

## Bối Cảnh

Nguyên tắc nền:

1. KMS chính thức của dự án là source of truth.
2. Executable specifications, acceptance tests, hoặc scenario tests là evidence khi có.
3. Agent workspace chỉ là nơi ghi observation, assumption, contradiction và lesson learned; không phải source of truth.
4. Source code là implementation hiện tại, không tự động là policy.
5. Lệnh verification phải lấy từ nguồn canonical của dự án: contract/spec, runbook, QA matrix, DoD, CI workflow, hoặc tài liệu task-specific.
6. Không được claim `VERIFIED` nếu bằng chứng còn thiếu hoặc warning/error chưa được đọc và phân loại.

## Đánh Giá Tương Thích

### Kết Luận

Thiết kế Agent Verification Model v2 **không vi phạm** `methodology.md`, `o_2`, `o_3`, hoặc `o_4` nếu áp dụng theo bản tối ưu này.

### Vấn Đề Trong Bản Trước Và Cách Tối Ưu

| Vấn đề | Rủi ro | Cách tối ưu |
|---|---|---|
| Bắt buộc BPMN và Gherkin cho mọi code change | Lệch với repo hiện tại vì không phải task nào cũng là business-flow change | Chỉ bắt buộc khi task thay đổi business behavior hoặc acceptance flow |
| Unit test được mô tả là bắt buộc tuyệt đối cho mọi code change | Lệch với DoD vì DoD thường yêu cầu verification nhẹ nhất có ý nghĩa theo scope | Dùng targeted verification phù hợp với risk và QA matrix của dự án |
| Direct application test là điều kiện `VERIFIED` tuyệt đối | Quá cứng cho documentation-only, comment-only, config-only, hoặc môi trường thiếu | Bắt buộc cho user-facing/runtime behavior khi environment có; nếu không, status tối đa phụ thuộc bằng chứng còn lại |
| Agent workspace được gọi không nhất quán | Lệch với KMS design của từng dự án | Dùng tên workspace canonical của dự án |
| Gói bằng chứng quá dài cho mọi task | Tạo ceremony quá nặng, trái tinh thần lightweight KMS/Code Intelligence | Dùng gói bằng chứng theo task type |
| Policy block cho `AGENTS.md` quá dài | Lệch với `o_4`, vốn khuyến nghị policy ngắn trỏ về methodology | Đưa vào snippet ngắn |

## Triết Lý Verification

```text
Không có bằng chứng thì không gọi là verified.
Bằng chứng phải tương xứng với risk và task type.
Source of truth chính thức thắng workspace observation.
Warning/error phải được đọc, không chỉ chạy command.
```

## Loại Task

| Loại task | Kỳ vọng verification |
|---|---|
| Chỉ sửa tài liệu | Kiểm tra cấu trúc Markdown, link, traceability tới source of truth, ý nghĩa gốc, và cập nhật manifest/catalog khi cần |
| Chỉ sửa comment/header | Kiểm tra syntax/compile cho code đã chạm; không cần report generation nếu runtime behavior không đổi |
| Sửa code hẹp | Targeted test hoặc compile/import check, kèm review log/error liên quan |
| Đổi business behavior | Official business source, acceptance scenario hoặc test mapping, targeted tests, runtime/debug evidence |
| Đổi interface hoặc artifact user-facing | Lệnh canonical từ contract/spec/runbook/QA matrix, cộng với review output/log/artifact |
| Đổi data-contract hoặc persistence/integration boundary | Data/contract/provider verification, caveat trong QA matrix, review path/schema/message/API shape |
| Đổi shared architecture/runtime | Regression check rộng hơn qua các module bị ảnh hưởng và cập nhật official docs khi behavior đổi |

## Chuỗi Verification

Chỉ dùng chuỗi đầy đủ khi task thật sự cần:

```text
User Request
  -> Grounding Sources
  -> Official Artifact hoặc Acceptance Source
  -> Code/Document Diff
  -> Targeted Verification
  -> Log/Output Inspection
  -> Knowledge Sync
  -> Final Verdict
```

Với thay đổi behavior hướng nghiệp vụ, mở rộng chuỗi thành:

```text
Business Source hoặc BPMN
  -> Gherkin / acceptance scenario / test case
  -> Unit hoặc integration test
  -> Runtime hoặc direct application check
  -> Gói bằng chứng
```

Nếu BPMN hoặc Gherkin không tồn tại, agent không được tự bịa. Agent phải báo `insufficient grounding`, dùng các nguồn official business/spec/test hiện có, và phân loại artifact thiếu là một gap.

## Các Cổng Thẩm Định

### Gate 1. Scope

Kiểm tra:

1. User request và kết quả mong muốn.
2. File đã thay đổi.
3. Module bị ảnh hưởng.
4. Edit ngoài scope.
5. Behavior, contract, schema, hoặc documentation có đổi không.

Fail hoặc hạ cấp verdict khi diff mở rộng scope mà không có bằng chứng.

### Gate 2. Grounding

Kiểm tra:

1. `AGENTS.md`.
2. Grounding manifest hoặc source registry của dự án.
3. Tài liệu task-specific như API/application/UI spec, C4, ADR, standards, QA matrix, runbook, DoD, hoặc handover.
4. Module notes trong agent workspace chỉ như working context.
5. Source code và tests liên quan tới thay đổi.

Nếu official sources mâu thuẫn với workspace notes, official sources thắng và conflict phải được ghi vào contradiction log của workspace.

### Gate 3. Mapping Spec Và Acceptance

Chỉ bắt buộc với thay đổi behavior, nghiệp vụ, contract, hoặc user-visible workflow.

Nguồn mapping hợp lệ:

1. BPMN hoặc business-flow doc.
2. Requirement hoặc handover doc.
3. API spec, application spec, UI spec, event contract, data contract, hoặc report contract.
4. Gherkin feature hoặc acceptance scenario.
5. Unit/integration/verification script đã tồn tại.

Mapping status:

1. `MAPPED`
2. `PARTIALLY_MAPPED`
3. `NOT_MAPPED`
4. `NOT_APPLICABLE`
5. `NEEDS_CLARIFICATION`

### Gate 4. Chất Lượng Code Và Kiến Trúc

Kiểm tra:

1. Layer responsibility được giữ đúng theo kiến trúc dự án.
2. Bounded context, module boundary, hoặc model boundary của dự án vẫn được giữ đúng.
3. Query, schema, migration, workflow definition, hoặc contract file nằm đúng nơi được standards/architecture quy định.
4. Không có operation nguy hiểm ngoài scope như DML/DDL/migration/destructive command nếu task không yêu cầu và chưa được approve.
5. Không có secrets hoặc PII trong code/logs/docs.
6. Không refactor ngoài scope.
7. Error handling rõ ràng.
8. Runtime behavior chỉ đổi khi đó là chủ ý của task.

### Gate 5. Bằng Chứng Test Và Automation

Chọn verification nhẹ nhất nhưng có ý nghĩa từ nguồn canonical.

Trường evidence:

```text
Lệnh:
Nguồn xác nhận lệnh:
Exit code:
Phạm vi:
Kết quả:
Warnings/Errors:
```

Quy tắc:

1. Relevant test fail thì verdict là `FAILED` hoặc `PARTIALLY VERIFIED` kèm caveat rõ.
2. Nếu một QA lane đã được QA matrix hoặc CI status ghi là partial/failing, không được gọi là full pass chỉ vì command chạy xong.
3. Nếu automated test không thực tế, dùng compile/import check, targeted script, manual inspection, hoặc ghi rõ limitation.

### Gate 6. Runtime, Debug Và Log Inspection

Bắt buộc với runtime behavior change và validation cho artifact/user-facing output.

Kiểm tra:

1. Lệnh đã chạy.
2. Nguồn xác nhận lệnh.
3. Exit code.
4. Output artifact path nếu có.
5. Output artifact metadata nếu có, ví dụ path, size, count, record count, page count, section count, response code, hoặc checksum tùy dự án.
6. Warnings và errors.
7. Warning là expected, accepted, hay unresolved.

Với runtime/report/artifact generation, bật debug/verbose logging khi canonical runbook yêu cầu. Nếu bỏ qua, phải nêu nguồn canonical hoặc lý do hợp lệ.

### Gate 7. Direct Application Hoặc Artifact Verification

Bắt buộc khi task đổi user-visible runtime behavior và environment sẵn có.

Ví dụ:

1. application run.
2. API call.
3. UI flow.
4. Batch/job execution.
5. Message/event consumer execution.
6. Report/file/artifact generation và inspection.
7. Data/integration output inspection.

Nếu không chạy được, phải nêu lý do và hạ cấp thành `PARTIALLY VERIFIED`, `BLOCKED`, hoặc `NEEDS HUMAN REVIEW` tùy risk.

### Gate 8. Knowledge Sync

Kiểm tra:

1. Agent workspace đã được cập nhật cho durable observations nếu cần.
2. Contradiction log đã được cập nhật cho conflict nếu cần.
3. Open questions đã được cập nhật cho unresolved grounding nếu cần.
4. Official docs chỉ được cập nhật khi source-of-truth behavior đổi.
5. Grounding manifest, source catalog, hoặc documentation index được cập nhật khi official artifact được thêm, xóa, đổi tên, hoặc promote.

## Trạng Thái Cuối

Chỉ dùng một trong các status sau:

| Status | Ý nghĩa |
|---|---|
| `VERIFIED` | Bằng chứng đủ cho task type và risk; command/log/artifact đã được kiểm tra. |
| `PARTIALLY VERIFIED` | Có bằng chứng hữu ích, nhưng một gate liên quan còn thiếu, partial, hoặc bị giới hạn bởi environment. |
| `FAILED` | Relevant test/runtime/log/artifact evidence fail, hoặc diff vi phạm official contract. |
| `BLOCKED` | Thiếu source, decision, test data, dependency, hoặc environment bắt buộc và không có fallback an toàn. |
| `NEEDS HUMAN REVIEW` | Cần human judgment cho business, architecture, security, compliance, hoặc risk decision. |

## Checklist Nghiệm Thu Cho Human

Reviewer nên xác nhận:

1. Scope đúng.
2. Grounding sources được liệt kê.
3. Evidence khớp với task type.
4. Command lấy từ canonical docs khi applicable.
5. Exit code và warnings/errors được báo cáo trung thực.
6. Output artifacts đã được kiểm tra khi user-visible.
7. Docs/manifest chỉ được cập nhật khi cần.
8. Agent workspace không bị dùng như source of truth.
9. Final status đi theo evidence.

## Prompt Cho Agent Thẩm Định Độc Lập

```text
Bạn là Independent QA Verification Agent v2.

Nhiệm vụ:
Thẩm định kết quả của Implementation Agent dựa trên evidence có thể kiểm tra lại.

Không tin vào kết luận trước đó.
Không sửa code trừ khi user yêu cầu.
Không dùng agent workspace làm source of truth.

Hãy kiểm tra:
1. Task type và scope.
2. Grounding sources.
3. Diff và affected modules.
4. Acceptance/spec mapping nếu task đổi behavior.
5. Test hoặc verification command và source của command.
6. Exit code, warnings, errors, logs.
7. Output artifacts nếu có.
8. Knowledge sync vào docs, manifest, hoặc agent workspace nếu cần.
9. Final status có đúng với evidence không.

Output:
Verification verdict:
Scope findings:
Grounding findings:
Evidence review:
Log/warning review:
Artifact review:
Knowledge sync review:
Remaining risks:
Required fixes:
```

## Đoạn Policy Ngắn Cho AGENTS.md

Dùng policy ngắn này nếu `AGENTS.md` cần tham chiếu verification:

```markdown
## Agent Verification Policy

- Verify bằng evidence, không bằng lời khẳng định của agent.
- Chọn verification theo task type và risk.
- Dùng nguồn command canonical: contract/spec, runbook, QA matrix, DoD, CI workflow, hoặc task-specific docs.
- Với runtime/report/artifact generation validation, bật debug/verbose logging khi canonical runbook yêu cầu.
- Kiểm tra exit code, warnings/errors, logs, và output artifacts khi applicable.
- Chỉ dùng agent workspace như workspace context.
- Dùng `VERIFIED`, `PARTIALLY VERIFIED`, `FAILED`, `BLOCKED`, hoặc `NEEDS HUMAN REVIEW`.
```

## Kết Luận

Agent Verification Model v2 tương thích với methodology của nhiều dự án khi nó vẫn evidence-based, scale theo task, và grounded trong KMS hiện có của từng dự án.

Quy tắc cốt lõi:

```text
Đừng bắt agent phải chạy một loại kiểm thử mà dự án chưa có hoặc chưa công nhận là gate chính thức. Nhưng cũng đừng claim `VERIFIED` nếu chưa chạy những kiểm tra chính thức phù hợp mà dự án đã có.
```
