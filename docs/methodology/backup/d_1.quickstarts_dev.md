# README Dev Quickstart

## Context

This file was normalized as part of the repository-wide Obsidian-compatible technical wiki migration. The original content below is preserved unless explicitly noted.

## Purpose

This document preserves the existing content of `docs/README-dev-quickstart.md` and connects it to the repository documentation graph for Agentic Spec First work.

### Quickstart chi tiết SDLC mode — Dev theo Spec-First & Agentic Development

Ghi chú chung
- Đường dẫn cố định: Spec (`specs/**`), Grounding (`docs/grounding/**`), Evals (`evals/**`), Quy tắc IDE (`.cursorrules`).
- Mọi câu trả lời/đề xuất của agent phải có citations (≥2) khớp `docs/grounding/manifest.yaml`.

1) Phân tích yêu cầu (Discovery & Requirements)
- Bước này làm gì? Thu thập bối cảnh, mục tiêu, phạm vi, ràng buộc; xác định use case/module.
- Input (phải có trước): đầu bài kinh doanh, stakeholder, dữ liệu sẵn có.
- Output tạo ra: `docs/01-requirements/requirements.md` (tài liệu yêu cầu dự án), `docs/01-requirements/01.use-cases.md`, acceptance criteria sơ bộ.
- Vai trò Agent: hỗ trợ chuẩn hóa yêu cầu thành use case/AC; đề xuất thông tin còn thiếu; gợi ý tiêu chí đo lường.

- Tài liệu phải tạo (bắt buộc):
  - `docs/01-requirements/requirements.md`
  - `docs/01-requirements/use-cases.md`
  - (nếu có) `docs/03-business-analysis/02.glossary.md`

Prompt ví dụ
```
Bạn là BA assistant. Chuẩn hóa yêu cầu thành use case + acceptance criteria.
- Tham chiếu: docs/01-requirements/requirements.md (nếu trống, hãy đề xuất khung điền)
- Đầu ra: bảng Use Case + AC theo GIVEN/WHEN/THEN, liệt kê giả định/chưa rõ cần làm rõ.
```

2) Thiết kế cấp cao & ràng buộc (High-level Design & Policies)
- Bước này làm gì? Chọn kiến trúc/biến thể, chuẩn hóa ràng buộc an toàn, dữ liệu, tuân thủ.
- Input: tài liệu yêu cầu bước 1; tham chiếu `docs/00-governance/07|13|14` (nếu có sẵn từ khung tri thức, hoặc tạo trong bước này nếu chưa có).
- Output: 
  - `docs/02-architecture/03.vision.md` (tầm nhìn hệ thống, mục tiêu, NFR — kim chỉ nam cho kiến trúc)
  - `docs/02-architecture/architecture-design.md` (sơ đồ luồng, boundary, ràng buộc)
  - `docs/02-architecture/adr.md`, `docs/02-architecture/standards.md`
  - (nếu chưa có) `docs/00-governance/00.data-governance.md`, `00.governance-rules.md`, `00.patterns-anti-patterns.md` -> làm trước khi tạo tài liệu thiết kế.
- Vai trò Agent: gợi ý pattern/anti-pattern, sinh skeleton kiến trúc và checklist rủi ro.

- Tài liệu phải tạo:
  - `docs/02-architecture/03.vision.md` (tầm nhìn hệ thống — tham chiếu checklist trong file này)
  - `docs/02-architecture/architecture-design.md`, `docs/02-architecture/adr.md`, `docs/02-architecture/standards.md`
  - (nếu chưa có) `docs/00-governance/00.data-governance.md`, `00.governance-rules.md`, `00.patterns-anti-patterns.md`

Prompt ví dụ
```
Bạn là kiến trúc sư. Đề xuất kiến trúc cao cấp (C4 level 2) cho module X.
- Ràng buộc: xem docs/00-governance/00.data-governance.md, 00.governance-rules.md, 00.patterns-anti-patterns.md
- Đầu ra: dàn ý `architecture-design.md` + 1 ADR về công nghệ chính.
```

3) Spec-First (chuẩn trước, mã sau)
- Bước này làm gì? Đặc tả hợp đồng API/sự kiện/tool I/O để mọi bên bám theo.
- Input: yêu cầu + thiết kế; tiêu chuẩn trong `00.governance-rules.md`.
- Output tạo ra:
  - `specs/10.openapi.yaml` (hoặc `specs/<module>/10.openapi.yaml`).
  - `specs/schemas/tools/<tool>.schema.json` (nếu có tool cho agent).
  - Kiểm tự động cục bộ: spectral/ajv (CI sẽ kiểm lại).
- Vai trò Agent: sinh đoạn đặc tả từ mẫu, phát hiện thiếu error model/pagination/versioning, gợi ý ví dụ payload.

- Tài liệu/artefact phải tạo:
  - `specs/10.openapi.yaml` (hoặc `specs/<module>/10.openapi.yaml`), `specs/schemas/tools/<tool>.schema.json`

- Hướng dẫn per-module (Agentic + Spec-First):
  - Input: paths theo tag/module trong `specs/10.openapi.yaml`; schema ở `components.schemas`.
  - Làm gì: trích (slice) phần module; rà soát thiếu error model/pagination/versioning trước khi code.
  - Output: danh sách endpoint/sự kiện module; mapping schema.
  - Vai trò Agent: đề xuất slice, nêu thiếu sót, sinh ví dụ payload.

Prompt ví dụ
```
Bạn là API designer. Hoàn thiện OpenAPI cho tag <module> trong specs/10.openapi.yaml.
- Bổ sung: error model thống nhất, pagination, examples.
- Đảm bảo hợp lệ spectral; nêu chỗ còn thiếu nếu có.
```

3b) Khởi tạo khung cấu trúc project/code (Project/Code Scaffolding)
- Bước này làm gì? Tạo bộ khung thư mục và tệp chuẩn để triển khai nhanh, đồng nhất giữa các dự án. Đảm bảo cấu trúc code phù hợp với team size, complexity, deployment frequency, và các constraints của dự án.
- Input: thiết kế cấp cao (bước 2), đặc tả bước 3, quy ước trong `docs/00-governance/00.governance-rules.md`.
- Output tạo ra (khuyến nghị tối thiểu — có thể thêm theo domain):
  - `src/`:
    - `src/<module>/` (mỗi module nghiệp vụ một thư mục)
    - `src/<module>/api/` (controller/route theo OpenAPI)
    - `src/<module>/services/` (logic nghiệp vụ)
    - `src/<module>/repositories/` (truy cập dữ liệu)
    - `src/shared/` (tiện ích chung, lỗi chuẩn hóa, middleware)
  - `tests/` hoặc `__tests__/`:
    - `tests/<module>/` (unit/integration theo spec)
  - `configs/` (cấu hình môi trường, mapping schema → runtime)
  - `sql/` hoặc `migrations/` (nếu dùng DB)
  - `templates/` (email/pdf/exports nếu có)
  - `specs/` (đã có): `10.openapi.yaml`, `schemas/tools/*.schema.json`
  - `docs/grounding/` (đã có): `manifest.yaml`, `prompts/`, `sources.md`
  - `evals/` (đã có): `config.yaml`, `cases/*`
  - `.cursorrules`, `.github/workflows/ci.yml`, `scripts/run_evals.*`
- Vai trò Agent: sinh skeleton thư mục/tệp theo module; tạo sẵn file khởi đầu (controller/service/test) bám `specs/10.openapi.yaml`; chèn mẫu xử lý lỗi chuẩn.
- **Hướng dẫn chi tiết**: Xem `docs/02-architecture/07.project-structure.md` để biết assessment checklist và quy trình thiết kế cấu trúc project (Phase 1-5).

Prompt ví dụ
```
Bạn là scaffolding bot. Sinh khung code cho module <module> theo specs/10.openapi.yaml và assessment checklist.
- Tham chiếu: docs/02-architecture/07.project-structure.md (assessment checklist và structure design)
- Tạo: src/<module>/{api,services,repositories}/ và tests/<module>/
- Yêu cầu code: bám spec, có TODO rõ cho phần cần dữ liệu/infra.
- Mapping: src/<module>/api/ ↔ specs/10.openapi.yaml paths
- Mapping: tests/<module>/ ↔ acceptance criteria trong docs/01-requirements/01.use-cases.md
```

4) Grounding & Prompts (Nguồn sự thật và hướng dẫn tác vụ)
- Bước này làm gì? Khai báo nguồn tri thức và prompts để agent bám đúng nguồn.
- Input: 07/13/14; đặc tả ở bước 3.
- Output tạo ra:
  - `docs/grounding/manifest.yaml`, `docs/grounding/sources.md` (khai báo nguồn, owner, độ tin cậy).
  - `docs/grounding/prompts/` (template system/tool prompts, cấm hardcode trong code).
- Vai trò Agent: sử dụng prompt + manifest để trả lời có citations; đề xuất nguồn còn thiếu; giữ đúng phong cách/trật tự trả lời.

- Tài liệu phải tạo/cập nhật:
  - `docs/grounding/manifest.yaml`, `docs/grounding/sources.md`, `docs/grounding/prompts/<module>.md`

Prompt ví dụ
```
Bạn là knowledge curator. Cập nhật docs/grounding/manifest.yaml cho module <module>.
- Thêm source_id, owner, trust, license; tạo prompts/<module>.md yêu cầu trả lời có ≥2 citations.
```

5) Mock & Contract Testing
- Bước này làm gì? Tạo mock từ spec và kiểm hợp đồng để tránh lệch implement.
- Input: `specs/10.openapi.yaml`, tool schemas.
- Output tạo ra: mock server/contract test đã chạy qua (Prism/Dredd/Schemathesis), ghi chú chênh lệch để sửa spec.
- Vai trò Agent: sinh kịch bản Dredd/Schemathesis mẫu; rà soát rule vi phạm; gợi ý cải tiến spec.

- Artefact/ghi chú:
  - Mock server/script (Prism/Dredd/Schemathesis); báo cáo lệch để sửa spec.

Prompt ví dụ
```
Bạn là contract tester. Tạo kịch bản Dredd/Schemathesis cho tag <module> từ specs/10.openapi.yaml.
- Mục tiêu: phát hiện lệch status code, schema, error model; in báo cáo lệch.
```

6) Implement (Coding theo hợp đồng)
- Bước này làm gì? Cài đặt tính năng tối thiểu bám spec và mock.
- Input: spec đã thông; mock/contract test qua; prompts/manifest ổn định.
- Output tạo ra: mã nguồn tính năng; tối thiểu unit/integration test theo spec; cập nhật `specs/` nếu có thay đổi hợp đồng.
- Vai trò Agent: sinh skeleton code theo spec; gợi ý test khung; cảnh báo chệch spec; đề xuất logging/observability.

- Tài liệu/artefact:
  - `src/<module>/{api,services,repositories}/`, `tests/<module>/*`; nếu có DB: `sql/` hoặc `migrations/`.

Prompt ví dụ
```
Bạn là developer. Triển khai endpoint <path> theo OpenAPI (method <verb>). Bám status code & schema.
- Sinh unit/integration test tối thiểu; nêu phần cần mock/stub.
```

6b) Hoàn thiện cấu trúc code theo module (Refine per Module)
- Bước này làm gì? Rà soát naming, tách lớp, chống lặp; đảm bảo mapping từ spec → route/controller/service/repo đầy đủ.
- Input: spec ổn định; skeleton từ bước 3b; rule/lint trong `00.governance-rules.md`.
- Output tạo ra: mã đã refactor nhẹ theo chuẩn (layering, error model, pagination); test tối thiểu cho từng API/flow.
- Vai trò Agent: đề xuất refactor nhỏ an toàn; kiểm tra mapping đủ/thiếu theo spec; tự động thêm case test khung nếu trống.

Prompt ví dụ
```
Bạn là reviewer. Rà soát module <module> về layering/naming/DRY.
- Đề xuất refactor nhỏ (an toàn), liệt kê mapping route↔service↔repo còn thiếu.
```

7) Evals (Đánh giá tự động — Guardrails cho Agent)
- Bước này làm gì? Chấm chất lượng grounded answers và độ liên quan/bao phủ.
- Input: prompts + manifest/sources; spec ổn; chức năng cơ bản đã chạy.
- Output tạo ra:
  - `evals/cases/*.jsonl` (golden/adversarial/regression) cập nhật theo module.
  - Chạy `scripts/run_evals.sh|ps1`; ngưỡng theo `evals/config.yaml`.
- Vai trò Agent: đề xuất thêm case; phân tích “fail reason” (thiếu nguồn, prompt mơ hồ, spec chưa rõ); gợi ý chỉnh.

- Tài liệu/artefact:
  - `evals/config.yaml`, `evals/cases/<module>/*.jsonl` (golden/adversarial/regression) có `source_id` hợp lệ; log điểm.

Prompt ví dụ
```
Bạn là eval builder. Tạo 2 case golden + 1 case adversarial cho module <module>.
- Case phải kèm expected_citations (source_id theo manifest.yaml); đặt ngưỡng theo evals/config.yaml.
```

8) PR & CI (Kiểm tự động trước hợp nhất)
- Bước này làm gì? Tích hợp kiểm tra tự động để chặn PR không đạt ngưỡng.
- Input: thay đổi trong `specs/**`, `docs/grounding/**`, `evals/**`, `src/**`.
- Output tạo ra: pipeline pass (lint/spec/evals). Nếu fail → log nguyên nhân/điểm số.
- Vai trò Agent: đọc log CI, đề xuất chỉnh bộ phận đúng chỗ (spec/prompt/manifest/evals/code) và sinh diff gợi ý.

- Cấu phần liên quan:
  - `.github/workflows/ci.yml`, `.cursorrules`

Prompt ví dụ
```
Bạn là CI fixer. Đọc log CI và gợi ý chỉnh sửa tối thiểu để pass (lint/spec/evals).
- Phân loại lỗi: spec vs code vs prompts vs manifest vs evals; xuất diff đề xuất.
```

9) Bàn giao & vận hành (Handover)
- Bước này làm gì? Chuẩn hóa tài liệu để chuyển giao/duy trì.
- Input: toàn bộ hiện trạng sau bước 8.
- Output tạo ra: `docs/development/DEPLOY.md`, `docs/development/RUNBOOK.md`, tài liệu bàn giao/kiểm thử, link chạy CI thành công.
- Vai trò Agent: tổng hợp tài liệu, tạo checklist vận hành, sinh hướng dẫn nhanh cho người tiếp nhận.

- Tài liệu phải có:
  - `docs/09-runbook/DEPLOY.md`, `docs/09-runbook/RUNBOOK.md`, `src/<module>/README.md`

Prompt ví dụ
```
Bạn là technical writer. Tổng hợp DEPLOY.md và RUNBOOK.md cho module <module> từ spec + tests + code.
- Bao gồm: yêu cầu môi trường, cách chạy, log/monitor, checklist sự cố phổ biến.
```

Gợi ý khi gặp lỗi
- Sai hợp đồng/thiếu lỗi mẫu → sửa `specs/10.openapi.yaml` + chạy lại mock/contract test.
- Trả lời thiếu citations → bổ sung nguồn trong `manifest.yaml`/`sources.md` hoặc chỉnh prompt rõ ràng hơn.
- Điểm evals thấp → điều chỉnh prompt/grounding/spec; bổ sung golden/adversarial tương ứng.
