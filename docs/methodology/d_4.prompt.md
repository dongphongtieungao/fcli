# D 4.Prompt

## Context

This file was normalized as part of the repository-wide Obsidian-compatible technical wiki migration. The original content below is preserved unless explicitly noted.

## Purpose

This document preserves the existing content of `docs/methodology/d_4.prompt.md` and connects it to the repository documentation graph for Agentic Spec First work.

Hãy cập nhật AGENTS.md để bắt buộc Codex triển khai đúng docs/methodology/methodology.md và grounding workflow trước khi review, test, hoặc chạy pipeline.

Mục tiêu:
- Loại bỏ tình trạng agent dùng shortcut command trong AGENTS.md mà bỏ qua docs/grounding/manifest.md, application spec, QA matrix, runbook.
- Biến grounding + methodology thành execution gate rõ ràng, không chỉ là mô tả tham khảo.
- Đảm bảo khi test report generation, agent phải đọc nguồn test canonical trước, đặc biệt các file được manifest.md trỏ tới.

Yêu cầu thay đổi cụ thể:

1. Thêm section mới trong AGENTS.md tên “Methodology Execution Gate” hoặc tương đương.
   Nội dung bắt buộc:
   - Trước mọi code review, refactor review, test pipeline, hoặc validation claim, agent phải đọc:
     1. docs/grounding/manifest.yaml
     2. docs/grounding/manifest.md
     3. docs/methodology/methodology.md
     4. source cụ thể do manifest.md trỏ tới cho loại việc đang làm, ví dụ:
        - application/test command: docs/04-spec-test/application-spec.md
        - QA/validation: docs/06-quality-assurance/14.qa-matrix.md
        - Runbook/DoD: docs/09-runbook/18.dod.md
        - Refactoring prompt lane: docs/grounding/prompts/refactoring.md
   - Agent phải nêu trong final report các source đã đọc.

2. Sửa phần Validation Commands hiện tại.
   - Không để command ví dụ thiếu debug trở thành canonical command.
   - Ghi rõ: “Commands below are examples only; before running, resolve canonical command from application spec/runbook/QA matrix.”
   - Với report generation test, command mặc định phải dùng debug logging.
   - Trên PowerShell, ưu tiên:
     $env:PYTHONPATH='.'; python src/main.py --debug
   - Hoặc wrapper theo runbook:
     .\run.cmd --debug
   - Nếu command không có --debug, agent phải giải thích vì sao.

3. Thêm rule “No Validation Shortcut”.
   Nội dung:
   - Agent không được chạy validation chỉ vì command xuất hiện trong AGENTS.md.
   - Agent phải xác định validation command từ source-of-truth hiện hành.
   - Nếu AGENTS.md và application spec/runbook khác nhau, application spec/runbook thắng cho command runtime.

4. Thêm report contract sau mỗi test/report generation:
   Agent phải báo:
   - Command chính xác đã chạy.
   - Source nào xác nhận command đó.
   - Exit code.
   - Output artifact path nếu có.
   - Artifact-specific metadata nếu có.
   - Có warning/error trong log không.
   - Nếu warning “file not found” hoặc “0 points” xuất hiện thì không được gọi là pass đầy đủ, chỉ được gọi là “file generation passed”.

5. Thêm cross-reference tới methodology.md:
   - Phase 0/1: Grounding and spec lookup.
   - Phase 3: Debugging/refinement must use root-cause analysis, not broad claims.
   - Phase 4: Verification and sync must use canonical validation evidence.

6. Giữ diff nhỏ, không rewrite toàn bộ AGENTS.md.
   - Chỉ thêm/sửa các section liên quan methodology, grounding, validation.
   - Không thay đổi kiến trúc ba model.
   - Không thay đổi lore commit protocol.

Sau khi sửa:
- Chạy diff review trên AGENTS.md.
- Không cần chạy pipeline.
- Final report phải nêu rõ các section đã cập nhật và cách thay đổi này ngăn lỗi “chạy command thiếu --debug” tái diễn.

## Hướng Dẫn Tạo `prompts.md` Và Prompt Module

Nên tạo prompt theo 2 tầng: tạo `docs/grounding/prompts.md` sớm như lớp governance/routing, nhưng chỉ tạo prompt module cụ thể khi đã có pattern lặp lại, rủi ro thật, hoặc module đủ ổn định. Không tạo prompt quá sớm và không tạo tràn lan.

### 1. Tạo `docs/grounding/prompts.md` Khi Nào?

Nên tạo `docs/grounding/prompts.md` ngay khi bắt đầu thiết lập KMS/grounding cho SDD, cùng nhóm file:

```text
docs/grounding/manifest.md
docs/grounding/manifest.yaml
docs/grounding/sources.md
docs/grounding/prompts.md
```

Lý do: `prompts.md` là prompt governance và prompt routing, không phải prompt mẫu chung. Mục đích của nó là giải thích prompt nào dùng cho lane nào, ràng buộc prompt vào grounding pack và `AGENTS.md`, giúp agent đi đúng flow:

```text
grounding -> spec -> code -> verify -> sync docs
```

Tức là `prompts.md` nên có từ rất sớm, khi dự án bắt đầu có nhu cầu dùng agent một cách có kiểm soát.

Thời điểm phù hợp để tạo `prompts.md`:

| Điều kiện | Vì sao cần `prompts.md` |
|---|---|
| Bắt đầu chuẩn hóa SDD/KMS | Cần nơi định nghĩa prompt lane và grounding expectation |
| Đã có `AGENTS.md` | Cần map từ policy tổng sang prompt routing cụ thể |
| Đã có `manifest.md`, `manifest.yaml`, `sources.md` | Cần nối prompt với source registry |
| Bắt đầu giao task cho Codex/Claude/Gemini coding agent | Cần agent biết task thuộc analysis, planning, implementation, review, testing hay KMS lane |
| Có nhiều loại task khác nhau | Cần tránh một prompt dùng cho mọi việc |
| Có nhiều module hoặc nhiều bounded context | Cần routing để agent không đọc sai nguồn và sửa sai boundary |

Trong runbook, `docs/grounding/prompts.md` được tạo trong bước tạo grounding files, cùng với manifest và sources. Nội dung mẫu định nghĩa prompt lanes, required grounding và các response sections bắt buộc cho implementation task.

### 2. Tạo Prompt Module Cụ Thể Khi Nào?

Các file như:

```text
docs/grounding/prompts/prompt-application-architect.md
docs/grounding/prompts/prompt-application-coder.md
docs/grounding/prompts/application-runtime.md
docs/grounding/prompts/application-logging.md
docs/grounding/prompts/application-validation.md
```

không nhất thiết tạo ngay từ ngày đầu. Chỉ nên tạo khi có lane hoặc module lặp lại đủ nhiều, đủ rủi ro, hoặc đủ đặc thù.

Nên tạo prompt module khi có 1 trong 7 tín hiệu sau:

| Tín hiệu | Nên tạo prompt gì |
|---|---|
| Một loại task lặp lại nhiều lần | Prompt lane/module riêng |
| Một module có rủi ro cao | Prompt module riêng với constraints rõ |
| Agent thường hiểu sai cùng một khu vực | Prompt để khóa boundary và nguồn đọc |
| Task cần nhiều bước cố định | Prompt workflow riêng |
| Task có yêu cầu verification đặc thù | Prompt validation/evidence riêng |
| Có nhiều role agent khác nhau | Prompt architect, coder, reviewer, debugger, QA |
| Có quy tắc bảo mật/vận hành quan trọng | Prompt guardrail chuyên biệt |

Ví dụ `prompt-application-coder.md` được tạo khi đã có nhu cầu sinh code, tests hoặc diff cho application pipeline. Prompt này khóa rõ intended use, constraints, references, workflow, nguyên tắc không ghi secrets, không tạo kết nối ghi, và output bắt buộc gồm `Plan`, `Changes`, `Verify`.

### 3. Thứ Tự Tạo Hợp Lý

#### Giai đoạn 0: Chưa Có SDD/KMS

Chưa cần tạo prompt module chi tiết. Chỉ cần prompt tạm thời để discovery:

```text
Hãy phân tích repo, xác định source of truth, module chính, task lane, rủi ro và artifact còn thiếu. Không sửa code.
```

Ở giai đoạn này nếu tạo nhiều prompt quá sớm, rất dễ bịa ra lane chưa tồn tại.

#### Giai đoạn 1: Bootstrap Grounding

Tạo tối thiểu:

```text
docs/grounding/manifest.md
docs/grounding/manifest.yaml
docs/grounding/sources.md
docs/grounding/prompts.md
```

`prompts.md` lúc này chỉ cần định nghĩa các lane nền:

```text
analysis
planning
implementation
review
testing
documentation
kms-maintenance
workspace-update
```

Chưa cần tạo prompt cho từng module.

#### Giai đoạn 2: Có Spec, Architecture, QA Sơ Bộ

Khi đã có Vision, Glossary, C4, NFR, ADR, Standards, BPMN hoặc contract/spec, bắt đầu tạo prompt theo vai trò agent:

```text
prompt-architect.md
prompt-coder.md
prompt-reviewer.md
prompt-debugger.md
prompt-verifier.md
prompt-kms-curator.md
```

Đây là prompt cấp orchestration hoặc role, dùng được cho nhiều module.

#### Giai đoạn 3: Có Module Quan Trọng Hoặc Dòng Công Việc Lặp Lại

Lúc này mới tạo prompt module-level:

```text
application-runtime.md
application-logging.md
application-validation.md
renderer.md
capture.md
visual-output.md
module-specific-prompt.md
```

Trong file hiện tại, `prompts.md` cũng đang phân loại rõ prompt tổng quan/orchestration, prompt module-level theo current architecture, và prompt lane chuyên biệt.

#### Giai đoạn 4: Sau Khi Vận Hành Thực Tế

Khi agent đã làm task và phát sinh lỗi lặp lại, mới bổ sung prompt để sửa hành vi agent.

| Lỗi lặp lại | Prompt nên thêm |
|---|---|
| Agent sửa code ngoài scope | `prompt-scope-control.md` hoặc bổ sung constraints vào prompt hiện có |
| Agent claim verified khi chưa chạy test | `application-validation.md` hoặc verification policy |
| Agent sửa logging làm lộ thông tin | `application-logging.md` |
| Agent refactor quá tay | `prompt-refactoring.md` |
| Agent không update KMS | `prompt-grounding-update.md` hoặc `prompt-kms-curator.md` |

### 4. Khi Nào Không Nên Tạo Prompt Mới?

Không nên tạo prompt mới nếu:

1. Task chỉ xảy ra một lần, không có khả năng lặp lại.
2. Prompt mới chỉ khác wording, không khác lane, source, constraint hoặc verification.
3. Spec/contract chưa rõ, vì prompt lúc này sẽ hợp thức hóa suy đoán.
4. Module chưa ổn định, đang thay đổi liên tục.
5. Có thể giải quyết bằng cập nhật `prompts.md` hoặc `AGENTS.md` thay vì thêm file mới.
6. Prompt mới không có references tới source chính thức.
7. Prompt chỉ chứa "hãy làm tốt hơn", không có role, source, scope, forbidden actions, output contract, verification.

Nguyên tắc thực tế:

```text
Tạo prompts.md sớm.
Tạo prompt module muộn hơn, khi có pattern lặp lại hoặc rủi ro thật.
```

### 5. Quy Tắc Quyết Định Nhanh

| Câu hỏi | Nếu có | Hành động |
|---|---|---|
| Dự án đã bắt đầu dùng agent theo SDD chưa? | Có | Tạo `docs/grounding/prompts.md` |
| Có nhiều loại task khác nhau chưa? | Có | Định nghĩa lane trong `prompts.md` |
| Có module nào agent hay sửa không? | Có | Tạo prompt module-level |
| Có bug/rủi ro lặp lại do agent không tuân thủ rule? | Có | Tạo hoặc cập nhật prompt chuyên biệt |
| Có verification đặc thù cho module? | Có | Tạo prompt validation |
| Có role agent riêng như architect/coder/QA/KMS curator? | Có | Tạo prompt role-level |
| Chỉ là task nhỏ một lần? | Có | Không tạo prompt mới |

### 6. Mẫu Cấu Trúc Tối Thiểu Của `prompts.md`

`prompts.md` nên có các phần sau:

```markdown
H1: Prompt Governance and Routing

## Purpose

File này định nghĩa prompt lanes và grounding expectations cho agent.

## Grounding rule

1. AGENTS.md luôn ưu tiên trước prompt.
2. manifest.yaml là source registry.
3. prompts.md chỉ route prompt, không thay thế spec.
4. Không tạo business rule nếu chưa có grounding.
5. Luôn yêu cầu verification theo lane.

## Prompt lanes

| Lane | Khi dùng | Required grounding | Output |
|---|---|---|---|
| analysis | Hiểu yêu cầu/docs/code | manifest, sources, docs, llm-wiki nếu có | findings, assumptions, open questions |
| planning | Lập plan | docs, tests, src, workspace notes | plan, risk, verification |
| implementation | Sửa code/test | spec, contract, guardrails, tests | diff, tests, evidence |
| review | Review code/docs/KMS | source files, docs, QA | findings, severity, recommendation |
| testing | Chạy/đề xuất verify | QA matrix, runbook, contract | evidence table |
| documentation | Sửa docs | metadata schema, source policy | docs diff, link check |
| kms-maintenance | Update manifest/sources/prompts | grounding files | registry update |
| workspace-update | Update llm-wiki | docs/tests/src evidence | observation, contradiction, lesson |

## Prompt registry

| Prompt file | Lane | Intended use | Required sources |
|---|---|---|---|

## Maintenance rules

1. Thêm prompt mới thì cập nhật file này.
2. Nếu prompt là source chính thức thì cập nhật manifest.yaml và sources.md.
3. Không tạo prompt không có owner, intended_use, references, constraints.
```

```markdown
Dùng khi nào.

2. Contract/spec cần đọc.
3. QA/runbook cần đọc.

## Role

Bạn là ...

## Scope

Được làm gì.

## Forbidden actions

Không được làm gì.

## Workflow

1. Đọc source.
2. Tóm tắt current state.
3. Nêu assumptions/open questions.
4. Lập plan.
5. Thực hiện nếu đủ gate.
6. Verify.
7. Report.

## Output contract

1. Phạm vi.
2. Nguồn grounding đã dùng.
3. Plan hoặc changes.
4. Verification.
5. Rủi ro còn lại.

## Verification

Command hoặc evidence cần dùng.

## Maintenance notes

Khi nào cần cập nhật prompt này.
```

### 8. Kết Luận

`prompts.md` nên tạo sớm, ngay khi bootstrap grounding cho SDD, vì nó là router và governance cho prompt.

Các prompt cụ thể nên tạo sau, khi đã có một trong ba thứ:

```text
1. Lane lặp lại.
2. Module quan trọng.
3. Rủi ro hoặc lỗi agent lặp lại cần khóa bằng contract.
```

Cách làm tốt nhất là bắt đầu với `prompts.md` rất gọn, sau đó chỉ thêm prompt module khi thực tế vận hành chứng minh là cần.
