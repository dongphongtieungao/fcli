# Runbook triển khai KMS bằng Obsidian Wiki, rg, Grounding Manifest và llm-wiki

## 1. Mục tiêu

Runbook này hướng dẫn triển khai kho tri thức dự án theo mô hình Spec First và Agentic Development.

Runbook này là lớp vận hành chi tiết cho thiết kế trong `docs/methodology/o_2.kms_intergrate.md`. Khi có khác biệt, dùng `o_2.kms_intergrate.md` làm logic chuẩn và dùng tài liệu này để biết các bước triển khai, kiểm tra và vận hành hằng ngày.

Trong [knowledge_base.md](./knowledge_base.md), runbook này thuộc trụ cột "Grounding, Retrieval & Traceability với `AGENTS.md`, manifest, `kb.ps1`, `rg`".

Thiết kế mục tiêu dùng:

```text
1. docs/ là KMS chính thức của project.
2. docs/grounding/manifest.md là bản đồ grounding chính cho human và agent.
3. docs/grounding/manifest.yaml là registry máy đọc.
4. docs/grounding/sources.md là catalog source ID.
5. docs/grounding/prompts.md là prompt governance và prompt routing.
6. llm-wiki/ là controlled workspace của agent.
7. rg và PowerShell là retrieval, audit, traceability và sync check layer.
8. AGENTS.md là policy vận hành agent.
9. Obsidian là giao diện đọc, backlink, graph, tags, properties và template.
```

Mục tiêu cuối cùng:

```text
1. Agent biết tài liệu nào là source of truth trước khi phân tích hoặc code.
2. Agent không đọc toàn bộ docs một cách lãng phí.
3. Agent dùng rg để truy xuất đúng context theo task.
4. Agent dùng llm-wiki để ghi nhận hiện trạng, mâu thuẫn, assumption và lesson learned.
5. Agent verify code, test và tài liệu theo grounding chính thức.
6. Human có thể dùng Obsidian để xem graph, tag, backlink và trạng thái tài liệu.
7. Team có thể xác định task READY, BLOCKED, SYNCED, PARTIALLY_SYNCED hoặc NOT_SYNCED dựa trên evidence.
```

## 2. Nguyên tắc kiến trúc

### 2.1. docs là KMS chính thức

`docs/` chứa tài liệu chính thức của project.

Ví dụ:

```text
docs/01-requirements/05.spec-v1.md
docs/02-architecture/02.glossary.md
docs/02-architecture/03.vision.md
docs/02-architecture/04.architecture-c4.md
docs/02-architecture/05.nfr.md
docs/02-architecture/06.adr.md
docs/02-architecture/08.standards.md
docs/03-business-analysis/09.bpmn.md
docs/04-spec-test/10.openapi.yaml
docs/06-quality-assurance/13.prompt-guardrails.md
docs/06-quality-assurance/14.qa-matrix.md
docs/09-runbook/17.dor.md
docs/09-runbook/18.dod.md
```

Quy tắc:

```text
1. docs/ nói hệ thống phải như thế nào.
2. docs/ là căn cứ chính để thiết kế, code, test, review và nghiệm thu.
3. Nếu source of truth thay đổi, docs/ phải được cập nhật.
4. Nếu artifact chính thức mới được thêm, manifest.md, manifest.yaml và sources.md phải được cập nhật.
```

### 2.2. llm-wiki là controlled workspace của agent

`llm-wiki/` không phải KMS chính thức. Nó là workspace có kiểm soát cho agent.

Nó chứa:

```text
1. Agent observations.
2. Codebase map.
3. Module notes.
4. Open questions.
5. Assumptions.
6. Contradictions.
7. Risks.
8. Lessons learned.
9. Verification notes.
10. Task memory.
```

Quy tắc:

```text
1. llm-wiki/ không được override docs/.
2. llm-wiki/ không được dùng làm nguồn chốt nghiệp vụ.
3. llm-wiki/ được dùng để hiểu hiện trạng và tránh lặp lại phân tích.
4. Claim quan trọng trong llm-wiki/ phải có source reference tới docs, tests hoặc src.
5. Nội dung trong llm-wiki/ chỉ được promote sang docs/ khi có quyết định chính thức.
```

### 2.3. Grounding precedence

Agent phải xử lý theo thứ tự ưu tiên sau:

```text
1. User task hiện tại.
2. AGENTS.md.
3. docs/grounding/manifest.md.
4. docs/grounding/manifest.yaml nếu cần registry máy đọc.
5. docs/grounding/sources.md nếu cần source ID chi tiết.
6. docs/grounding/prompts.md nếu cần prompt lane.
7. docs/ official artifacts.
8. tests/features hoặc tests/bdd.
9. llm-wiki workspace context.
10. src current implementation.
11. Agent inference.
```

Giải thích:

```text
1. docs/ là intended behavior.
2. tests/features là executable acceptance evidence.
3. llm-wiki là observed workspace context, dùng để biết agent đã ghi nhận gì nhưng không được override docs/.
4. src là current implementation state, dùng để kiểm chứng hiện trạng code sau khi đã đọc context workspace liên quan.
5. Agent inference luôn có độ ưu tiên thấp nhất.
```

Nếu `docs/`, `tests/`, `src/` và `llm-wiki/` mâu thuẫn:

```text
1. Không tự chọn một bên là đúng nếu chưa có rule trong sources.md hoặc manifest.md.
2. Ghi vào llm-wiki/contradiction-log.md.
3. Đề xuất hướng xử lý.
4. Chỉ sửa code hoặc docs khi scope task cho phép hoặc human chốt.
```

## 3. Phạm vi áp dụng

Áp dụng cho project có:

```text
1. Tài liệu Markdown trong docs/.
2. Tài liệu muốn chuẩn hóa theo Obsidian compatible Markdown.
3. Agent cần khai thác tài liệu bằng rg, PowerShell và grounding manifest.
4. Số lượng tài liệu vừa đến lớn, ví dụ 500 file .md khoảng 20 MB.
5. Chưa cần vector database hoặc semantic search trong giai đoạn đầu.
6. Máy có PowerShell và rg, hoặc có quy trình cài đặt rg đã được dự án cho phép.
```

Không áp dụng như giải pháp duy nhất cho:

```text
1. Enterprise search đa người dùng có phân quyền phức tạp.
2. Semantic search đa ngôn ngữ quy mô lớn.
3. Graph database chính thức.
4. Knowledge portal có workflow approval phức tạp.
```

Các nhu cầu này nên đánh giá sau khi mô hình rg không còn đủ.

```markdown
## Official documentation groups

| Group | Path | Role | Source of truth |

## Agent workspace

| Path | Role | Source of truth |
|---|---|---|

| llm-wiki/codebase-map.md | Observed codebase structure | no |

| llm-wiki/lessons-learned.md | Agent lessons | no |

## Agent rule

## Maintenance rules

1. Add new official artifacts to this manifest.
2. Add machine readable entries to manifest.yaml.
3. Add source IDs to sources.md.
```

```markdown
# Metadata Schema

## Required fields

| Field | Required | Applies to | Meaning |
|---|---|---|---|
| id | yes | docs, llm-wiki | Stable document ID |
| title | yes | docs, llm-wiki | Human readable title |
| type | yes | docs, llm-wiki | Controlled document type |
| category | yes | docs, llm-wiki | Controlled category |
| status | yes | docs, llm-wiki | Lifecycle state |
| owner | yes | docs, llm-wiki | Responsible owner |
| workspace | yes | docs, llm-wiki | docs or llm-wiki |
| truth_level | yes | docs, llm-wiki | official, working, observed, assumption, archived |
| source_policy | yes | docs, llm-wiki | source_of_truth, workspace_only, reference_only or related policy |
| source_of_truth | yes | docs, llm-wiki | Source documents used for grounding |
| related | no | docs, llm-wiki | Related documents |
| verified_against | no | llm-wiki | Evidence used to verify workspace note |
| confidence | no | llm-wiki | low, medium, high, verified |
| tags | yes | docs, llm-wiki | Search and graph tags |

## Rules

1. Official docs must use `workspace: docs`.
2. Workspace notes must use `workspace: llm-wiki`.
3. Official docs should use `truth_level: official` after review.
4. Workspace notes must not use `truth_level: official`.
5. Workspace notes should use `source_policy: workspace_only` or `reference_only`.
6. Every important workspace claim must reference docs, tests or src.
```

```powershell
function SearchAll {
```

## 16. Bước 11: Cập nhật AGENTS.md

Thêm section sau vào `AGENTS.md`:

```markdown
# Lightweight KMS with rg, Grounding and Agent Workspace

## Documentation model

This project uses two documentation spaces.

1. `docs/` is the official KMS and source of truth for requirements, architecture, business flow, specification, guardrails, QA, runbook and handover.
2. `llm-wiki/` is the controlled agent workspace for observations, codebase map, module notes, assumptions, open questions, contradictions, risks, verification notes and lessons learned.

`llm-wiki/` must not override `docs/`.

## Canonical grounding entrypoint

Before any non-trivial task, read:

1. `AGENTS.md`
2. `docs/grounding/manifest.md`

Read these when needed:

1. `docs/grounding/manifest.yaml` for machine-readable source registry.
2. `docs/grounding/sources.md` for source ID catalog.
3. `docs/grounding/prompts.md` for prompt lane routing.
4. `docs/methodology/o_2.kms_intergrate.md` for KMS rg workflow and canonical taxonomy logic.
5. `docs/methodology/o_3.kms_obsidian_rg_runbook.md` for operational setup steps.
6. `llm-wiki/index.md` for workspace navigation.
7. `llm-wiki/contradiction-log.md` for known mismatches.
8. `llm-wiki/open-questions.md` for unresolved questions.

## Grounding precedence

Use this precedence order:

1. Current user task.
2. `AGENTS.md`.
3. `docs/grounding/manifest.md`.
4. `docs/grounding/manifest.yaml`.
5. `docs/grounding/sources.md`.
6. `docs/grounding/prompts.md`.
7. Official artifacts under `docs/`.
8. Executable specs under `tests/features/` or `tests/bdd/`.
9. Workspace context under `llm-wiki/`.
10. Current source code under `src/`.
11. Agent inference.

## Required retrieval workflow

Before analysis, planning, coding, review or refactoring:

1. Resolve grounding from `AGENTS.md` and `docs/grounding/manifest.md`.
2. Use `rg` to search official docs first.
3. Search executable specs and tests next.
4. Search `llm-wiki/` for workspace observations, assumptions, open questions and contradictions.
5. Search source code to verify current implementation.
6. Read only the files relevant to the task.
7. Report source files used.

## Workspace warm-up rule

If `llm-wiki/` does not contain enough information for the task:

1. Search docs with `rg`.
2. Search tests with `rg`.
3. Create or update workspace notes in `llm-wiki/` with unresolved questions, source references and contradictions.
4. Inspect source code with `rg` only after official docs/tests/workspace context are clear enough for the task.
5. Do not change official docs during workspace warm-up.

## No invention rule

Do not invent business rules, API behavior, data fields, statuses, permissions, validations or error handling that are not grounded in:

1. Official docs.
2. Executable specs.
3. Official contract or interface spec.
4. ADR.
5. Decision log.
6. Guardrails.
7. Explicit user instruction.

If behavior is unclear, write it to `llm-wiki/open-questions.md`.

## Contradiction rule

If docs, tests, source code and workspace notes conflict:

1. Record the conflict in `llm-wiki/contradiction-log.md`.
2. Do not silently choose one source unless the source precedence is explicit.
3. Propose a resolution.
4. Modify code or docs only if the task scope allows it.

## Implementation gate

Do not start implementation unless retrieval confirms that these artifacts exist or the user explicitly allows partial context:

1. Vision or scope document.
2. Glossary or domain terms.
3. Coding standards.
4. BPMN or business flow.
5. BDD, acceptance criteria or test scenario.
6. Contract or interface specification.
7. Guardrails or QA rules.

If required artifacts are missing, output `BLOCKED` and list missing artifacts.

## Verification rule

A task is not done because the agent says it is done.

A task is done only when one or more of these pass:

1. Unit test.
2. Integration test.
3. BDD scenario.
4. Contract check.
5. Static analysis.
6. Review checklist.
7. Manual verification.

## Workspace update rule

Update `llm-wiki/` when the task produces useful new knowledge:

1. Actual codebase structure.
2. Module behavior.
3. Missing documentation.
4. Mismatch between docs and code.
5. Assumptions.
6. Open questions.
7. Defects and lessons learned.
8. Verification notes.

## Docs promotion rule

Update `docs/` only when an official artifact changes:

1. Requirement changed.
2. Spec changed.
3. Contract changed.
4. ADR changed.
5. Business flow changed.
6. Guardrail changed.
7. QA checklist changed.
8. DoR or DoD changed.
9. Runbook changed.

When updating `docs/`, update `docs/grounding/manifest.md`, `manifest.yaml` and `sources.md` if source registry changes.

## Required task output

Before changing files, provide:

1. Task scope.
2. Grounding sources.
3. Relevant official docs.
4. Relevant executable specs.
5. Relevant workspace notes.
6. Current implementation findings.
7. Assumptions.
8. Contradictions.
9. Plan.
10. Verification method.

After changing files, provide:

1. Files changed.
2. Verification result.
3. Workspace updates.
4. Docs updates if any.
5. Remaining risks.
```

### 17.4. /kb_sync_check

```text
Command:

Purpose:
Check whether current code is synchronized with business documentation.

Steps:
1. Search docs for related requirement, business rule, business flow, acceptance criteria and contract.
2. Search tests for related acceptance or feature behavior.
3. Search llm-wiki for known contradictions and observations.
4. Search src for related implementation.
5. Compare the chain: Requirement -> Business Rule -> Flow -> Acceptance -> Contract -> Test -> Code.
6. Do not modify code.
7. Do not claim synced without test or explicit evidence.

Output:
1. SYNCED, PARTIALLY_SYNCED or NOT_SYNCED.
2. Evidence table with source files.
3. Missing links.
4. Code without business trace.
5. Business rules without tests.
6. Tests without clear business rule.
7. Open questions or contradictions.
8. Recommended next actions.
```

## 18. Bước 13: Quy trình vận hành chuẩn cho mỗi task

