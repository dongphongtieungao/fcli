# QA Matrix — OpenCode PrivateGPT Bridge MVP

## Purpose

Define what must be tested, which specification authorizes each check, and what evidence is required before implementation, PR or release can advance.

## Context

The suite currently contains Gherkin acceptance specifications but no selected BDD runner or complete runtime fixtures. Rows marked `required/pending` are gates to implement, not tests claimed to be passing.

## Verification levels

| Level | Purpose | Minimum evidence |
|---|---|---|
| Static | Validate docs, manifests, schemas and dependency boundaries. | Parser/search result, exit code and inspected warnings. |
| Unit | Protect pure validation, mapping, policy and parser behavior. | Test name/count and result. |
| Contract | Validate OpenAPI/application contracts and examples. | Parser/linter/contract-test result. |
| Integration | Exercise auth and PrivateGPT provider boundaries with approved fixtures/mocks. | Request/response fixture and test result. |
| Runtime | Prove the installed local process and cancellation behavior. | Command, logs and redacted response. |
| Pilot | Prove the real OpenCode client workflow. | Redacted capture and reviewer acceptance. |

## Artifact traceability

| Artifact | Upstream | Downstream verification/review |
|---|---|---|
| Requirement/use cases | Business scope and acceptance | BPMN, BDD, application/OpenAPI |
| BPMN | Main/alternate/error flows | Feature scenarios and QA-F rows |
| BDD features | Given/When/Then behavior | Automated steps or manual scenario evidence |
| Application Spec | CLI/runtime/state/error rules | Unit/integration/runtime checks |
| OpenAPI | HTTP request/response schema | Parser/linter and contract tests |
| Prompt Guardrails | AI scope/safety rules | Prompt review and diff checks |
| QA Matrix | Required checks/evidence | CI Quality Matrix and PR checklist |

## Evidence requirements

Every executed check records: source row, exact command, exit code, warnings/errors, relevant artifact/log path and verdict. Synthetic fixtures must not contain real credentials, prompts or source code.

## Verdict rules

| Verdict | Meaning |
|---|---|
| `VERIFIED` | Required checks for the scoped change ran and support the claim. |
| `PARTIALLY_VERIFIED` | Useful evidence exists but runtime, automation, fixture or human gate remains. |
| `FAILED` | Evidence contradicts the expected behavior. |
| `BLOCKED` | Required source, schema, environment or decision is absent. |
| `NEEDS_HUMAN_REVIEW` | Security, data, business or release decision is required. |

## Transition gate

Implementation may proceed when affected QA rows, fixtures and expected evidence are identified. PR/release may not pass by documentation alone: all applicable blocking rows must have executable or approved manual evidence, and open runtime gaps must be reported.
