# QA Matrix

## Purpose

Define a reusable verification matrix for documentation, contracts, code, runtime behavior, release artifacts, and evidence quality.

## Context

Project-specific test commands are not known yet. This matrix defines the verification categories and evidence shape that future project checks should use.

## Verification Levels

| Level | Use When | Evidence |
|---|---|---|---|
| Static review | Docs, config, schemas, prompts, or small code changes | Diff review, link check, parser, lint |
| Targeted test | A specific behavior, function, command, or contract changed | Unit test, contract test, schema validation, smoke command |
| Runtime check | application, API, UI, batch, job, or generated output changed | Exact command, exit code, logs, artifact metadata |
| Release evidence | Package, install, handover, or release readiness changed | Build output, checksum, smoke test, known issues, rollback note |
| Human review | Business, security, compliance, UX, or release approval is needed | Reviewer decision and unresolved risk |

## QA Matrix

| Area | Required Check | Method | Minimum Evidence |
|---|---|---|---|
| Documentation | Frontmatter, one H1, purpose/context, related links | Markdown review or script | Files checked and unresolved gaps |
| Grounding | Manifest, sources, prompts, and source IDs remain aligned | YAML parse, source path check, catalog review | Parse result and missing path list |
| Requirements | Requirement maps to use case and acceptance criteria | Trace review | Requirement ID, AC, open questions |
| Business flow | Workflow maps to BPMN/use case and exception paths | Flow review or scenario test | Flow ID and evidence gap |
| Contracts | API/application/schema/output contract remains parseable | Parser, schema lint, contract test | Command/result and warnings |
| Code behavior | Changed behavior is tested at the smallest useful level | Unit, integration, or smoke test | Exact command and exit code |
| Runtime | Entry point, config, logs, and outputs work together | Smoke check or dry run | Command, output path, warnings/errors |
| Data | Sensitive values are redacted and fixtures are approved | Review, scanner, fixture validation | Data governance decision |
| Release | Package can be installed or handed over with evidence | Build, install, smoke, checksum | Evidence pack and known issues |

## Verdict Rules

| Verdict | Meaning |
|---|---|
| `VERIFIED` | Relevant checks ran and evidence supports the claim. |
| `PARTIALLY_VERIFIED` | Some evidence exists but scope, environment, data, warnings, or artifacts are incomplete. |
| `FAILED` | A relevant check failed or evidence contradicts the expected result. |
| `BLOCKED` | Required source, environment, decision, or dependency is missing. |
| `NEEDS_HUMAN_REVIEW` | Human decision is required for business, security, compliance, UX, or release. |

## Promotion Rules

Promote new verification commands, caveats, known warnings, or release gates into this matrix when they become reusable. Do not claim a pass from examples, stale commands, or agent assertions without evidence.
