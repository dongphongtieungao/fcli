# Prompt Guardrails — OpenCode PrivateGPT Bridge

## Purpose

Constrain AI agents so generated plans, code, tests and documentation remain inside the confirmed OpenCode PrivateGPT Bridge specification. These rules prevent an agent from silently turning observations, reference plugin behavior or assumptions into product behavior.

## Context

The canonical requirement defines a local Python Bridge, OpenCode as the only MVP client, `gemini-2.5-pro` as the only public model, mandatory workspace Agent binding and no Bridge-side tool execution. `application-spec.md`, OpenAPI and BDD refine that scope but still contain explicit runtime-validation gaps.

## Source precedence for AI work

1. Current user instruction and repository `AGENTS.md` files.
2. Canonical requirement and accepted architecture decisions.
3. Application/OpenAPI contracts and BDD scenarios.
4. QA, quality gates, review checklist and runbooks.
5. Reference plugin/OpenCode documentation as observed evidence only.
6. Current implementation and agent inference last.

If sources conflict, stop and report the conflict. Do not choose the behavior that is easiest to implement.

## Mandatory guardrails

| ID | Guardrail | Required AI behavior | Forbidden behavior | Verification |
|---|---|---|---|---|
| PG-001 | Grounding | Cite the requirement/spec/test that authorizes behavior. | Inventing requirements from code or reference plugin. | Traceability review. |
| PG-002 | Scope | Implement only the local Bridge MVP and requested files. | Adding LiteLLM, gateway, multi-model/provider or extra API surfaces without approval. | Diff and spec review. |
| PG-003 | Model | Expose only `gemini-2.5-pro`; resolve its internal ID explicitly. | Model fallback or reuse of plugin legacy default. | Unit/contract test. |
| PG-004 | Agent | Require synchronized workspace Agent and `metadata.agent_id`. | Inline-instruction fallback or Agent recreation per request. | Agent integration test. |
| PG-005 | Tool boundary | Leave file/edit/bash/tool execution and approval to OpenCode. | Copying IntelliJ `AgentLoopRunner`, `ToolRegistry` or executing `<tool_call>`. | Dependency/code review and tests. |
| PG-006 | File context | Accept text already supplied by OpenCode; do not open paths or upload that text. | Bridge-side workspace reads or unapproved PrivateGPT file upload. | File-context test/review. |
| PG-007 | Auth/secrets | Use secure refresh-token storage, memory-only access token and redacted outputs. | Hardcoded/logged tokens, cookies, API keys, prompts or source code. | Secret/log review. |
| PG-008 | Streaming | Retry only before first relayed chunk and cancel upstream on disconnect. | Whole-request retry after output or orphan background work. | Stream/cancellation tests. |
| PG-009 | Contract gaps | Preserve `draft`/open status for unverified wire details. | Claiming exact OpenCode payload, SSE schema or error mapping without capture. | Evidence review. |
| PG-010 | Verification | Report exact checks, warnings, artifacts and limitations. | Claiming `VERIFIED`, CI pass or runtime compatibility without evidence. | Evidence pack review. |
| PG-011 | Change control | Update affected spec/test/registry surfaces when behavior changes. | Code-only contract drift or silent source-of-truth changes. | KMS/traceability check. |
| PG-012 | User work | Preserve unrelated changes and use reversible edits. | Broad cleanup, destructive reset or unrelated refactor. | Diff review. |

## Required prompt input contract

Before implementation, an AI task must identify:

1. Objective and files in/out of scope.
2. Requirement, architecture, application/OpenAPI and BDD sources.
3. Affected actor, input, output, state and failure paths.
4. Security/data impact and tool/file boundary impact.
5. Acceptance criteria and smallest meaningful verification.
6. Known gaps that require runtime evidence or human decision.

If any critical item is missing, limit work to analysis/specification or ask for clarification.

## Required AI output contract

Every implementation/review response must provide:

| Output | Required content |
|---|---|
| Scope | Files and behavior changed. |
| Sources | Official sources actually used. |
| Traceability | Requirement/spec/scenario IDs affected. |
| Verification | Exact command/check, result and warning/error inspection. |
| Gaps | Tests, runtime captures or decisions still missing. |
| Drift | Docs/tests/code/registry sync decision. |
| Safety | Confirmation that secrets/raw source/tool execution were not introduced. |

## Stop and escalation conditions

AI must stop implementation and report `BLOCKED` or `NEEDS_HUMAN_REVIEW` when:

1. A requested behavior conflicts with the canonical requirement, ADR or contract.
2. Supporting native tools, binary attachments or new public endpoints would change MVP scope.
3. Secure-store, retention or source-code transmission policy requires approval.
4. Required runtime payload/schema evidence is absent and implementation would require guessing.
5. A destructive action, external write or secret-handling change lacks explicit authorization.

## Prompt review gate

- [ ] Prompt names the source and acceptance target.
- [ ] Scope and forbidden actions are explicit.
- [ ] Tool/file/auth/streaming boundaries are preserved.
- [ ] Unknown wire behavior is treated as a gap.
- [ ] Verification and KMS sync expectations are present.
